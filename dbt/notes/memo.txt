# dbt（data build tool）入門資料

## Page 1

### dbt（data build tool）の概要

dbt（data build tool）は、データウェアハウス内でのデータ変換を管理するためのオープンソースツールです。データアナリストやアナリティクスエンジニアが、SQLを書くだけでデータパイプラインを構築・テスト・ドキュメント化できるように設計されています。

従来、データ変換の処理は Python や Java などのプログラミング言語で複雑なスクリプトを書く必要がありました。データの抽出、変換、ロードという一連の処理を、それぞれ異なるツールやフレームワークで管理しなければならず、データエンジニアリング分野の専門知識が求められていました。

dbt はこの状況を大きく変えました。SQL さえ書ければ、データ変換パイプラインを構築できるようなので、これにより、SQL に親しんだデータアナリストが、データエンジニアリングの専門知識がなくてもデータ変換の作業込み複雑・連続できるようになりました。

ここからは、dbt の主要機能を一つずつ見ていきましょう。

---

#### SQL によるデータ変換の定義

dbt では、SELECT 文を書くだけでデータ変換処理を定義できます。従来のデータパイプラインでは、CREATE TABLE でテーブルを作成し、INSERT 文や MERGE 文でデータを投入するという手続き的な SQL を書く必要がありました。dbt はこの手間を解消し、分析者は「どんなデータが欲しいか」を SELECT 文で記述するだけで済みます。テーブルやビューの作成に必要な DDL/DML 文は dbt が自動生成してくれるため、ビジネスロジックの記述に集中できます。

---

#### 依存間係の自動解決

dbt では、モデル間の依存関係を ref() 関数で定義します。例えば、売上集計モデルが顧客マスタモデルを参照している場合、ref(customers) と書くだけで依存関係が記録されます。dbt はこの情報をもとに DAG（有向非循環グラフ）を構築し、正しい実行順序を自動的に決定します。モデルの数が増えても、手動で実行順序を管理する必要はありません。変更があったモデルとその下流だけを選択的に実行することも可能です。

---

#### テスト機能

dbt では、データ品質を検証するテストを YAML ファイルで宣言的に定義できます。「このカラムは NULL にならない」「このカラムの値はユニークである」「このカラムの値は別テーブルのカラムに存在する」といったテストを、SQL を書かずに設定ファイルに記述するだけで実施できます。dbt test コマンド一つでそのテストを一括実行でき、データパイプラインの品質を継続的に担保できます。より複雑な検証が必要な場合は、SQL でカスタムテストを書くこともできます。

---

#### ドキュメント自動生成

dbt では、モデルやカラムの説明を YAML ファイルに記述しておくと、Web サイト形式のドキュメントを自動生成できます。生成されたドキュメントにはデータリネージ（データの流れ）がグラフとして可視化されるため、「このデータはどこから来ているのか」が一目で分かるようになります。ドキュメントはコードと同じリポジトリで管理されるため、モデルの変更に合わせてドキュメントを自動に更新される仕組みになっています。

---

#### バージョン管理との親和性

dbt では、モデルは SQL ファイルと YAML ファイルで構成されるため、Git によるバージョン管理を非常に相性が良いです。プルリクエストを使ったコードレビュー、ブランチを使った並行開発、CI による自動テストなど、ソフトウェアエンジニアリングで培われたベストプラクティスをデータ変換の世界にそのまま持ち込めます。「誰が、いつ、なぜこのロジックを変更したのか」がコミット履歴として残るため、変更の追跡や障害発生時の原因調査を容易になります。

---

### dbt の全体像

dbt がデータパイプライン全体の中でどのような位置づけにあるかを把握しておきましょう。

一般的なデータパイプラインは、以下のような流れで構成されます。

1. **データソース**：業務システム、SaaS アプリケーション、IoT デバイスなど、様々なシステムからデータが生成されます
2. **抽出・ロード（EL）**：Fivetran、Airbyte、Stitch などのツールを使って、データソースからデータウェアハウスにデータを取り込みます
3. **変換（T）**：取り込まれたデータを、分析に適した形に変換します。**ここが dbt の担当領域**です
4. **分析・可視化**：Tableau、Looker などの BI ツールを使って、変換されたデータを可視化・分析します

dbt は「変換」の部分に特化しているため、データの抽出やロード以降の処理を担います。このようにこのツール群管理を集中し、それらを組み合わせてデータを管理するアプリケーションはモジュラー・データスタックと呼ばれ、現在のデータ基盤標準のスタンダードになっています。

---

### dbt が解決する課題

dbt が登場する以前、データチームは以下のような課題を抱えていました。

#### データ変換ロジックの散在

データ変換のロジックが、BI ツールの計算フィールド、Excel のマクロ、Python スクリプト、ストアドプロシージャなど、様々な場所に散らばっていました。「この数字はどうやって計算されているのか？」という質問に答えるために、複数のシステムを横断して調査する必要がありました。

dbt を使うと、全てのデータ変換ロジックを SQL モデルとして一箇所に集めることができ、バージョン管理されたリポジトリのもとで、誰でもロジックを確認・修正できるようになります。

#### データ基盤上バッチの管理が困難

データ変換パイプラインでは、複数のデータ加工バッチがそれぞれ独立して動いており、バッチ間の依存関係を管理するのが困難でした。あるテーブルを更新した後に、下流のどのバッチを再実行する必要があるかが分からず、複数のシステムを横断した調査の調整が必要になっていました。

dbt は、モデル間の依存関係が ref() 関数を通じて明示的に記述され、データリネージ（自動有向非循環グラフ）で可視化できるようになります。これにより、変更の影響範囲が即座に把握でき、データ加工パイプライン全体を安定に管理作業できるようになっています。

#### テストとドキュメントの欠如

dbt 以前は、テストとドキュメントが後回しにされることが多かったです。データ品質の問題が発生するのは、経営報告で「この数字はおかしい」と指摘された時でした。

dbt では、テストとドキュメントがワークフローに自然に組み込まれているため、モデルの定義と同じファイルにテストやドキュメントを記述するため、「テストを書く」「ドキュメントを書く」ことが特別な作業でなく、データモデリングの一部として自然に行うことも可能になっています。

---

### ELT と ETL の違い

データパイプラインの構築方法として、ETL と ELT という２つのアプローチがあります。dbt の位置づけを理解するために、両者の違いを確認しておきましょう。

#### ETL（Extract → Transform → Load）

ETL は「抽出（Extract）→ 変換（Transform）→ ロード（Load）」の略です。データソースからデータを抽出した後、外部の処理サーバーでデータを変換し、最終的にデータウェアハウスにロードするアプローチです。

このアプローチには、データ変換を担う処理サーバーを別途用意する必要があります。変換処理のために Python や Java のプログラムを書き、その実行環境を維持・管理しなければなりません。データ量が増えると、サーバーのスケーリングも必要になります。

#### ELT（Extract → Load → Transform）

ELT は「抽出（Extract）→ ロード（Load）→ 変換（Transform）」の略です。データソースから抽出したデータを、まずそのままデータウェアハウスにロードし、その後データウェアハウスの中で SQL を使って変換するアプローチです。

近年、BigQuery、Snowflake、Redshift などのクラウドデータウェアハウスの性能が劇的に向上したため、これらのサービスはメガバイト規模のデータに対しても SQL クエリを高速に実行できるため、データウェアハウス内で変換処理を行う ELT アプローチが主流になっています。

#### dbt は ELT の「T」を担う

dbt のアプローチにおける「T（Transform）」の部分を担当するツールです。データの抽出（E）やロード（L）は、Fivetran、Airbyte、Stitch などの専用ツールが担います。dbt は、データウェアハウスにロードされた生データを、分析に適した形に変換する役割に特化しています。

この役割分担により、各ツールが得意な領域に集中でき、パイプライン全体の構築・運用がシンプルになります。

---

### dbt Core と dbt Cloud の違い

dbt には、**dbt Core** と **dbt Cloud** の２つのバージョンがあります。

#### dbt Core

dbt Core はオープンソース版です。Apache 2.0 ライセンスで公開されており、誰でも無料で利用できます。ターミナルからコマンドを実行して操作する CLI ツールで、自分の環境に合わせて自由にカスタマイズできる柔軟性があります。一方、自分のインフラで実行するため、ジョブのスケジューリングや CI/CD 連携は自分で管理する必要があります。

#### dbt Cloud

dbt Cloud は、dbt Labs が提供する SaaS（Software as a Service）版です。dbt Core の機能に加えて、ブラウザ上で SQL の編集や dbt の実行ができるジョブスケジューリング機能が搭載されています。また、プルリクエスト作成時に自動的にテストを実行する CI/CD 連携や、組み・ツールから dbt の変更を直接参照できる Semantic Layer 機能も備えています。モデルの実行結果やテスト結果を GUI で確認できる管理画面もあります。チームでの運用を進めている画面です。

#### 本書では dbt Core を使用

本書では dbt Core を使って学習を進めていきます。無料で始められ、dbt の本質的な機能をて体験できるためです。dbt Core で学習内容は、dbt Cloud にも十分活用できるままです。dbt Cloud は、チームでの運用やプロダクション環境での利用を検討する段階で導入を考えると良いでしょう。

---

#### Column：dbt Fusion について

2025 年、dbt Labs は dbt Fusion と称される新しいランタイムエンジンを発表しました。これは、dbt Core の後継となる次世代のコアエンジンです。

Rust で書き直された新しいエンジンにより、従来の dbt Core と比べて大幅な高速化が期待されています。dbt Cloud だけでなくローカル CLI でも利用可能とされており、dbt Core ユーザーにとっても恩恵があります。

dbt Fusion は段階的にリリースが予定されており、本書の内容は dbt Core をベースにしています。dbt の基本的な概念（モデル、テスト、ドキュメント、マテリアライゼーションなど）は dbt Fusion でもそのまま通用します。まずは dbt Core でしっかり基礎を身につけてから、今後の dbt の進化にも対応するための姿勢です。

### DuckDB とは

DuckDB は、分析用途に特化した軽量なデータベース管理システムです。

#### なぜ DuckDB を使うのか

dbt を学習する際、データウェアハウス（BigQuery、Snowflake、Redshift など）を用意するのはハードルが高く感じられるかもしれません。クラウドサービスのアカウント作成、課金設定、ネットワーク設定など、dbt の学習とは直接関係のない事前準備が必要になります。

DuckDB を使えば、こうした事前準備を一切省略できます。ファイル一つでデータベースが起動できるため、インストールしたその瞬間から dbt を試すことができます。

#### DuckDB の特徴

DuckDB には、学習用途だけでなく実際の分析業務でも役立つ多くの特徴があります。

DuckDB はインプロセス（組み込み型）のデータベースです。SQLite のように外部サーバーや依存関係なしで動作し、アプリケーションのプロセス内で直接実行されます。データベースは単一のファイル（例：jaffle-shop.duckdb）として永続化されるため、ファイルをコピーするだけでデータベースを別の環境に移動できます。また、Linux、macOS、Windows のあらゆる環境を選ばず利用可能なポータビリティの高さも魅力です。

分析性能を非常に優秀です。DuckDB は列指向のベクトル化エンジンを採用しています。列指向ストレージとは、データを行単位ではなく列単位で格納する方式です。分析処理では特定の列だけを集計することが多いため、必要な列だけのデータの読み込みを省略でき、処理速度が大幅に向上します。ベクトル化実行エンジンは、一度に大量のデータをまとめて処理することで、CPU キャッシュの効率を最大化します。

SQL 機能も充実しています。複雑なクエリのサポート、ウィンドウ関数、豊富な組み込み関数など、分析に必要な機能が一通り揃っています。さらに、CSV、Parquet、JSON など、様々なファイル形式を直接クエリできるため、外部ツールを使ってデータを変換・インポートする手間も不要です。

Python や R との統合も深く、データサイエンスのワークフローに自然に組み込めます。拡張メカニズムにより、カスタムのデータ型や関数、ファイルフォーマットの追加も可能です。MIT ライセンスのオープンソースプロジェクトであり、完全に無料で利用できます。

DuckDB のインストールは、次のステップで dbt と一緒に行いますので、ここでは別個にインストールする必要はありません。

### uv とは

uv は、Python のパッケージ管理ツールです。Rust 製の高速なツールで、従来の pip や venv に比べて非常に高速に動作します。

#### なぜ uv を使うのか

dbt Core は Python で書かれたツールです。Python のツールを使う際には、パッケージの依存関係管理と仮想環境の管理が重要になります。

仮想環境とは、プロジェクトごとに独立した Python 実行環境を作る仕組みです。仮想環境を使わずにグローバルにパッケージをインストールしてしまうと、異なるプロジェクトで必要なパッケージのバージョンが衝突するという問題が起こりえます。仮想環境を使えば、プロジェクト A では dbt 1.8 を、プロジェクト B では dbt 1.11 を、というように異なるバージョンを共存させられます。

従来は pip でパッケージをインストールし、venv で仮想環境を作るという２つのツールを組み合わせて使う必要がありました。uv はこれらの機能を統合し、一つのツールで仮想環境の作成からパッケージ管理まで全てシンプルにできるようにしたものです。

#### uv の特徴

uv の最大の強みは、Rust で実装されたことによる圧倒的な速度です。pip の 10〜100 倍高速にパッケージをインストールでき、大量の依存パッケージを持つ dbt のインストールでも、待ち時間がほとんどありません。

また、uv net でプロジェクトを作成すると仮想環境が自動的に準備されるため、venv コマンドを手動で実行する必要がありません。依存関係は pyproject.toml と uv.lock で厳密に管理され、チームメンバー全員が同じバージョンのパッケージを使えることが保証されます。さらに、uv 自身が Python のバージョン管理機能も備えているため、pyenv などの別ツールなしに Python のインストール・切り替えが可能です。

---

### インストール

それでは、実際に開発環境を構築していきましょう。開発環境は Mac を前提としていますが、Linux や WSL（Windows Subsystem for Linux）でも同様の手順で進められます。

#### uv のインストール

まず、uv をインストールします。以下のコマンドを実行してください。

```
curl -LsSf https://astral.sh/uv/install.sh | sh
```

Note：Homebrew を使っている場合は以下でもインストールできます。

```
brew install uv
```

インストール後、ターミナルを再起動するか、source ~/.zshrc（bash の場合は source ~/.bashrc）を実行して、uv コマンドが使えることを確認しましょう。

```
uv --version
```

バージョン番号が表示されれば、インストール成功です。

#### プロジェクトの作成

チュートリアル用のプロジェクトを uv で作成します。任意のディレクトリで以下のコマンドを実行してください。

```
uv init dbt-tutorial
cd dbt-tutorial
```

uv init コマンドを実行すると、プロジェクトディレクトリが作成されます。主要なファイルは pyproject.toml で、ここにプロジェクトの設定と依存パッケージの情報が管理されます。uv add でパッケージをインストールすると、このファイルに依存関係が自動的に追記されます。

#### dbs-duckdb のインストール

プロジェクトが作成できたら、いよいよ dbt をインストールします。dbt からデータプラットフォームに接続するためには、接続先に対応したアダプターが必要です。アダプターとは、dbt と各データプラットフォームの間の通信を仲介するパッケージです。

今回は DuckDB を使用するので、DuckDB 用アダプターである dbt-duckdb をインストールします。

```
uv add dbt-duckdb
```

このコマンドを実行すると、dbt-duckdb パッケージとその依存関係（dbt-core 本体を含む）が自動的にインストールされます。また、このタイミングで uv.venv（仮想環境のディレクトリ）も自動的に作成されます。

インストール完了後、仮想環境を有効化します。

```
source .venv/bin/activate
```

仮想環境が有効化されると、ターミナルのプロンプトに（dbt-tutorial）と表示されます。この状態で dbt コマンドが使えるようになります。

Note：仮想環境を有効化する代わりに、uv run dbt <subcommand> という形式でコマンドを実行することもできます。例えば uv run dbt debug のように使います。仮想環境の有効化を忘れがちな場合や、スクリプトから実行する場合はこの方法が便利です。本書では source .venv/bin/activate での有効化を前提として解説しますが、どちらの方法でも同じ結果が得られます。

dbt のバージョンを確認してみましょう。

```
$ dbt --version
Core:
  - installed: 1.11.0
Plugins:
  - duckdb: 1.10.1
```

dbt Core と DuckDB アダプターのバージョンが正しく表示されれば、インストールは完了です。これで dbt を使い始める準備が整いました。次の章では、実際に dbt を動かしてみましょう。

### jaffle-shop とは

jaffle-shop は、dbt Labs（dbt の開発元）が公式に公開しているサンプルプロジェクトです。架空の EC サイトのデータを使って、dbt のデータ変換、テスト、ドキュメント生成、セマンティックモデルといった主要機能を一通り体験できるようになっています。

#### 公式リポジトリを clone する

まず、jaffle-shop のリポジトリを GitHub から clone します。適当で作成した .dbt-tutorial ディレクトリ内にあることを確認して、以下のコマンドを実行してください。

```
cd dbt-tutorial
git clone https://github.com/dbt-labs/jaffle-shop.git
cd jaffle-shop
```

clone が完了すると、jaffle-shop ディレクトリ内に dbt プロジェクトの構成ファイルが展開されます。

---

#### プロファイルの設定

dbt を実行するには、データプラットフォームへの接続情報を設定する必要があります。この接続情報を管理するのがプロファイルです。

**■ プロファイル名を変更する**

まず、dbt_project.yml の profile の値を animal から jaffle_shop に変更します。

dbt_project.yml は dbt プロジェクトの設定ファイルで、プロジェクト名やモデルの設定などを管理しています。

**■ DuckDB への接続情報を設定する**

dbt のプロファイル（接続情報）は、~/.dbt/profiles.yml というファイルで管理されています。~ はホームディレクトリを指します。なお、dbt_project.yml と同じプロジェクトの root 直下に profiles.yml を配置することも可能です。チーム開発ではホームディレクトリ、個人のローカル環境ではどちらでも構いません。

このファイルがまだ存在しない場合は、~/.dbt/ ディレクトリ手動にファイルを新規作成してください。以下の内容で記述します。

```
~/.dbt/profiles.yml
```

各設定項目の意味は以下の通りです。

- **jaffle_shop**：プロファイル名。dbt_project.yml の profile と一致させます
- **outputs**：接続先の環境を定義するセクション。開発（dev）、本番（prod）など、複数の環境を定義できます
- **dev**：環境名。target で指定した環境がデフォルトで使用されます
- **type: duckdb**：使用するデータプラットフォームの種類
- **path: jaffle_shop.duckdb**：DuckDB のデータベースファイルのパス。この名前のファイルにデータが保存されます
- **threads: 1**：並列実行するスレッド数
- **target: dev**：デフォルトで使用する環境

Note：DuckDB はファイルベースのデータベースなので、BigQuery や Snowflake のようにホスト名やポート番号、追加情報の設定は不要です。path に指定したファイル（ここでは jaffle_shop.duckdb）にデータが保存されます。

#### dbt seed によるサンプルデータの生成


コマンドのオプションについて補足します。

- **--full-refresh**：既存のテーブルを削除して新しくデータをロードし直します。2 回目以降の実行でもクリーンな状態からデータを投入したい場合に使います。
- **--vars '{"local_source_data": true}'**：dbt 内の変数 local_source_data を true に設定します。jaffle-shop では、この変数が true の場合に seeds ディレクトリ内のデータが参照されるように dbt_project.yml で制御されています

Note：--vars オプションで設定した変数は、dbt_project.yml 内の {%if var('local_source_data', false) %} のように参照されています。デフォルト値は false なので、切り忘れる必要はありません。

実行が完了すると、顧客データ（raw_customers）、注文データ（raw_orders）、商品データ（raw_products）がDuckDBにロードされます。

### そもそもdbtモデルとは

dbtにおけるモデルとは、データ変換処理を定義したSQLファイルのことです。BIでは、データウェアハウス内のデータ変換処理を「モデル」という形式で管理します。

データのモデルは、データを収集してからデータ分析基盤と共用できるようにするための間に、様々なデータ変換処理が必要になります。例えば、

- カラム名称をわかりやすく言い換える
- 複数するテーブルをJOINして後から分析しやすくする
- モニタリングしたい指標値を算出する

といった処理です。

これらのデータ変換処理をSQLモデルとして定義し、dbtのコマンドで実行できます。内モデルは一つのSELECT文で構成され、その出力結果がテーブルやビューとしてデータウェアハウスに保存されます。

なお、モデルを記述する方法はSQL と Python の2通りありますが、本書ではSQLモデルのみで解説します。SQLモデルは、dbt の構文基本的かつ広く使われているモデル記述方法です。

### サンプルモデルでやっていること

サンプルモデルでは、架空ECサイト（Jaffle Shop）のデータ変換処理をやっています。全体像を理解するために、モデルが互いの上で多層構造になっているかを見ていきましょう。

モデルはおよそ3つの層に分かれており、左からへとデータ変換処理が進んでいきます。

- **データソース層**（raw_で始まる項目）: CSV から基本抽入されるデータ
- **ステージング層**（stg_で始まる項目）: 基本的な前処理を施した中間データ
- **データマート層**（customers、orders など）: 最終的な全情報データ

### ■ ステージング層

上で見たデータの問題点を解決するために、ステージング層でデータ加工を行います。ステージング層は、「後続するでの処理で必要になるような基本的な前処理」を行う層で、メタデータを「見える」状態にしてステージに吸い込む（ステージング）というイメージです。

dbt の SQL モデルは models ディレクトリに格納されています。ステージング層のモデルは models/staging ディレクトリにあります。

### ■ データマート層

データマート層は、最終的な成果物として、BIツールやアプリケーションから直接参照される데이터を作成する層です。

jaffle-shop では、customers、orders、order_items、products、suppliers、locations という6つのデータマートがあります。

ここでは代表的な order_items と customers を見てみましょう。

**order_items.sql**

```sql
with
  order_items as (
    select * from {{ ref('stg_order_items') }}
  ),
  orders as (
    select * from {{ ref('stg_orders') }}
  ),
  products as (
    select * from {{ ref('stg_products') }}
  ),
  suppliers as (
    select * from {{ ref('stg_suppliers') }}
  ),
  order_supplies_summary as (
    select
      product_id,
      coalesce(sum as supply_cost)
    from suppliers
    group by 1
  ),
  joined as (
    ...
    select
      order_items.*,
      orders.*,
      ...
  )
```

```sql
order_supplies_summary.supply_cost

left join products on order_items.order_id = orders.order_id
left join order_supplies_summary on order_items.product_id = order_supplies_summary.product_id
...

select * from joined
```

ここで注目すべきは、{{ ref('stg_order_items') }} という記述です。これは dbt モデルを参照するための関数で、Staging 層モデルを参照するために使われています。

1. **存在確認の依存関係**: 参照するモデルが先に実行されるよう、実行順序を自動的に決定します
2. **データリネージの追跡**: モデル間の依存関係がDAG（有向非巡回グラフ）として記録され、下キャンメントで可視化ができます

このモデルでは、ステージング層複数のモデルをJOINして、注文明細ごとの詳細情報をまとめています。

### customers.sql

次に customers.sql を見てみましょう。

```sql
with
  customers as (
    select * from {{ ref('stg_customers') }}
  ),
  orders as (
    select * from {{ ref('stg_orders') }}
  ),
  customer_order_summary as (
    select
      order_id as
      count(distinct customer_order_summary.count_lifetime_orders,
      customer_order_summary.first_ordered_at,
      customer_order_summary.last_ordered_at,
      customer_order_summary.lifetime_spend_pretax,
      customer_order_summary.lifetime_spend
  ),
  ...
  when customer_order_summary.is_repeat_buyer then
    'returning'
  else
    'new'
  end as customer_type
  from customers
  left join customer_order_summary
  ...
  select * from joined
```

注目すべきモデルとして、{{ ref('stg_orders') }}でデータマート層の orders モデルを参照しています。このように、データマート層の orders モデルを自動的に解析してくれるため、stg_orders モデルが先に実行されることが保証されます。

このモデルでは、ステージング層の顧客データとデータマート層の注文データを取込み合わせ、顧客ごとのサマリー情報を作成しています。具体的には以下の情報を提供します。

- **total_lifetime_orders**: 顧客ごとの通算注文回数
- **first_ordered_at**: 最初の注文日
- **last_ordered_at**: 最後の注文日
- **lifetime_spend**: 通算支出額（税込）
- **customer_type**: 顧客タイプ（new＝新規、returning＝リピーター）

注目すべきモデルとして、{{ ref('orders') }}でデータマート層のorders モデルを参照しています。

この時、どのモデルから指定処理を実行するかを気にする必要はありません。モデル間の参照関係（ref 関数や source 関数で宣言された依存関係）を dbt が自動的に判断し、正しな配布でで実行してくれます。これは、dbt がモデルの実行間関係を DAG（Directed Acyclic Graph：有向非巡回グラフ）として管理しているためです。

モデル実行にあたっての設定情報は、dbt_project.yml に記述します。

**dbt_project.yml**

```yaml
config-version: 2
name: 'jaffle_shop'
version: '0.0.14'

profile: jaffle_shop

model-paths: ['models']
analysis-paths: ['analyses']
test-paths: ['tests']
seed-paths: ['seeds']
macro-paths: ['macros']
snapshot-paths: ['snapshots']

target-path: 'target'
clean-targets: ['target', 'dbt_packages']

models:
  jaffle_shop:
    staging:
      +materialized: view
    marts:
      +materialized: table
    finance:
      +schema: finance
```

この設定ファイルの主な内容を見ていきましょう。

- **model-paths**: モデルファイルを配置するディレクトリ
- **seed-paths**: CSV ファイル（シードデータ）を配置するディレクトリ
- **macro-paths**: マクロを配置するディレクトリ
- **node**: シードのの設定。nodes.yml により、シードデータは nows スキーマに紐解されます

### ■ マテリアライゼーション

マテリアライゼーションとは、dbt のモデル出力方法のことです。モデルが SQL 実行結果を、どのような形式でデータウェアハウスに保存するかを指定します。適切なマテリアライゼーションを選択することで、パフォーマンスとコストを最適化できます。

### ■ table

通常のものと、モデルの実行結果をテーブルとして出力します。

jaffle-shop では、データマート層（マート層）のモデルがテーブルとして出力されるよう設定されています。

### ■ view

ビューとは、SQL のクエリ定義を保存したもので、参照するたびに SQL が実行されます。テーブルとは異なり、データを保存しません。

## Page 3

ステージング層のモデルはビューとして出力されています。ステージング層のモデルはデータマート作成時に一時的に参照されるものであり、BI ツールなどから直接参照されることは想定していません。テーブルとして実体化してストレージを消費するよりも、ビューとして定義しておく方が効率的です。ビューであれば、常に最新のソースデータを反映した結果が得られるというメリットもあります。

### ■ incremental

増分更新を行うマテリアライゼーションです。毎回テーブルを全件作り直す（フルリフレッシュする）のではなく、新しく追加・変更されたレコードだけを更新します。大量データを扱う場合にビルド時間を大幅に短縮できます。

```sql
{{
  config(
    materialized='incremental'
  )
}}

select * from {{ source('ecom', 'raw_orders') }}
{% if is_incremental() %}
  where ordered_at > (select max(ordered_at) from {{ this }})
{% endif %}
```

is_incremental() は、初回実行かどうかを判定する関数です。初回実行時（テーブルがまだ存在しない場合）は全件ロードし、2 回目以降の実行時は where 句の条件に合致するレコード（前回実行時間以降の新しいレコード）だけが追加されます。{{ this }} は、このモデル自身のテーブルを参照する特殊な関数です。

### ■ ephemeral

テーブルやビューとして実体化せず、参照元の SQL に CTE（Common Table Expression）として埋め込まれるマテリアライゼーションです。

```yaml
models:
  jaffle_shop:
    staging:
      +materialized: ephemeral
```

ephemeral モデルはデータベースにオブジェクトが作成されないため、一時的な中間計算にのみ必要で、データベースにテーブルやビューを残す必要がないケースに適しています。ただし、データベース上に実体がないため、デバッグ時に出力結果を直接確認しにくいというトレードオフがあります。

**packages.yml**

```yaml
packages:
```

ここで定義されたパッケージは、dbt deps コマンドでインストールされ、dbt_packages ディレクトリに配置されます。

例えば dbt_utils パッケージには、テストや SQL 生成に便利なマクロが多数含まれており、後のテストの章でも登場します。パッケージを活用することで、自分で一から実装しなくても、コミュニティが作成した便利な機能をすぐに利用できます。

### dbt プロジェクト構造の全体像

dbt プロジェクトでは、データ変換の流れに沿って以下の3つの層にモデルを構成する習慣があります。

#### ■ 1. Staging 層

データソースをもとに、dbt モデルの構成素をつくる最初の層で、ステージング層では基本的な前処理（カラム名の変更、型変換など）を行い、下流のモデルで使いやすい形に整えます。

#### ■ 2. Intermediate 層

Staging 層に近いモデルをデータ変換ロジックを連携するための層です。Staging 層と Mart 層の間の橋渡しとして、複雑な集計処理を分割・整理するときに役割を担います。

#### ■ 3. Mart 層

Staging 層、Intermediate 層で処理した積成果をもとに、最終的に必要とするデータマートを提供する層です。BIツールやアプリケーションから直接参照される最終成果物です。

#### アンチパターン：Staging 層でのるべきではないこと

以下のような処理は Staging 層では実施しない方がいいです。

- **JOIN**: Staging 層のモデルは、下流モデルで様々な組み合わせで利用される可能性があります。これを JOIN してしまうと、テーブル間の依存が複雑な関係に陥ってしまいます。下流モデルで柔軟な再利用が難しい状態になります。
- **集約処理**: 例えば「注文データを1行1注文に集計する」とした場合、下流モデルで データの形状が変わってしまいます。下流モデルよりも上流で粒度の変更は行う必要がないように気になります。

### ■ YAML 設定ファイルの構成

dbt プロジェクトでは、モデルのドキュメントやテストの設定をYAMLファイルで管理します。このYAMLファイルの整理方法は、プロジェクトの保守性に大きく影響します。

**ベストプラクティス：ディレクトリごとにYAMLファイルを作成する**

推奨されるのは、各ディレクトリに対して 上 ディレクトリ各 \_models.yml という命名規則でYAMLファイルを作成する方法です。

```
models/staging
├── _staging_models.yml
├── _staging_sources.yml
├── stg_customers.sql
├── stg_orders.sql
...

models/marts
├── _marts_models.yml
├── orders.sql
└── customers.sql
```

ファイル名の先頭にアンダースコアをつけることで、ディレクトリ内で YAML ファイルが常に上部にソートされ、見つけやすくなります。また、ディレクトリ名を含めることで、IDEのファジー検索でも素早くアクセスできます。

**Note**: jaffle-shop では各モデルごとに個別のYAMLファイル（orders.yml、customers.yml など）を用意しています。これは「テストが定義されていないモデルを発見しやすい」というメリットがある一方、ファイル数が増えるというトレードオフもあります。プロジェクトの規模やチームの組みにじて選択しましょう。重要なのはチーム内で一貫性を保つことです。

### ■ Macros

Macro は、繰り返し使われる SQL 変換処理を DRY（Don't Repeat Yourself）に保つための仕組みです。jaffle-shop の cents_to_dollars マクロが好例です。

**macros/cents_to_dollars.sql**

```sql
{% macro cents_to_dollars(column_name) -%}
  {{ return(adapter.dispatch('cents_to_dollars')(column_name)) }}
{%- endmacro %}

{% macro default__cents_to_dollars(column_name) -%}
  {{ column_name }} / 100::numeric(16, 2)
{%- endmacro %}
```

adapter.dispatch パターンを使うことで、接続先のデータプラットフォームに応じた適切な SQL 実装が自動的に選択されます。DuckDB など多くのデータベースでは default__ が実行されます。

マクロを定義したら、_macros.yml で目的と引数の説明をドキュメント化しておくと、チームメンバーが再利用しやすくなります。

### ■ プロジェクト成長のロードマップ

最後に、プロジェクトの成長に応じたモデル構造の進め方を示しておきます。

**スモールスタート（モデル数 1〜10）**

Staging 層と Mart 層の2層構造で十分です。jaffle-shop のようにシンプルな構造から始めましょう。Intermediate 層は不要です。

**成長期（モデル数 10〜50）**

Intermediate 層の導入を検討してきたら、Intermediate 層の導入を検討します。JOIN が5つ以上になる Mart 層モデルを見つけたら、Intermediate 層に切り出しましょう。YAML 設定ファイルの構成もディレクトリ単位に整理します。

**成熟期（モデル数 50 以上）**

ビジネスグループ別のサブディレクトリ構成を導入し、アクセス制御やカスタムスキーマの設定も検討します。codegen などの自動化ツールを活用して、新規モデル作成の効率化を図りましょう。

いずれの段階でも、「今の規模に合った構造」を選択することが大切です。将来を見越して過度に複雑な構造を作り込む必要はありません。プロジェクトやチームの事情に合わせて、段階的に構造を進化させていきましょう。
