# Lesson 2 サマリー — プロジェクト基盤の構築とデータロード

## 学習目標

- プロジェクト用のデータベース・スキーマ・テーブル・ステージ・ファイルフォーマットを作成できる
- JSONファイルを **VARIANT** カラムにロードできる
- JSONの解析パスと型キャストを使って個別カラムに分離できる
- SELECT文を**ビュー**（View）としてラップして再利用可能にできる

## 学んだこと

### プロジェクトロールの分担

| メンバー | プロジェクトロール | 担当内容 |
|---|---|---|
| **Agnieszka（Agnie）** | Game Developer | VRゲームの開発・データフィードの管理 |

> **なぜロールを分けるのか:** 実務のデータプロジェクトでは、データの提供者（ソースシステムの開発者）とデータの変換者（データエンジニア）は異なる人物です。この分担を意識することで、「誰に何を聞けばいいか」「誰が何に責任を持つか」が明確になります。

### データエンジニアのベストプラクティス

1. **ソースデータファイルは必ず探索する** — 中身を確認してからロードを計画する
2. **ほぼすべてに SYSADMIN を使う** — クリエイティブなロールとして使用
3. **ACCOUNTADMIN はほとんど使わない** — 必要最小限に留める

> **なぜSYSADMINを「クリエイティブなロール」と呼ぶのか:** Snowflakeの **SYSADMIN** はデータベース・スキーマ・テーブルなどのオブジェクトを作成（Create）するためのロールです。「クリエイティブ」とは、新しいものを作り出すという意味で使われています。一方 **ACCOUNTADMIN** はアカウント全体の管理者で、実務では通常のデータエンジニアには付与されません。

| ロール | 用途 | データエンジニアの使用頻度 |
|---|---|---|
| **SYSADMIN** | オブジェクトの作成・管理 | ほぼ常時 |
| **ACCOUNTADMIN** | アカウント管理・セキュリティ設定 | ごく稀（DORA実行時のみ） |
| **USERADMIN** | ユーザー・ロール管理 | 通常使わない |

### プロジェクトインフラストラクチャの作成

以下のオブジェクトを **SYSADMIN** で作成します：

```
AGS_GAME_AUDIENCE（データベース）
    └── RAW（スキーマ）
        ├── GAME_LOGS（テーブル：VARIANTカラム1つ）
        ├── LOGS（ビュー）
        ├── UNI_KISHORE（外部ステージ）
        └── FF_JSON_LOGS（ファイルフォーマット）
```

> **なぜPUBLICスキーマを削除するのか:** Snowflakeではデータベース作成時に **PUBLIC** スキーマが自動的に作られます。しかし、プロジェクトでは意味のある名前のスキーマ（ここでは **RAW**）を使うことで、データの精製レベルが一目でわかるようになります。不要な **PUBLIC** を削除しておくと、混乱を防げます。

### 外部ステージ（External Stage）

S3バケット上のファイルにSnowflakeからアクセスするために外部ステージを作成します。

> **なぜAWSバケット名にアンダースコアが使えないのか:** AWSのS3バケット命名規則では、バケット名にアンダースコア（_）を使えません（DNS互換性のため）。一方、Snowflakeのステージ名にはダッシュ（-）を使えません（SQLの識別子としてマイナス記号と混同されるため）。このため、同じ概念を指すのに異なる名前形式になることがあります。

### ファイルフォーマットの作成

**FF_JSON_LOGS** ファイルフォーマットのポイント：

| プロパティ | 設定値 | 理由 |
|---|---|---|
| TYPE | JSON | ソースファイルがJSON形式であるため |
| STRIP_OUTER_ARRAY | TRUE | JSONファイルの最外部の `[]` を除去し、各要素を個別の行として扱うため |

> **なぜSTRIP_OUTER_ARRAYが必要なのか:** 多くのJSONファイルは全体が `[{...}, {...}, ...]` という配列で囲まれています。この設定をTRUEにすると、配列の各要素がテーブルの1行として扱われます。FALSEのままだと、配列全体が1行としてロードされてしまいます。

### JSONデータのロードと解析

**COPY INTO** でJSONファイルを **GAME_LOGS** テーブル（VARIANTカラム）にロードした後、JSON解析パスと型キャストで個別カラムに分離します：

| テクニック | 構文例 | 用途 |
|---|---|---|
| JSON解析パス | `RAW_LOG:user_login` | VARIANTからキーを指定して値を取得 |
| 型キャスト | `::text`, `::timestamp_ntz` | VARIANT値を特定のデータ型に変換 |

> **なぜVARIANTカラムに一旦ロードするのか:** JSON、XML、Parquetなどの半構造化データは、構造が変化する可能性があります。VARIANTカラムにそのままロードしておけば、スキーマ変更なしにデータを受け入れられます。解析はビューやSELECTで後から行えます。これが「**スキーマオンリード**（Schema-on-Read）」のアプローチです。

### ビュー（View）の作成

SELECT文を **LOGS** ビューとしてラップすることで、JSONの解析ロジックを再利用可能にします。

```sql
CREATE VIEW my_view AS (SELECT x, y, z);
```

> **なぜビューを使うのか:** 複雑なJSON解析のSELECT文を毎回書くのは非効率です。ビューにしておけば、`SELECT * FROM LOGS` だけで整形済みのデータにアクセスできます。ビューはデータを保存せず、実行時に基礎テーブルを参照するため、テーブルのデータが更新されるとビューの結果も自動的に変わります。

## 🤖 DORAチェック — DNGW01

```sql
SELECT GRADER(step, (actual = expected), actual, expected, description) AS graded_results
FROM (
    SELECT
        'DNGW01' AS step,
        (
            SELECT COUNT(*)
            FROM AGS_GAME_AUDIENCE.RAW.LOGS
            WHERE IS_TIMESTAMP_NTZ(TO_VARIANT(DATETIME_ISO8601)) = TRUE
        ) AS actual,
        250 AS expected,
        'Project DB and Log File Set Up Correctly' AS description
);
```

> **このチェックが確認していること:** LOGSビューの **DATETIME_ISO8601** カラムが正しく **TIMESTAMP_NTZ** 型にキャストされているかを検証しています。IS_TIMESTAMP_NTZ関数がTRUEを返す行が250行あれば、JSONの解析と型キャストが正しく設定されていることを意味します。
