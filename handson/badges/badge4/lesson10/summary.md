# Lesson 10: 概念実証の成功とプロジェクトの次のステップ

## 学習目標

ワークショップ全体の成果を振り返り、「データを着地した場所に残す（Leave-it-where-it-lands）」アプローチの有効性を確認する。各チームメンバーの次のステップと、プロトタイプから本番への移行で考慮すべき点を理解する。

---

## 学んだこと

### 1. ワークショップ全体の振り返り — 何を実証できたか

このワークショップを通じて、MelとZenaは**データを一切ロードせずに**概念実証（Proof of Concept）を完成させました。

| 担当 | やったこと | 使った手法 |
|---|---|---|
| **Zena** | 画像データにアクセスしてWebカタログを構築 | ステージ + **ディレクトリテーブル** + ビュー |
| **Mel** | S3バケット上のデータからトレイル分析を構築 | 外部ステージ + ファイルフォーマット + ビュー |
| **Mel** | GeoJSONとParquetの異なる形式のデータを統合 | 文字列加工 + UNION ALL + 地理空間関数 |

> **なぜ「ロードしない」アプローチが成功したのか:** プロジェクトの初期段階では、テーブル設計やETLパイプラインの構築に時間をかけるよりも、データの使い方を素早く検証する方が価値があります。ビュー、ファイルフォーマット、ステージを組み合わせることで、データの探索と分析を即座に始められました。

> **ただし忘れてはいけないこと:** このアプローチはプロトタイプに最適ですが、本番運用ではパフォーマンス・保守性・信頼性の観点から、データをテーブルにロードすることが推奨されます。Lesson 9で学んだマテリアライズドビュー、外部テーブル、Icebergテーブルは、ロードせずにパフォーマンスを改善する「中間的な選択肢」です。

### 2. ワークショップで使った技術の全体像

このワークショップで学んだ技術を振り返ると、以下のように整理できます:

| レッスン | 学んだ技術 | なぜ必要か |
|---|---|---|
| **L1-4** | ステージ、ディレクトリテーブル、JOIN、CROSS JOIN | ロードせずにデータにアクセスし、カタログを構築 |
| **L5** | 緯度経度、WKT、ステージ、ファイルフォーマット | 地理空間データの基礎と環境構築 |
| **L6** | Parquetクエリ、geoJSON、ビュー | 異なるフォーマットのデータをビューで構造化 |
| **L7** | 地理空間関数、UNION ALL、ポリゴン | データの統合と空間分析 |
| **L8** | マーケットプレイス、UDF、オーバーロード | サードパーティデータの活用と再利用可能な関数 |
| **L9** | マテリアライズドビュー、外部テーブル、Iceberg | パフォーマンス改善とデータレイクハウス |

### 3. プロジェクトの次のステップ — 役割分担

メラニーとイゴール（投資家）がプロジェクトへの資金提供に合意しました。チームは次のフェーズに向けた役割を分担します:

| 担当者 | 責任 | 必要な技術 |
|---|---|---|
| **Mel** | モバイルアプリ開発（ログイン、チェックイン、リーダーボード） | フロントエンド開発、認証、リアルタイムデータ |
| **Zena** | ショッピングアプリをリワード引き換えアプリに変換し、Melのアプリと統合 | UI/UX、API統合、データ連携 |
| **Camila** | 外部トラッキングサービス（Strava等）とのデータ共有提携、自転車ショップとのプロモーション提携 | API連携、ビジネス交渉 |
| **Klaus** | パートタイムアドバイザーとしてデータ統合の選択を支援 | データエンジニアリング、アーキテクチャ設計 |

> **Klausの役割が特に重要な理由:** プロトタイプでは「ロードしない」戦略が有効でしたが、本番ではデータの一部を構造化テーブルにロードする必要が出てきます。どのデータをいつロードすべきかを判断するのがKlausのアドバイザーとしての役割です。すべてのデータを最初からロードする必要はなく、アクセス頻度やパフォーマンス要件に応じて段階的に移行するのがベストプラクティスです。

### 4. コース完了の確認手順

バッジを取得するための最終確認:

1. すべてのDORAテスト（**DLKW01〜DLKW09**）が **PASSING** かつ **VALID** であること
2. **YSAアプリ**（ysa.snowflakeuniversity.com）に **UNI_ID** と **UUID** でログイン
3. リンクレコードに **ACCOUNT ID** と **ACCOUNT LOCATOR** の**両方**が含まれていること（片方だけではバッジは発行されません）
4. バッジは自動発行（コースワーク完了から**最大1時間**）

> **ACCOUNT ID と ACCOUNT LOCATOR の違い:** Snowflakeには2種類のアカウント識別子があります。**ACCOUNT LOCATOR** は `ABC12345` のような短い文字列で、**ACCOUNT ID** は `org_name.account_name` のような組織名を含む形式です。バッジ発行システムは両方を使ってアカウントを特定するため、どちらか一方だけでは不十分です。
