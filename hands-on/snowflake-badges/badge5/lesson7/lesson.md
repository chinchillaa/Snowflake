## 🎯 SELECTに合わせた新しいターゲットテーブルを作成する（CTASでもOK）

**RAW** スキーマに **ED_PIPELINE_LOGS** という名前の新しいログテーブルを作成します。テーブル定義はどの方法で書いても構いません。テンプレートを使う、手動で書く、またはCTASを使うことができます。（「**ED**」は後で説明する「**Event-Driven**」の略です）

CTASで作成するには、SELECTステートメントの上に `CREATE TABLE <テーブル名> AS` と書いて実行するだけです。

CTASを使うとテーブルの作成とデータロードを1ステップで行えて便利ですが、少し困った点があります。すべてのVARCHARフィールドが非常に大きなサイズで定義されてしまいます。そのため、CTASをショートカットとして使った場合は、初回作成後にテーブル定義を調整することをお勧めします。

定義をコピーしてワークシートに貼り付けて編集すると、既存の行はすべて消えます。これは問題ありません。次のステップでCOPY INTOを書くからです。

> **参考:** すべてのVARCHARに100文字を使用しましたが、**USER_EVENT** だけは25文字にしました。

## 🥋 新しいCOPY INTOを作成する

CTASで入力された行をTRUNCATEします（CTASを使用し、短いVARCHARフィールドで再作成しなかった場合）：

```sql
truncate table ED_PIPELINE_LOGS;
```

COPY INTOを使ってテーブルを再ロードします：

```sql
COPY INTO ED_PIPELINE_LOGS
FROM (
    SELECT 
    METADATA$FILENAME as log_file_name 
  , METADATA$FILE_ROW_NUMBER as log_file_row_id 
  , current_timestamp(0) as load_ltz 
  , get($1,'datetime_iso8601')::timestamp_ntz as DATETIME_ISO8601
  , get($1,'user_event')::text as USER_EVENT
  , get($1,'user_login')::text as USER_LOGIN
  , get($1,'ip_address')::text as IP_ADDRESS    
  FROM @AGS_GAME_AUDIENCE.RAW.UNI_KISHORE_PIPELINE
)
file_format = (format_name = ff_json_logs);
```

より上級のデータエンジニアに自分の仕事の改善点を指摘してもらうことは非常に重要です。チームによっては設計レビュー会議が協調的な場合もあれば、リードDEが個人の設計哲学や上位管理層の要件に基づいて特定の変更を指示する場合もあります。

組織によって優先事項は異なります：

- パイプラインの実行速度が最重要な場合
- コストが最重要な場合
- 両者の適切な妥協点を常に目指す場合

Kishoreはリードデータエンジニアに、チームで最も重視される設計原則は何か、それらが相反するように見えるときにどう優先順位をつけるべきかを尋ねるべきです。定期的なデザインレビューにより、バランスが崩れた場合にリードがフラグを立てることもできます。

多くのチームでは、**PL_LOGS** ビューとCOPY INTOのステップを統合するかどうかの議論は、2つの別々のステップのまま残すという結論になることもあります。この判断は**メンテナンスのしやすさ**と**原子性（Atomicity）** のバランスに帰着します。別々のステップのままにすればより原子的になり、統合すればコードを容易に読める上級DEにとってメンテナンスがやや楽になります。

データエンジニアリングには技術スキルが必要ですが、技術力が高いだけでは優れたデータエンジニアとは言えません。最高のデータエンジニアは**創造性**も兼ね備えています。

Kishoreの同僚たちは、厳密に正しい・間違いという設計上の判断はほとんどないと説明しました。ジュニアデータエンジニアとしての彼の役割は、試行錯誤しながらでもSnowflakeの機能を把握し、それらの知識とスキルを組み合わせて組織の優先事項に沿った方法で目標を達成することです。

## 📓 イベント駆動パイプライン（Event-Driven Pipeline）

Kishore（とあなた！）が以前作成したパイプラインは**時間駆動パイプライン（Time-Driven Pipeline）** です。

時間駆動パイプラインは常に最適な解決策とは限りません。データがないのに探しに行ってコストを無駄にしたり、スケジュールが需要に合っていないとバックログが溜まったりする可能性があるからです。

時間駆動パイプラインの主な代替手段が**イベント駆動パイプライン**であり、Snowflakeの**Snowpipe**というオブジェクトによって実現されます。

イベント駆動パイプラインは、特定のイベントが発生するまで「スリープ」し、イベントが発生すると起動して応答します。今回のケースでは、バケットに新しいファイルが書き込まれることが「イベント」です。パイプがファイルの到着を「検知」すると、ファイルを取得してSnowflakeに取り込みます。

# 📓 クラウドコンピューティング速習

## 📓 現代のデータパイプラインのためのクラウドベースサービス

現代のデータパイプラインは、Amazon Web Services（AWS）、Microsoft Azure、Google Cloud Platform（GCP）などの主要クラウドプロバイダーが提供するクラウドベースサービスに依存しています。

Snowflakeでイベント駆動パイプラインを作成するには、クラウドプロバイダーが作成・管理する3種類のサービスが必要です。

| サービス種類 | AWS | Azure | GCP |
|---|---|---|---|
| **ストレージ（Storage）** | S3バケット | Blobストレージ | GCSバケット |
| **Pub/Sub通知サービス（Hub & Spoke）** | Simple Notification Services (SNS) | Azure Web PubSub / Azure Event Hub | Cloud Pub/Sub |
| **メッセージキューイング（Linear Messaging）** | Simple Queue Services (SQS) | Azure Storage Queues / Azure Service Bus Queues | Cloud Tasks |

---

## 📓 Pub/Subサービスの詳細

**Publish & Subscribe（Pub/Sub）サービス** は**ハブ＆スポーク（Hub & Spoke）パターン** に基づいています。

- **ハブ（HUB）**: メッセージの受信と送信を管理する中央コントローラー
- **スポーク（SPOKE）**: ハブに対して通知を送信または受信するソリューションやサービス

スポークが**パブリッシャー（Publisher）** であれば、ハブにメッセージを送信します。**サブスクライバー（Subscriber）** であれば、ハブからメッセージを受信します。1つのスポークがパブリッシャーとサブスクライバーの両方を兼ねることもできます。

多くの場所からPub/Subサービスにメッセージが出入りするため、すぐに混乱する可能性があります。そこでPub/Subサービスには**イベント通知（Event Notification）** と**トピック（Topic）** という概念があります。

- **トピック**: イベントタイプの集まり
- スポークは**トピック**に対して**通知**をパブリッシュし、**トピック**（通知のストリーム）にサブスクライブします
