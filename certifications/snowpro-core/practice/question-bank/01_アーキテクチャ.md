## 問題1: Snowflakeの3層アーキテクチャ

Snowflakeの3つのレイヤーとその役割を書け。

### あなたの回答

```
| レイヤー | 役割 |
|---------|------|
| 1. [アカウント] |わからない|
| 2. [データベース] |ディレクトリ的な|
| 3. [スキーマ] |テーブルを管理|

従来のデータウェアハウスとの最大の違い:
データ管理がSnowFlakeだけで全て完結するようになっている


このアーキテクチャの利点:
AWSとかみたく、データを管理するS3とコンピュートリソースを管理するEC2を接続して・・・みたいな煩わしい作業が必要ない
```

### 正解・解説

<details>
<summary>クリックして確認</summary>

| レイヤー | 役割 |
|---------|------|
| Storage Layer | データをカラム形式で圧縮・保存（クラウドオブジェクトストレージを使用） |
| Compute Layer | 仮想ウェアハウス（MPPクラスタ）によるクエリ実行 |
| Cloud Services Layer | 認証、メタデータ管理、クエリ最適化、トランザクション管理 |

**従来との違い:** ストレージとコンピュートが**分離**されている（従来は同一マシン上に結合）

**利点:**
- ストレージとコンピュートを独立してスケール可能
- コスト効率が高い（必要な分だけ課金）
- 複数のウェアハウスが同じデータに同時アクセス可能

</details>

  1.

```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**マイクロパーティション:** Snowflakeがデータを自動的に分割・格納する単位

- サイズ: **50〜500MB**（圧縮後）
- 保存形式: **カラム形式**（列指向）

**格納されるメタデータ:**
1. 各列の最小値・最大値（MIN/MAX）
2. NULL値の数
3. ユニーク値の数（Distinct Count）

このメタデータにより、クエリ実行時に不要なパーティションをスキップ（**プルーニング**）できる。

</details>

## 問題3: クエリプルーニング ★試験頻出

クエリ実行時にスキャンするマイクロパーティション数を削減する技術は何か？

### あなたの回答

```
技術名: [              ]

仕組み:


効果的なケース:


効果が得られにくいケース:

```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**技術名:** Pruning（プルーニング）

**仕組み:**
マイクロパーティションに格納されたメタデータ（MIN/MAX値など）を使用して、WHERE句の条件に合致しないパーティションをスキャンせずにスキップする。

**効果的なケース:**
- 日付範囲でフィルタリングするクエリ
- 特定の値で絞り込むクエリ
- データがクラスタリングされている場合

**効果が得られにくいケース:**
- データが散在している場合（同じ値が複数パーティションに分散）
- LIKE句での部分一致検索
- 関数を適用した列でのフィルタリング

</details>

## 問題4: 仮想ウェアハウス

仮想ウェアハウスについて説明せよ。

### あなたの回答

```
仮想ウェアハウスとは:


サイズの種類（小さい順）:
  [      ] → [      ] → [      ] → [      ] → [      ] → [      ]

サイズを1段階上げると:
  コンピュート能力: [   ]倍
  クレジット消費: [   ]倍

ウェアハウス間の特徴:
  データの共有: [可能 / 不可能]
  パフォーマンスへの相互影響: [ある / ない]
```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**仮想ウェアハウス:** SQLクエリを実行するための独立したMPP（超並列処理）コンピュートクラスタ

**サイズ:** X-Small → Small → Medium → Large → X-Large → 2X-Large → ...（6X-Largeまで）

**サイズを1段階上げると:**
- コンピュート能力: **2倍**
- クレジット消費: **2倍**

**ウェアハウス間の特徴:**
- データの共有: **可能**（同じストレージレイヤーにアクセス）
- パフォーマンスへの相互影響: **ない**（完全に独立）

各ウェアハウスのノードは他では使用されず、クエリが他のウェアハウスの影響を受けない。

</details>

## 問題5: Auto-suspend と Auto-resume

ウェアハウスの自動管理機能について説明せよ。

### あなたの回答

```
Auto-suspend:
  機能: 
  デフォルト値: [      ]分
  設定可能な最小値: [      ]秒

Auto-resume:
  機能: 
  トリガー: 

これらの機能のメリット:

```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**Auto-suspend:**
- 機能: 指定時間アイドル状態が続くとウェアハウスを自動停止
- デフォルト値: **10分**（600秒）
- 設定可能な最小値: **60秒**（1分）

**Auto-resume:**
- 機能: 停止中のウェアハウスにクエリが来ると自動起動
- トリガー: SQLクエリの実行リクエスト

**メリット:**
- コスト削減（使用していない時間は課金されない）
- 手動管理が不要
- ユーザーは常にクエリを実行可能（自動起動）

</details>

マルチクラスタウェアハウスとは:


設定項目:
  最小クラスタ数: 
  最大クラスタ数: 

スケーリングポリシー:
  Standard: 
  Economy: 

必要なエディション: [          ] Edition以上

```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**マルチクラスタウェアハウス:** 高い同時実行性が必要な場合に、同じサイズのクラスタを自動的に追加・削除する機能

**設定項目:**
- 最小クラスタ数: 常時稼働するクラスタ数
- 最大クラスタ数: スケールアウト時の上限

**スケーリングポリシー:**
- **Standard:** クエリがキューに入るとすぐにクラスタを追加（パフォーマンス優先）
- **Economy:** 6分間負荷が続いた場合にクラスタを追加（コスト優先）

**必要なエディション:** Enterprise Edition以上

**使用ケース:**
- 多数のユーザーが同時にクエリを実行
- BIツールからの大量の同時接続
- ピーク時間帯のワークロード対応

</details>

## 問題10: データクラスタリング

クラスタリングキーについて説明せよ。

### あなたの回答

```
クラスタリングキーとは:


自動クラスタリング:
  機能: 
  必要なエディション: [          ]以上

クラスタリングキーに適した列:
  1. 
  2. 

クラスタリングキーに不適切な列:
  1. 
  2. 

クラスタリングの効果を確認するシステム関数:
  SYSTEM$[                    ]('テーブル名')
```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**クラスタリングキー:** データを物理的に整理するために指定する列。プルーニング効率を向上させる。

**自動クラスタリング:**
- 機能: バックグラウンドでデータを自動的に再クラスタリング
- 必要なエディション: **Enterprise Edition**以上

**クラスタリングキーに適した列:**
1. WHERE句で頻繁にフィルタリングされる列
2. JOIN条件で使用される列
3. カーディナリティが適度な列（日付など）

**クラスタリングキーに不適切な列:**
1. カーディナリティが極端に高い列（UUID等）
2. 頻繁に更新される列

**クラスタリング効果の確認:**
```sql
SYSTEM$CLUSTERING_INFORMATION('テーブル名')
```

</details>
