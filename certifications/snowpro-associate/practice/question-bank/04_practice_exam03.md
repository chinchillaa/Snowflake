## 問題1: クローンに関する記述のうち、正しいもの

Snowflakeのクローンに関する記述のうち、正しいものはどれですか？

- A) クローンはデータベースからのみ作成可能であり、テーブルやスキーマからは作成できない。
- B) クローン作成時点ではストレージをほとんど（または全く）消費せず、元のデータかクローンされたデータのいずれかに変更が加えられた場合にのみストレージが消費される。
- C) クローンは元のオブジェクトと自動的に同期される。
- D) クローンは元のオブジェクトと同じ容量のストレージを消費する。

### あなたの回答

```
選択: []

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: B) クローン作成時点ではストレージをほとんど（または全く）消費せず、元のデータかクローンされたデータのいずれかに変更が加えられた場合にのみストレージが消費される。**

Snowflakeのクローン機能は、作成時点では「メタデータ（データを管理するための情報）」のみをコピーする操作です。これには以下の特徴があります。

- クローンは、作成時点では元のオブジェクトと同じデータを共有します。
    
- そのため、作成時には追加のストレージ容量を消費しません。
    
- 元のデータ、またはクローンされたデータのどちらかに変更が加えられた時に初めて、変更分のストレージが消費されます（コピー・オン・ライト動作：変更があった部分だけを新たにコピーして保存する仕組みで、変更前のデータは元のまま保持されます）。
    

これにより、テスト環境や開発環境の構築、バックアップなどの作業を非常に効率的に行うことができます。

❌ **クローンは元のオブジェクトと同じ容量のストレージを消費する**：作成時点では消費しません。

❌ **クローンは自動的に同期される**：作成後はそれぞれ独立したオブジェクトとして扱われ、同期はされません。

❌ **クローンはデータベースからのみ作成可能**：スキーマやテーブルなど、サポートされている他のオブジェクトもクローン可能です。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション7（オブジェクト階層）

</details>

## 問題3: 半構造化データ内のネストされた（階層の深い）フィールドを（2つ選択）

Snowflakeで半構造化データ内のネストされた（階層の深い）フィールドをクエリ（検索）するにはどうすればよいですか？（2つ選択）

- A) ブラケット記法（ `[]` ）を使用してアクセスする
- B) Cloud Functions（クラウド機能）を使用する
- C) ドット記法（ `.` ）を使用してアクセスする
- D) Snowflakeの専用関数を使用する

### あなたの回答

```
選択: []

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: A) ブラケット記法（ `[]` ）を使用してアクセスする、C) ドット記法（ `.` ）を使用してアクセスする**

Snowflakeでは、`VARIANT`型の列に保存されたJSONなどの半構造化データを扱う際、以下の記法を使って階層の深いデータに直接アクセスできます。

- **ドット記法**：
    
    例: `SELECT data:customer.name FROM my_table;`
    
- **ブラケット記法**：フィールド名に特殊文字や空白が含まれている場合に便利です。
    
    例: `SELECT data['customer']['name'] FROM my_table;`
    

これらの記法を使えば、事前にデータを平坦化（フラット化）したり変換したりすることなく、必要な要素を直接取得できます。

❌ **Snowflakeの専用関数を使用する**：`FLATTEN()` や `OBJECT_KEYS()` のような関数は階層データを展開したり調べるのに役立ちますが、フィールドにアクセスするための主要な方法ではありません。

❌ **Cloud Functionsを使用する**：無関係です。Snowflake内で完結するため、外部のクラウド機能は必要ありません。

📖 **参照先:** まとめ/03_データ管理_ロード.md > セクション7（半構造化データ）

</details>

### 実際に確認してみよう

**SQLでの確認:**
```sql
-- テスト用にVARIANT型のデータを挿入する
CREATE TABLE nested_json_test (data VARIANT);
INSERT INTO nested_json_test
  SELECT PARSE_JSON('{"customer": {"name": "Hanako", "address": {"city": "Osaka"}}, "tags": ["vip", "premium"]}');

-- ドット記法でネストされたフィールドにアクセスする
SELECT data:customer.name::STRING AS name FROM nested_json_test;
SELECT data:customer.address.city::STRING AS city FROM nested_json_test;

-- ブラケット記法でアクセスする（特殊文字を含むキー名にも対応）
SELECT data['customer']['name']::STRING AS name FROM nested_json_test;

-- 配列要素へのアクセス（0始まり）
SELECT data:tags[0]::STRING AS first_tag FROM nested_json_test;
```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: C) `ALTER STAGE ... SET ENABLE_DIRECTORY_TABLE = TRUE;` を使用して、外部ステージの ENABLE_DIRECTORY_TABLE プロパティを TRUE に設定する。**

Snowflakeにすでに存在する外部ステージでディレクトリアリテーブルを有効にするには、`DIRECTORY` パラメータを指定して `ALTER STAGE` コマンドを実行します。

❌ **ビューを作成する**：ビューを作成してもディレクトリアリテーブルの機能は有効になりません。

**SQLでの確認:**

## 問題10: ステージからSnowflakeのテーブルに構造化データを

ステージからSnowflakeのテーブルに構造化データを読み込む（ロードする）ために、一般的に使用されるSQLコマンドはどれですか？

- A) LOAD DATA
- B) INSERT INTO
- C) IMPORT DATA
- D) COPY INTO

### あなたの回答

```
選択: []

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: D) COPY INTO**

Snowflakeにおいて、ステージ（Snowflake内部の置き場、またはAmazon S3などの外部の置き場）からテーブルへデータを大量に読み込む際の標準的なコマンドが `COPY INTO` です。以下の特徴があります。

- CSV、JSON、Parquetなどのファイルからの高速な一括読み込み（バルクロード）をサポート。
    
- ファイル形式の指定、エラー発生時の動作、読み込み時の簡単なデータ変換が可能。
    

**例:**

SQL

```
COPY INTO my_table
FROM @my_stage/file.csv
FILE_FORMAT = (TYPE = 'CSV');
```

- B) MPP（超並列処理）コンピュートを使用する。
- C) 共有ディスク型アーキテクチャで実行する。
- D) クラウドストレージ内のデータを最適化することで実行する。

```
選んだ理由・メモ:


```

<details>
<summary>クリックして確認</summary>

**正解: B) MPP（超並列処理）コンピュートを使用する。**

Snowflakeの計算処理層は、MPP（Massively Parallel Processing：超並列処理）方式でクエリを実行します。具体的には以下のように動きます。

- クエリを細かいタスク（作業の単位）に分割し、複数のコンピュートノード（計算を行うサーバー群＝仮想ウェアハウス）に割り振ります。

❌ **クラウドストレージ内のデータを最適化する**：データの最適化（列指向フォーマットへの変換や圧縮）はデータがロードされて保存されるタイミングで行われるものであり、クエリを実行する仕組みそのものとは別の話です。

```
選択: []

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: A) そのテーブルが含まれている「データベース」または「スキーマ」に対する USAGE（使用）権限を持っていない。**

Snowflakeの権限管理では、一番奥にある「テーブル」を見る許可だけを持っていてもアクセスできません。建物の部屋に入るようなもので、以下の条件がすべて揃っている必要があります。

3. 部屋を見る許可（テーブルに対する `SELECT` 権限）
    

上位の階層（データベースやスキーマ）に対する `USAGE` 権限が欠けていると、システムはオブジェクトの場所にたどり着けず、アクセスをブロックします。

❌ **SELECT 権限が取り消された**：もしそうなら「権限がありません」という明確なエラーが出ます。

❌ **アカウントがロックされている**：ログイン自体ができず、他のクエリも一切実行できなくなります。

❌ **暗号化キーの問題**：Snowflakeではデータの暗号化と復号はシステムが裏側で自動的に行います。ユーザーが暗号化キーを手動で管理したり意識したりする必要はなく（Customer Managed Key「顧客管理キー」という高度な機能を除く）、これが原因でアクセスできなくなることは通常ありません。

</details>

### 正解・解説

<details>
<summary>クリックして確認</summary>

`PARSE_JSON()` 関数は、単なる文字列（テキスト）として扱われているJSONデータを読み取り、Snowflakeが半構造化データとして扱える `VARIANT` 型に変換する役割を持ちます。

**例:**

SQL


❌ **TO_JSON()**：これは逆に、`VARIANT` 型のデータを単なる「JSON形式の文字列」に変換して出力する機能です。

❌ **EXTRACT_JSON() / CONVERT_JSON()**：Snowflakeにはこのような関数は存在しません。

</details>

CSVデータを読み込む際、FILE FORMATの `SKIP_HEADER` オプションはどのような働きをしますか？

- A) ヘッダー行をデータの一部として含める。
- B) ヘッダー行から列の名前を自動的に作成する。
- C) データファイル内の空行を無視する。
- D) 指定した数のヘッダー（見出し）行を読み飛ばす。

### あなたの回答

```
選択: []

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: D) 指定した数のヘッダー（見出し）行を読み飛ばす。**

CSVファイルの上部に列名などの見出し行がある場合、それを実データとして読み込んでしまうとエラーになったりノイズになったりします。

`SKIP_HEADER` オプションに数字（通常は `1`）を指定することで、ファイルの一番上から数えて指定した行数分だけ、読み込み処理をスキップさせることができます。

❌ **ヘッダー行をデータの一部として含める**：`SKIP_HEADER` を設定しないと、「name, age, city」のような見出し行が実際のデータの1行目として読み込まれてしまい、意図しないデータが混入します。

❌ **空行を無視する**：これは `SKIP_BLANK_LINES = TRUE` という別のオプションで行います。`SKIP_HEADER` は行数の指定であり、空行とは別の概念です。

❌ **列の名前を自動的に作成する**：`SKIP_HEADER` はあくまで指定した行数分をスキップするだけです。列名を自動推定する機能は `INFER_SCHEMA` が担当します。

📖 **参照先:** まとめ/03_データ管理_ロード.md > セクション3（ファイルフォーマット）

</details>

### 実際に確認してみよう

**SQLでの確認:**
```sql
-- SKIP_HEADER = 1 の動作を確認する
CREATE FILE FORMAT test_csv
  TYPE = 'CSV'
  SKIP_HEADER = 1;

-- SKIP_HEADER なしとありの違いを確認するため、
-- まずSKIP_HEADERなしでロードしてみると1行目（ヘッダー）がデータとして混入する
COPY INTO my_table FROM @my_stage/data.csv
  FILE_FORMAT = (TYPE = 'CSV');  -- 危険: ヘッダー行もデータに含まれる

-- SKIP_HEADER = 1 で正しくヘッダーをスキップする
COPY INTO my_table FROM @my_stage/data.csv
  FILE_FORMAT = (TYPE = 'CSV' SKIP_HEADER = 1);

-- 空行を無視するには SKIP_BLANK_LINES を使う（別オプション）
COPY INTO my_table FROM @my_stage/data.csv
  FILE_FORMAT = (TYPE = 'CSV' SKIP_HEADER = 1 SKIP_BLANK_LINES = TRUE);
```
