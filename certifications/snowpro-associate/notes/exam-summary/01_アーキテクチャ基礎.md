## 1. Snowflakeとは

### 1.1 概要

Snowflakeは**クラウドネイティブなデータプラットフォーム**です。

> **クラウドネイティブとは？**
> 最初からクラウド環境で動作することを前提に設計されたシステムのこと。従来のオンプレミス（自社サーバー）向けソフトウェアをクラウドに移植したものとは異なり、クラウドの特性（弾力的なリソース、従量課金など）を最大限活用できる設計になっています。

### 1.2 主な特徴

| 特徴                 | 説明                           | メリット                    |
| ------------------ | ---------------------------- | ----------------------- |
| **SaaS提供**         | ソフトウェアをインストールせずWebブラウザからアクセス | インフラ管理が不要、すぐに利用開始可能     |
| **マルチクラウド**        | AWS、Azure、GCPで動作             | ベンダーロックイン回避、既存クラウド環境と統合 |
| **ストレージ・コンピュート分離** | データ保存とデータ処理が独立               | それぞれ別々にスケール可能、コスト最適化    |
| **ゼロ管理**           | パッチ適用、バックアップ等が自動             | 運用負荷の大幅削減               |

### 1.3 実際の使われ方

- **データウェアハウス**: 社内の様々なシステムからデータを集約し、分析に使用
- **データレイク**: 構造化データ（CSV等）と半構造化データ（JSON等）を一元管理
- **データ共有**: 社外パートナーや顧客とデータを安全に共有
- **機械学習基盤**: Cortex AIを使った予測分析や自然言語処理

## 2. 3層アーキテクチャ

### 2.1 なぜ3層に分かれているのか

従来のデータベースでは、ストレージ（データ保存）とコンピュート（データ処理）が一体化していました。そのため：

- ストレージを増やしたいだけなのに、コンピュートも一緒に増やす必要があった
- 夜間バッチ処理中は、分析クエリが遅くなった
- リソースの無駄が発生し、コストが高くなりがち

Snowflakeは**3つの層を完全に分離**することで、これらの問題を解決しています。

### 2.2 3層の全体像

```
┌─────────────────────────────────────────┐
│         Cloud Services Layer            │  ← 「頭脳」
│   認証、メタデータ管理、クエリ最適化      │    全体を制御・最適化
├─────────────────────────────────────────┤
│         Query Processing Layer          │  ← 「作業者」
│      （Virtual Warehouse / Compute）     │    実際にクエリを処理
├─────────────────────────────────────────┤
│         Database Storage Layer          │  ← 「倉庫」
│   クラウドストレージにデータを保存        │    データを安全に保管
└─────────────────────────────────────────┘
```

### 2.3 Database Storage Layer（ストレージ層）

**役割**: データを保存する「倉庫」

| 項目         | 説明                                   |
| ---------- | ------------------------------------ |
| **保存場所**   | 各クラウドのオブジェクトストレージ（S3、Azure Blob、GCS） |
| **フォーマット** | Snowflake独自の圧縮された列指向形式（後述）           |
| **管理**     | Snowflakeが完全管理（ユーザーは直接アクセス不可）        |
| **課金**     | 保存データ量に基づく月額課金                       |

> **列指向（Columnar）フォーマットとは？**
> 従来のデータベースは「行」単位でデータを保存しますが、Snowflakeは「列」単位で保存します。
>
> 例：売上テーブル（日付、商品名、金額）の場合
> - 行指向: [2026-01-01, りんご, 100], [2026-01-02, みかん, 150], ...
> - 列指向: [2026-01-01, 2026-01-02, ...], [りんご, みかん, ...], [100, 150, ...]
>
> **メリット**: 「金額の合計を出して」といったクエリでは、金額の列だけ読めばよいので高速。分析処理に最適。

### 2.4 Query Processing Layer（コンピュート層）

**役割**: クエリを実行する「作業者」

| 項目 | 説明 |
|------|------|
| **実体** | Virtual Warehouse（仮想ウェアハウス） |
| **役割** | SQLクエリの実行、データのロード/アンロード |
| **特徴** | 複数のWarehouseが独立して動作 |
| **課金** | 使用時間に基づく（クレジット消費） |

**実際の使われ方**:
- 「分析用Warehouse」「ETL用Warehouse」「レポート用Warehouse」のように用途別に作成
- 繁忙期はサイズを大きく、閑散期は小さくして**コストを最適化**

### 2.5 Cloud Services Layer（クラウドサービス層）

**役割**: 全体を統括する「頭脳」

| 機能          | 説明               | 具体例                 |
| ----------- | ---------------- | ------------------- |
| **認証・認可**   | ユーザーの本人確認とアクセス制御 | ログイン処理、ロール確認        |
| **メタデータ管理** | テーブル定義や統計情報を管理   | テーブルの行数、カラムの最小値/最大値 |
| **クエリ最適化**  | 最適な実行計画を作成       | 不要なデータをスキップする判断     |
| **インフラ管理**  | リソースの自動スケーリング    | Warehouseの自動起動/停止   |

> **ポイント**: Cloud Services層は常時稼働しており、Warehouseが停止中でも動作します。
> そのため、テーブル定義の確認（`SHOW TABLES`）やメタデータを使った`COUNT(*)`は**Warehouse不要**で即座に応答できます。

### 2.6 理解度チェック

- Q1: ストレージとコンピュートが分離されていると、どんなメリットがある？
- Q2: Cloud Services層が管理するものを3つ挙げられる？
- Q3: Warehouseが停止中でも実行できる操作は何？

## 3. Virtual Warehouse（仮想ウェアハウス）

> 🔗 **検証SQL**: [`verification/01_architecture/01_warehouse_operations.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/01_architecture/01_warehouse_operations.sql)

### 3.1 Virtual Warehouseとは

**Virtual Warehouse**は、クエリを実行するための**コンピュートリソースの集合体**です。

> **なぜ「仮想」なのか？**
> 物理的なサーバーを購入・管理する必要がなく、必要な時に必要なだけクラウド上に「仮想的に」作成できるためです。数秒で起動し、不要になれば停止してコストを抑えられます。

### 3.2 構成

```
Virtual Warehouse (例: X-Large)
├── ノード1: [CPU] [メモリ] [SSD]
├── ノード2: [CPU] [メモリ] [SSD]
├── ...
└── ノード16: [CPU] [メモリ] [SSD]
```

- **MPP（Massively Parallel Processing）** アーキテクチャを採用
- 複数ノードが協調してクエリを**並列処理**

> **MPPとは？**
> 大量のデータを複数のノードで分担して同時に処理する方式。1人で10時間かかる作業を10人で分担すれば1時間で終わる、という考え方です。

### 3.3 重要な特徴

| 特徴 | 説明 | なぜ重要か |
|------|------|-----------|
| **独立性** | 各Warehouseは完全に独立 | 他のWarehouseの負荷に影響されない |
| **同時アクセス** | 複数Warehouseが同じデータにアクセス可能 | ストレージ共有だがコンピュートは分離 |
| **即時スケーリング** | サイズ変更が数秒で反映 | 負荷に応じた柔軟な対応 |
| **自動サスペンド** | 非アクティブ時に自動停止 | 無駄なコストを削減 |
| **自動レジューム** | クエリ実行時に自動起動 | 手動操作不要 |

**実際の使用シーン**:
```sql
-- Warehouseの作成（自動停止5分、自動起動ON）
CREATE WAREHOUSE analytics_wh
  WAREHOUSE_SIZE = 'MEDIUM'
  AUTO_SUSPEND = 300        -- 5分間アイドルで停止
  AUTO_RESUME = TRUE        -- クエリ時に自動起動
  INITIALLY_SUSPENDED = TRUE;

-- 使用するWarehouseを指定
USE WAREHOUSE analytics_wh;

-- クエリを実行（Warehouseが自動起動）
SELECT * FROM sales WHERE date = '2026-01-01';

-- 5分間クエリがなければ自動停止（課金も停止）
```

### 3.4 サイズと課金

| サイズ | ノード数 | クレジット/時 | 適したユースケース |
|--------|:--------:|:-------------:|-------------------|
| X-Small | 1 | 1 | 開発・テスト、軽量クエリ |
| Small | 2 | 2 | 小規模分析 |
| Medium | 4 | 4 | 一般的な分析業務 |
| Large | 8 | 8 | 大規模データ処理 |
| X-Large | 16 | 16 | 複雑な集計、大量データ |
| 2X-Large〜 | 32〜 | 32〜 | 超大規模処理 |

> **サイズ選びのコツ**:
> - サイズが1段階上がると、ノード数と**クレジット消費が2倍**に
> - ただし処理能力も上がるため、**処理時間は約半分**に
> - 結果として、**総コストはほぼ同じ**（ただし完了が早い）
> - 「大きいサイズで短時間」か「小さいサイズで長時間」かは要件次第

### 3.5 スケーリングの種類

Warehouseのスケーリングには方向が2種類あります。

| 方向 | 名称 | 内容 | 対象エディション |
|------|------|------|----------------|
| **サイズ変更** | スケールアップ（UP） | WarehouseサイズをXS→Sなど大きくしてサーバー数を増やす | 全エディション |
| **サイズ変更** | スケールダウン（DOWN） | WarehouseサイズをXL→Mなど小さくしてサーバー数を減らす | 全エディション |
| **クラスタ増減** | スケールアウト（OUT） | 同時クエリ増加時にクラスタを自動追加（マルチクラスタ） | Enterprise以上 |
| **クラスタ増減** | スナップバック（Snap Back） | 需要が落ちるとクラスタを自動削除してスケールイン | Enterprise以上 |

> **混同注意**: スケールアップ/ダウンは「1つのWarehouseのサイズ変更」、スケールアウト/イン（Snap Back）は「クラスタ数の増減」です。

### 3.6 マルチクラスタウェアハウス（Enterprise Edition以上）

同時実行クエリが増えた時に、**自動的にクラスタを追加**する機能。

```
通常時:
[クラスタ1] ← 全クエリを処理

繁忙時（自動スケールアウト）:
[クラスタ1] ← クエリA, B を処理
[クラスタ2] ← クエリC, D を処理（自動追加）
[クラスタ3] ← クエリE, F を処理（自動追加）

需要減（Snap Back）:
[クラスタ1] ← クラスタ2, 3が自動削除される
```

| スケーリングポリシー   | 動作              | 適したケース  |
| ------------ | --------------- | ------- |
| **Standard** | 負荷に応じて素早くクラスタ追加 | レスポンス重視 |
| **Economy**  | クラスタ追加を抑制しコスト優先 | コスト重視   |

### 3.7 トライアルアカウントの標準Warehouse

Snowflakeトライアルアカウントには、初期状態で以下の3つのWarehouseが作成されています。

| Warehouse名 | 用途 | オーナー |
|---|---|---|
| **COMPUTE_WH** | 一般的なクエリ・ラボ作業用 | ACCOUNTADMIN |
| **SNOWFLAKE_LEARNING_WH** | 新規アカウントのデフォルト付与用 | ACCOUNTADMIN |
| **SYSTEM$STREAMLIT_NOTEBOOK_WH** | StreamlitアプリとNotebookの実行専用（Snowflakeが自動使用） | システム |

> **ポイント**: `SYSTEM$STREAMLIT_NOTEBOOK_WH` はユーザーが直接使うものではなく、Snowflakeが自動的にStreamlit/Notebookの実行に使うWarehouseです。

## 4. マイクロパーティション

### 4.1 マイクロパーティションとは

Snowflakeがデータを内部で管理する論理的な**最小単位**です。ユーザーが意識する必要はありませんが、仕組みを理解するとパフォーマンスチューニングに役立ちます。

### 4.2 特徴

| 特徴 | 説明 | なぜそうなっているか |
|------|------|---------------------|
| **サイズ** | 50〜500 MB（圧縮前） | 大きすぎず小さすぎない最適サイズ |
| **自動管理** | Snowflakeが自動で作成・管理 | ユーザーの手間ゼロ |
| **不変（Immutable）** | 一度作成されると変更されない | 整合性確保、Time Travel実現 |
| **メタデータ保持** | 各パーティションの統計情報を保存 | 高速なクエリプルーニング |

> **Immutable（不変）とは？**
> データを更新すると、元のパーティションを変更するのではなく、新しいパーティションが作成されます。古いパーティションは一定期間保持されるため、**Time Travel**（過去データへのアクセス）が可能になります。

### 4.3 メタデータの活用（クエリプルーニング）

各マイクロパーティションは以下のメタデータを保持：

- 各カラムの**最小値/最大値**
- **NULL値の数**
- **ユニーク値の概算数**

**クエリプルーニングの動作例**:

```sql
SELECT * FROM sales WHERE sale_date = '2026-03-15';
```

```
パーティション1: sale_date 範囲 [2026-01-01 〜 2026-01-31] → スキップ ✗
パーティション2: sale_date 範囲 [2026-02-01 〜 2026-02-28] → スキップ ✗
パーティション3: sale_date 範囲 [2026-03-01 〜 2026-03-31] → スキャン ✓
パーティション4: sale_date 範囲 [2026-04-01 〜 2026-04-30] → スキップ ✗
```

→ **4パーティション中1つだけ**読めばよいので、処理が高速化

※ 個々のマイクロパーティションそのものの実体を一覧表示することは不可能ですが、クエリプロファイルでTable Scan オペレータを確認することで特定のクエリに対してどの程度のマイクロパーティションがスキャンされたかは確認可能。

### 4.4 よくある間違い

- ✗「マイクロパーティションはユーザーが作成・管理する」→ ○ 自動管理
- ✗「パーティションを指定してクエリする」→ ○ Snowflakeが自動判断
- ✗「データを更新するとパーティションも更新される」→ ○ 新規作成される

## 5. キャッシング

> 🔗 **検証SQL**: [`verification/01_architecture/03_caching.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/01_architecture/03_caching.sql)

### 5.1 なぜキャッシュが重要か

同じデータに何度もアクセスする場合、毎回ストレージから読み込むのは非効率です。Snowflakeは**3種類のキャッシュ**を使って、処理を高速化しコストを削減します。

### 5.2 3種類のキャッシュ

```
┌─────────────────────────────────────────────────────────┐
│                  Cloud Services層                        │
│  ┌─────────────────────────────────────────────────┐    │
│  │ Result Cache（結果キャッシュ）                    │    │
│  │ - クエリ結果そのものを保存                        │    │
│  │ - 24時間保持                                     │    │
│  │ - Warehouse不要で即座に応答                      │    │
│  └─────────────────────────────────────────────────┘    │
├─────────────────────────────────────────────────────────┤
│                    Compute層                            │
│  ┌─────────────────────────────────────────────────┐    │
│  │ Local Disk Cache（ローカルディスクキャッシュ）     │    │
│  │ - WarehouseのSSDにデータをキャッシュ              │    │
│  │ - Warehouse稼働中のみ有効                         │    │
│  │ - 頻繁にアクセスするデータを高速読み込み           │    │
│  └─────────────────────────────────────────────────┘    │
├─────────────────────────────────────────────────────────┤
│                  Cloud Services層                        │
│  ┌─────────────────────────────────────────────────┐    │
│  │ Metadata Cache（メタデータキャッシュ）            │    │
│  │ - テーブル統計情報を保存                          │    │
│  │ - COUNT(*)等が即座に応答                         │    │
│  └─────────────────────────────────────────────────┘    │
└─────────────────────────────────────────────────────────┘
```

### 5.3 各キャッシュの詳細

| キャッシュ | 保存場所 | 保持期間 | 利用条件 | コスト |
|-----------|---------|---------|---------|--------|
| **Result Cache** | Cloud Services | 24時間 | 同一クエリ、データ未変更 | **無料** |
| **Local Disk Cache** | Warehouse SSD | Warehouse稼働中 | 同じWarehouseを使用 | 通常課金 |
| **Metadata Cache** | Cloud Services | 常時 | メタデータ参照クエリ | 無料 |

**実際の動作例**:

```sql
-- 1回目: ストレージからデータを読み込んで処理（10秒）
SELECT SUM(amount) FROM sales WHERE year = 2026;

-- 2回目（同一クエリ、24時間以内）: Result Cacheから即座に応答（0.1秒）
SELECT SUM(amount) FROM sales WHERE year = 2026;

-- 3回目（WHERE条件が異なる）: Local Disk Cacheを活用（3秒）
SELECT SUM(amount) FROM sales WHERE year = 2025;
```

### 5.4 キャッシュを意識した運用

- **Result Cache活用**: 定型レポートは同じクエリを使う → 2回目以降は無料で高速
- **Local Disk Cache活用**: 同じWarehouseを使い続ける → キャッシュが蓄積
- **キャッシュ無効化**: `ALTER SESSION SET USE_CACHED_RESULT = FALSE;` でテスト時に無効化

## 6. クラスタリング

> 🔗 **検証SQL**: [`verification/01_architecture/04_clustering.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/01_architecture/04_clustering.sql)

### 6.1 マイクロパーティションとプルーニングの関係

§4 で説明したとおり、Snowflake はデータをマイクロパーティションに分割して保存し、クエリ実行時に各パーティションのメタデータ（最小値・最大値など）を参照して**不要なパーティションを読み飛ばす（プルーニング）**ことで処理を高速化します。

ここで重要なのは、**プルーニングが効くかどうかはデータの並び順に大きく依存する**という点です。

**自然なクラスタリング（プルーニングが効く状態）**

日付順にデータをロードしているテーブルでは、`sale_date` の近いデータが同じパーティションに固まります。`WHERE sale_date = '2026-03-15'` のクエリは少数のパーティションだけを読めばよいため、プルーニングが効きます。

**クラスタリングの崩れ（プルーニングが効かない状態）**

以下の場合、関連データが多くのパーティションに分散し、スキャン効率が悪化します。

- データの更新（UPDATE / DELETE）が繰り返されてパーティションの内容が混在する
- ロード順とは異なる列（例：地域、顧客ID）での検索が頻繁に発生する

---

### 6.2 クラスタリングキーによる改善

特定の列を「クラスタリングキー」として指定すると、Snowflake はその列の値が近いデータを同じマイクロパーティションに集約します。

#### 具体例：売上テーブル（SALES）

数億件のデータを持つテーブルで、`REGION`（地域）と `SALE_DATE`（販売日）で頻繁にフィルタリングする場合を考えます。

```
【クラスタリング前】
REGION = 'Tokyo' のデータが全 1,000 パーティションに分散
→ クエリ実行時に 1,000 パーティション全スキャン（スキャン率 100%）

【クラスタリング後（REGION でクラスタリング）】
REGION = 'Tokyo' のデータが 50 パーティションに集約
→ クエリ実行時に 50 パーティションだけスキャン（スキャン率 5%）
```

設定方法：

```sql
-- 単一キー
ALTER TABLE sales CLUSTER BY (region);

-- 複合キー（複数カラム）
ALTER TABLE sales CLUSTER BY (region, sale_date);

-- クラスタリング状態の確認
SELECT SYSTEM$CLUSTERING_INFORMATION('sales', '(region, sale_date)');
```

---

### 6.3 Automatic Clustering（自動クラスタリング）

クラスタリングキーを設定すると、データの変更によって崩れたクラスタリング状態をバックグラウンドで自動的に修正する **Automatic Clustering** が有効になります。

| 項目 | 内容 |
|------|------|
| **管理** | 完全自動（ユーザーがデータを手動で並べ替える必要はない） |
| **コスト** | **Serverless 機能のためクレジットを消費**（クラスタリングメンテナンスの計算分） |
| **適用基準** | すべてのテーブルに設定すべきではない |

> **いつ導入を検討するか**
> - **マルチテラバイト級の巨大テーブル**であること
> - かつ、クエリプロファイルを確認してプルーニングが効かず**パフォーマンスが実際に課題**になっていること
>
> 小〜中規模テーブルや、ロード順と検索条件が一致するテーブルでは、デフォルトのパーティショニングで十分なパフォーマンスが出ることがほとんどです。

---

### 6.4 要点まとめ

| 項目 | 内容 |
|------|------|
| **対象** | 物理的なデータの並び順（マイクロパーティションの再構成） |
| **目的** | 不要なパーティションの読み飛ばし（プルーニング）による高速化 |
| **適した列** | WHERE 句・JOIN 句で頻繁に使われる列。カーディナリティが高すぎず低すぎない列（日付・地域など） |
| **コスト** | ストレージコスト ＋ 再クラスタリングの計算リソース（クレジット） |
| **導入ステップ** | まずクエリプロファイルでスキャン効率を確認 → 課題があれば検討 |

## 7. オブジェクト階層

> 🔗 **検証SQL**: [`verification/01_architecture/02_object_hierarchy.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/01_architecture/02_object_hierarchy.sql)

### 7.1 全体構造

```
Organization（組織）← オプション。複数アカウントを持つ大企業向け
└── Account（アカウント）← Snowflakeの基本契約単位
    ├── User（ユーザー）← ログインするアカウント
    ├── Role（ロール）← 権限をまとめたもの
    ├── Warehouse（ウェアハウス）← コンピュートリソース
    └── Database（データベース）← データの論理的なまとまり
        └── Schema（スキーマ）← テーブル等をグループ化
            ├── Table（テーブル）
            ├── View（ビュー）
            ├── Stage（ステージ）← データロード用の一時領域
            ├── File Format
            ├── Stored Procedure
            ├── UDF（ユーザー定義関数）
            ├── Stream（変更データキャプチャ）
            └── Task（スケジュール実行）
```

### 7.2 オブジェクト参照の書き方

```sql
-- 完全修飾名（フルパス）
SELECT * FROM database_name.schema_name.table_name;

-- コンテキスト設定後は省略可能
USE DATABASE sales_db;
USE SCHEMA public;
SELECT * FROM customers;  -- sales_db.public.customers と同じ
```

### 7.3 USE DATABASE / USE SCHEMA コマンド

`USE DATABASE` と `USE SCHEMA` は、**現在の作業対象（コンテキスト）を設定するコマンド**です。

> **なぜ使うのか？**
> テーブルを参照するたびに `sales_db.public.customers` のように完全なパスを書くのは手間がかかります。USE DATABASEとUSE SCHEMAを使ってコンテキストを設定しておくと、以降は短い名前でテーブルにアクセスできます。

```sql
-- データベースを設定（このセッションでのデフォルトDBになる）
USE DATABASE sales_db;

-- スキーマを設定（このセッションでのデフォルトスキーマになる）
USE SCHEMA public;

-- 設定後は短い名前でアクセス可能
SELECT * FROM customers;         -- sales_db.public.customers と同じ
SELECT * FROM orders;            -- sales_db.public.orders と同じ

-- 現在の設定を確認
SELECT CURRENT_DATABASE();       -- SALES_DB
SELECT CURRENT_SCHEMA();         -- PUBLIC
```

**ポイント**:
- コンテキストはセッションごとに独立している
- ワークシートごとにもUIから設定可能（Snowsightの上部ドロップダウン）
- コンテキストを設定しても、完全修飾名（`db.schema.table`）はいつでも使用可能

### 7.4 よく使うデフォルトスキーマ

データベースを作成すると、自動的に2つのスキーマが生成されます。

| スキーマ | 説明 | 削除・変更 |
|---------|------|-----------|
| `PUBLIC` | 空のスキーマ。自由にテーブル等を追加できる | **可能**（DROP / RENAME / MOVE が可） |
| `INFORMATION_SCHEMA` | データベースのメタデータを保持するシステムスキーマ | **不可**（読み取り専用） |

#### メタデータとは

**メタデータ = 「データに関するデータ」**。データ本体ではなく、データを説明する情報です。

```
例: CUSTOMERS テーブルのメタデータ
  - テーブル名: CUSTOMERS
  - 作成日: 2026-04-09
  - 行数: 50,000
  - カラム名: id, name, email, region
  - データ型: INT, VARCHAR(100), VARCHAR(200), VARCHAR(50)
  - オーナーロール: SYSADMIN
```

→ これらは INFORMATION_SCHEMA や SHOW TABLES などで確認できます。

### 7.5 データベースとスキーマの基本操作

```sql
-- データベースの作成
CREATE DATABASE GARDEN_PLANTS;

-- 自動作成されたPUBLICスキーマを削除（不要な場合）
DROP SCHEMA GARDEN_PLANTS.PUBLIC;

-- 新しいスキーマを作成
CREATE SCHEMA GARDEN_PLANTS.VEGGIES;
CREATE SCHEMA GARDEN_PLANTS.FRUITS;
CREATE SCHEMA GARDEN_PLANTS.FLOWERS;

-- 作成したリソースを確認
SHOW DATABASES;
SHOW SCHEMAS;
SHOW SCHEMAS IN ACCOUNT;  -- 全データベースのスキーマを一覧

-- スキーマの名前変更
ALTER SCHEMA GARDEN_PLANTS.WEGGIES RENAME TO GARDEN_PLANTS.VEGGIES;

-- スキーマを別のデータベースに移動
ALTER SCHEMA DEMO_DB.VEGGIES RENAME TO GARDEN_PLANTS.VEGGIES;
```

## 8. エディション

### 8.1 エディション比較

| エディション | 主な機能 | 適したユースケース |
|-------------|---------|-------------------|
| **Standard** | 基本機能、Time Travel 1日 | 開発・テスト、小規模利用 |

### 8.2 エディション選択のポイント

- **Standard**: コストを抑えたい、基本機能で十分
- **Enterprise**: 本番環境で高い可用性やセキュリティが必要
- **Business Critical**: コンプライアンス要件がある

## 9. Resource Monitor（リソースモニター）

### Resource Monitorとは

Warehouseのクレジット消費を監視し、**日次・月次などの上限を設けてコストを制御**する機能です。上限に達するとWarehouseが停止し、それ以上課金されなくなります。

> **なぜ必要か**: 大きいWarehouseを誤って長時間動かしてしまうと、想定外の高額請求が発生します。Resource Monitorは「クレジットが○○に達したら止める」という安全装置です。

### 設定場所

Admin → Cost Management（または Resource Monitors）

### 主な設定項目

| 設定 | 内容 | 例 |
|------|------|-----|
| **クレジット上限** | 指定期間内に消費できる最大クレジット数 | 1日あたり3クレジット |
| **監視対象** | アカウント全体 or 特定Warehouse | COMPUTE_WH のみ |
| **アクション** | 上限到達時の動作 | Warehouseを停止 |
| **通知** | メールや画面で警告 | 上限の75%で警告 |

### 上限到達時のエラー例

```
Warehouse 'COMPUTE_WH' cannot be resumed because resource monitor
'{X}' has exceeded its quota.
```

→ Resource Monitorの上限を引き上げるか、翌日（新しいクレジットが付与）まで待つ。

> **コスト感覚**: XS Warehouseは約1クレジット/時間（数ドル以下）。6XL Warehouseは512倍のコスト。**まずXSから始め、必要なら段階的にスケールアップ**することが推奨されています。
