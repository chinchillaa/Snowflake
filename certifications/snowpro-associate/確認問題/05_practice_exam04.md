- A) Temporary Stages（一時ステージ）
- B) External Stages（外部ステージ）
- C) Permanent Stages（永続ステージ）
- D) Managed Stages（管理対象ステージ）
- E) Internal Stages（内部ステージ）

### あなたの回答

```

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: B) External Stages（外部ステージ）、E) Internal Stages（内部ステージ）**

Snowflakeは、データのロード・アンロード用に2つの主要なステージをサポートしています。

- **内部ステージ（Internal Stages）**：Snowflakeシステム内部にあるストレージ領域です。これには以下の3種類が含まれます。
    
    - ユーザーステージ（`@~`）
        
    - テーブルステージ（`@%table_name`）
        
    - 名前付き内部ステージ（`@stage_name`）
        
- **外部ステージ（External Stages）**：Amazon S3、Azure Blob Storage、Google Cloud Storageなどの外部クラウドストレージを参照します。ストレージの場所と認証情報を指定する必要があります。
    

これらは、Snowflakeのテーブルへデータを出し入れする際の中間ストレージとして機能します。

❌ 「Managed（管理対象）」「Temporary（一時）」「Permanent（永続）」といったステージは公式な分類としては存在しません。内部ステージの挙動として一時的・永続的な使い方は可能ですが、ステージの種類名ではありません。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション7（オブジェクト階層）

</details>

<details>

Snowflakeの「ゼロコピークローン（Zero-Copy Cloning）」に関する記述として正しいものはどれですか？（2つ選択）

- A) オブジェクトの書き込み可能なコピー（複製）を作成する。
- B) ストレージ容量を大幅に追加で必要とする。
- C) 元のデータストレージを物理的に複製（二重化）する。
- D) 即座に完了する、メタデータ（管理情報）のみの操作である。

### あなたの回答

```
選択: []

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: A) オブジェクトの書き込み可能なコピー（複製）を作成する。、D) 即座に完了する、メタデータ（管理情報）のみの操作である。**

Snowflakeのゼロコピークローンは、基礎となる実データを複製することなく、データベース、スキーマ、テーブルの即時かつ書き込み可能なクローンを作成します。データブロックそのものをコピーするのではなく、元のデータへの「ポインタ（参照情報＝メタデータ）」を作成するため、以下のような特徴があります。

- **高速**：クローンは数秒で作成されます。
    
- **効率的**：クローン後、または元のデータに変更が加えられない限り、追加のストレージは消費されません。
    
- **書き込み可能**：クローンされたオブジェクトは、元のデータとは独立して変更（更新・削除など）が可能です。
    

これにより、開発、テスト、バックアップなどの作業に非常に適しています。

❌ データストレージを物理的に複製しません。

❌ したがって、大幅な追加ストレージも必要としません。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション7（オブジェクト階層）

</details>

### 実際に確認してみよう

**SQLでの確認:**
```sql
-- テーブルのゼロコピークローンを作成する
CREATE TABLE my_table_clone CLONE my_table;

-- データベースのゼロコピークローンを作成する
CREATE DATABASE my_db_clone CLONE my_database;

-- スキーマのゼロコピークローンを作成する
CREATE SCHEMA my_schema_clone CLONE my_schema;

-- クローンが独立していることを確認（クローン側を更新しても元は変わらない）
INSERT INTO my_table_clone VALUES (999, 'clone_test');
SELECT COUNT(*) FROM my_table;        -- 元テーブルの件数は変わらない
SELECT COUNT(*) FROM my_table_clone;  -- クローンのみ増加する
```

<details>
<summary>クリックして確認</summary>

**正解: A) 上位のロール（親ロール）が、下位のロールから権限を継承できるようにすること。**

Snowflakeのロールベースアクセス制御（RBAC）モデルでは、ロールの階層化によって「権限の継承」が可能になります。

- 上位のロールは、下位のロールに付与されているすべての権限を自動的に継承（引き継ぎ）します。
    
- これにより、アクセス制御をモジュール化し、拡張しやすくすることができます。例えば、上位の管理者ロール（`SYSADMIN`）が、特定の部門のロール（`ANALYST_ROLE`）の権限を内包するといった管理が可能です。
    
- 権限を重複して付与する手間が省け、権限管理がシンプルになります。
    

❌ SQLの実行順序、ウェアハウスのサイズ設定、ユーザーの単なるカテゴリ分けを目的としたものではありません。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション3（Virtual Warehouse）

</details>

**正解: A) データベースストレージ層 (Database Storage Layer)、D) クエリ処理層 (Query Processing Layer)、F) クラウドサービス層 (Cloud Services Layer)**

Snowflakeのアーキテクチャは、明確な役割を持つ3つの主要な層で構成されています。

- **データベースストレージ層**：すべてのデータを圧縮・列指向フォーマットでクラウドのオブジェクトストレージに永続的に保存します。

❌ **サードパーティ製のETLツール**：外部ツールとの連携は可能ですが、Snowflake自体のアーキテクチャの一部ではありません。

❌ **ユーザーインターフェース層**：Snowsightなどの画面はありますが、アーキテクチャの根幹をなす3層構造には含まれません。

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: C) ユーザーにはロールが割り当てられる。、D) 権限はロールに対して付与される。、F) ロールは他のロールに付与することができる（ロールの階層化）。**

SnowflakeのRBACモデルは、スケーラブルで安全なアクセス管理を行うために以下の原則に基づいています。

- **権限はロールに対して付与される**：ユーザー個人に直接権限を与えるのではなく、ロール（役割）に対して権限を付与することで管理を簡素化します。
    
- **ロールの階層化**：ロールを別のロールに付与することで階層を作り、上位のロールが権限を継承できるようにします。
    
- **ユーザーへの割り当て**：ユーザーには1つ以上のロールが割り当てられ、セッション中にアクティブにしたロールに基づいてアクセス権が決定されます。
    

❌ **ユーザー属性によるアクセス決定**：これは「属性ベースのアクセス制御（ABAC）」であり、Snowflakeの主要モデルではありません。

❌ **`ACCOUNTADMIN` のみがロールを作成できる**：`SECURITYADMIN` など、適切な権限を持つ他のロールでも作成可能です。

❌ **権限はユーザーにのみ直接付与される**：Snowflakeではユーザーに直接権限を付与することは推奨されておらず、原則としてロールを介して行います。

</details>

- B) JSONデータをXML形式に変換する。
- C) JSONデータをPARSE形式で保存する。
- D) 文字列のJSONデータを解析し、リレーショナルテーブルで扱える形式に変換する。

### あなたの回答

```
選択: []

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: D) 文字列のJSONデータを解析し、リレーショナルテーブルで扱える形式に変換する。**

（※問題48と同内容の質問です）

`PARSE_JSON('{"key": "value"}')` のように使用すると、JSONの文字列を `VARIANT` 型に変換し、ドット記法（`:key`）でSQLから中身の値を取り出せるようになります。すでにVARIANT型として格納されているデータには不要ですが、VARCHAR型で受け取ったJSON文字列を処理する際に活用します。

❌ **外部ステージからJSONデータをロードする**：外部ステージからのデータロードは `COPY INTO` コマンドが担当します。`PARSE_JSON` はすでに取り込んだ文字列の変換処理に使います。

❌ **JSONデータをXML形式に変換する**：フォーマット間の変換機能ではありません。文字列→VARIANTの型変換が目的です。

❌ **JSONデータをPARSE形式で保存する**：「PARSE形式」というデータ形式は存在しません。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション7（オブジェクト階層）

</details>

### 実際に確認してみよう

**SQLでの確認:**
```sql
-- PARSE_JSON関数でJSON文字列をVARIANT型に変換
SELECT PARSE_JSON('{"name": "Alice", "age": 30}') AS parsed;

-- VARCHAR列に格納されたJSON文字列をPARSE_JSONで変換して値を取得
CREATE TABLE raw_json_data (id INT, json_text VARCHAR(10000));
INSERT INTO raw_json_data VALUES
  (1, '{"product": "Apple", "price": 150, "tags": ["fruit", "fresh"]}'),
  (2, '{"product": "Banana", "price": 80, "tags": ["fruit", "yellow"]}');

-- PARSE_JSONを使ってVARCHARのJSON文字列からキーの値を取得
SELECT
  id,
  PARSE_JSON(json_text):product::STRING AS product_name,
  PARSE_JSON(json_text):price::INT AS price,
  PARSE_JSON(json_text):tags[0]::STRING AS first_tag
FROM raw_json_data;
```
