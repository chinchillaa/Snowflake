# summary.md 初学者向け充実化スキル

## 目的

既存の `summary.md` ファイルを読み込み、初学者がファイルを読むだけで十分に理解が深まるよう、理由・背景・比較・補足を追加して書き直す。(既存の `summary.md` ファイルがない場合は作成すること)

## いつ使うか

- summary.md の内容が事実の列挙にとどまっており、「なぜ」「どういう背景で」が不足している場合
- 初学者向けにわかりやすくしたい場合
- 技術的な制約や設計判断の理由を補足したい場合

## 実行手順

### Step 1: ソースの読み込み

1. 対象の `summary.md` を読み込む。または存在しない場合は対応する `lesson.md`を基に作成する
2. 対応する `lesson.md` も読み込む（summary.md だけでは背景情報が不足する場合があるため）
3. 必要に応じて同ディレクトリの `.sql` ファイルも参照する

### Step 2: 不足している情報の特定

以下の観点で既存の summary.md を分析し、不足を洗い出す:

| チェック観点 | 不足している場合の対応 |
|---|---|
| **理由・背景の説明** | 「なぜそうなるのか」「なぜこの方法を選ぶのか」を `>` 引用ブロックで補足する |
| **比較・対比** | 類似概念（ビュー vs マテリアライズドビュー等）の比較表を追加する |
| **用語の初出時の解説** | 技術用語の初出時に、その意味と「なぜ必要か」を丁寧に説明する |
| **構造・階層の可視化** | 複数のオブジェクトの関係がある場合、テキストベースの図や表で構造を示す |
| **実務との接続** | プロトタイプ vs 本番、ベストプラクティスとの関連を補足する |
| **前提知識の復習** | 初学者が躓きやすいポイントに、前提知識の簡単な復習を挟む |

### Step 3: 充実化の実行

以下のルールに従って summary.md を書き直す:

#### 構造ルール

- 既存の大構造（学習目標 → 学んだこと → DORA チェック）は維持する
- 各トピック（### 見出し）の中に補足情報を追加する形で充実させる
- トピックの順序はソースの lesson.md に従う

#### 補足の書き方

1. **理由・背景の補足** — `>` 引用ブロックを使い、「なぜ〜なのか:」で始める

   ```markdown
   > **なぜステージにマテリアライズドビューを直接配置できないのか:** マテリアライズドビューの自動リフレッシュ機能は、基礎となるテーブルの変更追跡に依存しています。ステージ上のファイルはSnowflakeのテーブルではないため、変更追跡の仕組みが適用できず、いつリフレッシュすべきか判断できません。
   ```

2. **比較表の追加** — 類似概念がある場合はマークダウンテーブルで対比する

   ```markdown
   | 特性 | 通常のビュー | マテリアライズドビュー |
   |---|---|---|
   | データの保存 | しない | する |
   | クエリ速度 | 元データに依存 | テーブルと同等 |
   ```

3. **用語解説の追加** — 初出の技術用語に `>` ブロックで補足する

   ```markdown
   > **`::varchar` とは:** Snowflakeの型キャスト構文です。VARIANT型のデータを明示的に文字列型に変換します。
   ```

4. **構造の可視化** — 複数オブジェクトの関係をテキストで図示する

   ```markdown
   ```
   ステージ（Parquetファイル）
       ↓ 参照
   外部テーブル（T_CHERRY_CREEK_TRAIL）
       ↓ 基盤
   セキュアマテリアライズドビュー（SMV_CHERRY_CREEK_TRAIL）
   ```
   ```

5. **実務との接続** — プロトタイプと本番の使い分けを補足する

   ```markdown
   > **ただし忘れてはいけないこと:** このアプローチはプロトタイプに最適ですが、本番運用ではパフォーマンス・保守性の観点から、テーブルへのロードが推奨されます。
   ```

#### テキスト表現ルール

- 日本語で記述する（「です・ます調」で統一）
- 技術用語は日本語訳の後に括弧で英語原語を併記する（初出時のみ）
- Snowflakeのオブジェクト名・キーワードは **太字** にする
- SQL は ` ```sql ` フェンスドコードブロックで囲む
- コードブロック内のSQLにはコメントを付けない（説明は外に書く）

#### 分量の目安

- 各トピック（### 見出し）につき、最低1つの `>` 引用ブロックによる補足を含める
- 比較可能な概念がある場合は必ず比較表を追加する
- DORA チェックの項目にも「何を確認しているのか」の一言説明を付ける

### Step 4: 整合性の確認

- lesson.md の内容と矛盾がないか確認する
- ソースにない内容を推測で追加していないか確認する
- 他のレッスンの summary.md と用語・表現の一貫性を確認する

## 注意事項

- ソースの lesson.md にない技術的主張を追加しない
- 補足はあくまで「理由・背景・文脈」であり、新しいトピックの追加ではない
- 既存の SQL コード例は変更しない（補足説明のみ追加する）
- 冗長になりすぎないよう、1つの引用ブロックは3〜5文以内に収める
