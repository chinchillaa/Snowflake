# データ管理とロード

> SOL-C01 対応 | 出題比率: 25%（最重要）

---

## 1. データロードの概要

### 1.1 データロードとは

Snowflakeにデータを取り込むことを「**データロード**」と呼びます。外部システムやファイルからSnowflakeのテーブルにデータを移す作業です。

```
┌──────────────────┐      ロード       ┌──────────────────┐
│   外部データ      │  ─────────────▶   │  Snowflakeテーブル │
│ (CSV, JSON等)    │                   │                  │
└──────────────────┘                   └──────────────────┘
```

### 1.2 ロード方法の比較

| 方法 | 特徴 | ユースケース |
|------|------|-------------|
| **COPY INTO** | バッチロード、高速 | 定期的な大量データロード |
| **Snowpipe** | 継続的、自動、サーバーレス | リアルタイムに近いロード |
| **INSERT** | 単一行/少量データ | 小規模データ、テスト |
| **Web UI** | GUIでアップロード | 小規模ファイル、検証用 |

### 1.3 どの方法を選ぶべきか

```
データロード方法の選択フローチャート
        │
        ▼
   データ量は多い？ ──No──▶ INSERT または Web UI
        │Yes
        ▼
 継続的に更新される？ ──No──▶ COPY INTO（バッチ処理）
        │Yes
        ▼
     Snowpipe（自動ロード）
```

**実際の使い分け例**:
- **日次の売上データを夜間に一括取込** → COPY INTO
- **IoTセンサーからのログが常時発生** → Snowpipe
- **1件のテスト用レコードを追加** → INSERT
- **Excelから100行のマスタデータをアップロード** → Web UI

## 2. ステージ（Stage）

> 🔗 **検証SQL**: [`verification/03_data_load/01_stages.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/01_stages.sql)

### 2.1 ステージとは何か

ステージは、データをSnowflakeテーブルにロードする前の「**一時的な置き場所**」です。

**なぜステージが必要なのか**：
1. **データ検証**: 本番テーブルに入れる前にデータの形式を確認できる
2. **効率的なロード**: ファイルをまとめてから一括でロードできる
3. **外部連携**: AWS S3やAzure Blobなど外部ストレージと接続できる

```
┌─────────────┐      PUT       ┌──────────────┐    COPY INTO    ┌──────────────┐
│ ローカルファイル │ ─────────▶ │    ステージ    │ ──────────────▶ │   テーブル    │
└─────────────┘               └──────────────┘                 └──────────────┘
```

### 2.2 ステージの種類

```
┌─────────────────────────────────────────────────┐
│                  Internal Stage                  │
│           （Snowflake内部のストレージ）            │
│  ┌───────────┬───────────┬───────────────────┐  │
│  │ User Stage│Table Stage│   Named Stage     │  │
│  │   @~      │   @%      │   @stage_name     │  │
│  │ 個人用    │テーブル専用│   汎用的         │  │
│  └───────────┴───────────┴───────────────────┘  │
├─────────────────────────────────────────────────┤
│                  External Stage                  │
│           （クラウドストレージを参照）             │
│  ┌─────────┬────────────┬────────────────────┐  │
│  │   S3    │ Azure Blob │        GCS         │  │
│  │  (AWS)  │ (Azure)    │   (Google Cloud)   │  │
│  └─────────┴────────────┴────────────────────┘  │
└─────────────────────────────────────────────────┘
```

### 2.3 Internal Stage（内部ステージ）の詳細

| 種類 | 参照方法 | 説明 | 作成方法 |
|------|---------|------|---------|
| **User Stage** | `@~` | ユーザーごとに1つ自動で存在 | 自動（削除・変更不可） |
| **Table Stage** | `@%table_name` | テーブルごとに1つ自動で存在 | 自動（テーブル作成時） |
| **Named Stage** | `@stage_name` | 自分で名前を付けて作成 | CREATE STAGEで作成 |

**記号の覚え方**:
- `@~` → チルダは「ホーム」を表す（ユーザーのホーム＝User Stage）
- `@%` → パーセントは「テーブル」に紐づく（%がテーブルを示す）
- `@name` → 自分で付けた名前をそのまま使う

```sql
-- Named Stageの作成
CREATE STAGE my_stage;

-- ファイル一覧表示
LIST @my_stage;           -- Named Stage
LIST @~;                  -- User Stage（自分専用）
LIST @%customers;         -- customersテーブル専用ステージ

-- ステージへファイルをアップロード（SnowSQL/クライアントから）
PUT file:///path/to/file.csv @my_stage;
```

### 2.4 どのステージを使うべきか

| シチュエーション | 推奨ステージ |
|-----------------|-------------|
| 個人的な一時作業 | User Stage (`@~`) |
| 特定テーブル専用のロード | Table Stage (`@%table`) |
| 複数テーブルで共有、チームで利用 | Named Stage |
| S3/Azure/GCSのファイルを読み込む | External Stage |

### 2.5 External Stage（外部ステージ）

外部のクラウドストレージを直接参照するステージです。データをSnowflake内部にコピーせずに、外部のファイルを読み取れます。

```sql
-- S3外部ステージの作成
CREATE STAGE my_s3_stage
  URL = 's3://mybucket/data/'
  STORAGE_INTEGRATION = my_s3_integration;

-- Azure外部ステージの作成
CREATE STAGE my_azure_stage
  URL = 'azure://myaccount.blob.core.windows.net/container/'
  STORAGE_INTEGRATION = my_azure_integration;
```

### 2.6 Storage Integration

外部ステージでクラウドストレージにアクセスするには認証が必要です。**Storage Integration**は、この認証情報を安全に管理するオブジェクトです。

**なぜStorage Integrationを使うのか**:
- **セキュリティ**: 認証情報（APIキーなど）をSQL文に直接書かなくて済む
- **管理性**: 認証情報の変更が一箇所で済む
- **再利用**: 複数のステージで同じIntegrationを使える

```sql
CREATE STORAGE INTEGRATION my_s3_integration
  TYPE = EXTERNAL_STAGE
  STORAGE_PROVIDER = 'S3'
  ENABLED = TRUE
  STORAGE_AWS_ROLE_ARN = 'arn:aws:iam::123456789012:role/myrole'
  STORAGE_ALLOWED_LOCATIONS = ('s3://mybucket/');
```

> **重要**: Storage Integrationの作成にはACCOUNTADMIN権限が必要です。

## 3. ファイルフォーマット

> 🔗 **検証SQL**: [`verification/03_data_load/02_file_formats.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/02_file_formats.sql)

### 3.1 サポートされる形式

| 形式 | 説明 | 特徴 | よく使うシーン |
|------|------|------|---------------|
| **CSV** | カンマ区切りテキスト | 人が読める、汎用的 | Excel出力、一般的なデータ交換 |
| **JSON** | 構造化テキスト | ネストしたデータ可 | API連携、ログデータ |
| **Parquet** | 列指向バイナリ | 圧縮効率が高い | 大量データ分析、データレイク |
| **Avro** | 行指向バイナリ | スキーマ埋め込み | Kafka連携、ストリーミング |
| **ORC** | 列指向バイナリ | Hive最適化 | Hadoop/Hiveからの移行 |
| **XML** | マークアップ言語 | 階層構造表現 | レガシーシステム連携 |

### 3.2 File Format オブジェクト

よく使うフォーマット設定を「File Formatオブジェクト」として保存できます。毎回同じ設定を書く手間が省けます。

```sql
-- CSVフォーマットの作成
CREATE FILE FORMAT my_csv_format
  TYPE = CSV
  FIELD_DELIMITER = ','      -- フィールドの区切り文字
  SKIP_HEADER = 1            -- 最初の1行（ヘッダー）をスキップ
  NULL_IF = ('NULL', 'null', '')  -- これらの文字をNULLとして扱う
  EMPTY_FIELD_AS_NULL = TRUE;     -- 空フィールドをNULLに

-- JSONフォーマットの作成
CREATE FILE FORMAT my_json_format
  TYPE = JSON
  STRIP_OUTER_ARRAY = TRUE;  -- 外側の配列括弧を除去

-- Parquetフォーマットの作成
CREATE FILE FORMAT my_parquet_format
  TYPE = PARQUET;
```

### 3.3 SHOW FILE FORMATS コマンド

現在のスキーマ（または指定したスキーマ）に存在するFile Formatオブジェクトの一覧を確認するコマンドです。

```sql
-- 現在のスキーマのFile Format一覧を表示
SHOW FILE FORMATS;

-- 特定スキーマのFile Format一覧を表示
SHOW FILE FORMATS IN SCHEMA sales_db.public;
```

**用途**:
- どのFile Formatオブジェクトが存在するか確認したい
- File Formatの名前・タイプ・設定を一覧で把握したい
- COPY INTOで使うFile Format名を確認する際に使用

### 3.3b よく使うFile Formatの実例

```sql
-- ① カンマ区切り（標準CSV）、ヘッダー1行スキップ
CREATE FILE FORMAT csv_with_header
  TYPE = 'CSV'
  FIELD_DELIMITER = ','
  SKIP_HEADER = 1;

-- ② パイプ区切り（| 区切り）、ヘッダー1行スキップ
--    ファイル例: VEG_NAME_TO_SOIL_TYPE_PIPE.txt
CREATE FILE FORMAT PIPECOLSEP_ONEHEADROW
  TYPE = 'CSV'
  FIELD_DELIMITER = '|'
  SKIP_HEADER = 1;

-- ③ タブ区切り（TSV）
CREATE FILE FORMAT tsv_format
  TYPE = 'CSV'
  FIELD_DELIMITER = '\t'
  SKIP_HEADER = 1;

-- ② を使ってCOPY INTO
COPY INTO VEGETABLE_DETAILS_SOIL_TYPE
FROM @MY_INTERNAL_STAGE
FILES = ('VEG_NAME_TO_SOIL_TYPE_PIPE.txt')
FILE_FORMAT = (FORMAT_NAME = PIPECOLSEP_ONEHEADROW);
```

> **区切り文字の種類**: CSV（カンマ）、TSV（タブ）、パイプ（|）は実務でよく登場します。ファイルを開いて実際の区切り文字を確認してから File Format を作成しましょう。

### 3.4 主なCSVオプション

| オプション                        | 説明               | デフォルト | 設定例                 |     |
| ---------------------------- | ---------------- | ----- | ------------------- | --- |
| FIELD_DELIMITER              | フィールド区切り文字       | `,`   | `'\t'`（タブ）、`'       | '`  |
| RECORD_DELIMITER             | レコード（行）区切り文字     | `\n`  | `'\r\n'`（Windows形式） |     |
| SKIP_HEADER                  | スキップするヘッダー行数     | 0     | `1`                 |     |
| FIELD_OPTIONALLY_ENCLOSED_BY | 引用符（フィールドを囲む文字）  | なし    | `'"'`（ダブルクォート）      |     |
| NULL_IF                      | NULLとして扱う文字列のリスト | `\\N` | `('NULL', '')`      |     |
| COMPRESSION                  | 圧縮形式             | AUTO  | `GZIP`, `BZIP2`     |     |

### 3.5 よくある間違い

1. **ヘッダー行のスキップ忘れ**: CSVの1行目がカラム名の場合、`SKIP_HEADER = 1`が必要
2. **区切り文字の間違い**: タブ区切りファイル（TSV）なのにカンマを指定
3. **文字コードの問題**: UTF-8以外のファイルは事前に変換が必要

## 4. COPY INTO コマンド

> 🔗 **検証SQL**: [`verification/03_data_load/03_copy_into.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/03_copy_into.sql)

### 4.1 COPY INTOとは

ステージにあるファイルをテーブルにロードする（または逆にテーブルからステージにエクスポートする）コマンドです。

### 4.2 データロード（ステージ → テーブル）

```sql
-- 基本構文
COPY INTO target_table
FROM @stage_name
FILE_FORMAT = (FORMAT_NAME = 'my_format');

-- インラインでフォーマットを指定（File Formatオブジェクトを使わない場合）
COPY INTO customers
FROM @my_stage/data/
FILE_FORMAT = (TYPE = CSV FIELD_DELIMITER = '|' SKIP_HEADER = 1);

-- ファイル名のパターンで絞り込み
COPY INTO sales
FROM @my_stage
PATTERN = '.*sales.*[.]csv';  -- "sales"を含むCSVファイルのみ
```

### 4.3 データアンロード（テーブル → ステージ）

```sql
-- テーブル全体をエクスポート
COPY INTO @my_stage/export/
FROM customers
FILE_FORMAT = (TYPE = CSV HEADER = TRUE);

-- クエリ結果をエクスポート
COPY INTO @my_stage/results/
FROM (SELECT * FROM sales WHERE year = 2026)
FILE_FORMAT = (TYPE = PARQUET);
```

### 4.4 主なオプション

| オプション | 説明 | 使用例 |
|-----------|------|--------|
| `FILES` | 特定のファイルを指定 | `FILES = ('file1.csv', 'file2.csv')` |
| `PATTERN` | 正規表現でファイルを指定 | `PATTERN = '.*[.]csv'` |
| `ON_ERROR` | エラー時の動作 | 下記参照 |
| `PURGE` | ロード後にファイル削除 | `PURGE = TRUE` |
| `FORCE` | 既にロード済みでも再ロード | `FORCE = TRUE` |
| `VALIDATION_MODE` | 検証のみ（ロードしない） | 下記参照 |

### 4.5 ON_ERROR オプション（重要！）

データに問題があった場合の動作を指定します。**試験頻出**。

| 値 | 動作 | 使いどころ |
|----|------|-----------|
| `ABORT_STATEMENT` | エラーで即座に中止 | **デフォルト**、厳格なデータ品質が必要な場合 |
| `CONTINUE` | エラー行をスキップして続行 | 多少の欠損を許容できる場合 |
| `SKIP_FILE` | エラーのあるファイルをスキップ | ファイル単位で品質管理したい場合 |
| `SKIP_FILE_n` | n個以上のエラーでファイルスキップ | `SKIP_FILE_10`（10件以上でスキップ） |
| `SKIP_FILE_n%` | n%以上のエラーでファイルスキップ | `SKIP_FILE_5%`（5%以上でスキップ） |

```sql
-- エラー行をスキップして続行
COPY INTO customers
FROM @my_stage
FILE_FORMAT = (TYPE = CSV)
ON_ERROR = CONTINUE;
```

### 4.6 VALIDATION_MODE

データを実際にロードせず、エラーがないか検証するモードです。本番ロード前のテストに有効。

```sql
-- データをロードせずに検証のみ
COPY INTO customers
FROM @my_stage
FILE_FORMAT = (TYPE = CSV)
VALIDATION_MODE = RETURN_ALL_ERRORS;
```

| 値 | 説明 |
|----|------|
| `RETURN_n_ROWS` | 最初のn行を検証して返す |
| `RETURN_ERRORS` | エラー行のみ返す |
| `RETURN_ALL_ERRORS` | すべてのエラーを返す |

## 5. Snowpipe

> 🔗 **検証SQL**: [`verification/03_data_load/04_snowpipe.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/04_snowpipe.sql)

### 5.1 Snowpipeとは

**Snowpipe**は、ファイルがステージに到着すると自動的にロードを開始する**継続的データロード**サービスです。

```
┌─────────────┐    アップロード    ┌──────────┐    自動検知     ┌──────────┐
│ 外部システム │ ───────────────▶ │  S3/Azure │ ─────────────▶ │ Snowpipe │
└─────────────┘                  └──────────┘                 └──────────┘
                                                                   │
                                                           自動でCOPY INTO実行
                                                                   ▼
                                                              ┌──────────┐
                                                              │ テーブル  │
                                                              └──────────┘
```

### 5.2 COPY INTO との比較

| 項目 | COPY INTO | Snowpipe |
|------|-----------|----------|
| 実行方法 | 手動実行またはスケジュール | ファイル到着で自動実行 |
| Warehouse | **必要** | **不要**（サーバーレス） |
| 課金 | Warehouseの実行時間 | 処理データ量に基づく |
| 遅延 | 実行タイミング次第 | 数分以内（ニアリアルタイム） |
| 向いている処理 | 大量データの一括ロード | 継続的に発生するデータ |

### 5.3 作成方法

```sql
-- Snowpipeの作成
CREATE PIPE my_pipe
  AUTO_INGEST = TRUE    -- クラウドイベントで自動起動
  AS
  COPY INTO target_table
  FROM @my_external_stage
  FILE_FORMAT = (TYPE = CSV);
```

### 5.4 トリガー方法

| 方法 | 説明 | 設定 |
|------|------|------|
| **AUTO_INGEST** | クラウドイベント通知で自動起動 | 推奨、要クラウド側設定 |
| **REST API** | insertFiles APIを手動呼び出し | プログラムから制御したい場合 |

### 5.5 クラウド別の連携設定

| クラウド | イベントサービス | 設定内容 |
|---------|-----------------|---------|
| AWS | S3 Event Notifications → SQS | S3バケットにイベント通知を設定 |
| Azure | Event Grid | Blobストレージにイベントを設定 |
| GCP | Pub/Sub | GCSバケットに通知を設定 |

### 5.6 監視コマンド

```sql
-- パイプの状態確認
SELECT SYSTEM$PIPE_STATUS('my_pipe');

-- ロード履歴（過去24時間）
SELECT * FROM TABLE(INFORMATION_SCHEMA.COPY_HISTORY(
  TABLE_NAME => 'target_table',
  START_TIME => DATEADD(hours, -24, CURRENT_TIMESTAMP())
));
```

### 5.7 Snowpipeの課金

Snowpipeは「**サーバーレスコンピュート**」で課金されます。
- Warehouse費用は不要
- 処理したファイル数とデータ量に基づく
- ファイルが小さすぎると非効率（まとめてからロードする方が効率的な場合も）

## 6. テーブルの種類

> 🔗 **検証SQL**: [`verification/03_data_load/05_tables.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/05_tables.sql)

### 6.1 3種類のテーブル

| 種類 | Time Travel | Fail-safe | 特徴 |
|------|:-----------:|:---------:|------|
| **Permanent** | 0-90日 | 7日 | デフォルト、本番データ向け |
| **Transient** | 0-1日 | **なし** | ステージングデータ向け |
| **Temporary** | 0-1日 | **なし** | セッション終了で消える |

### 6.2 使い分けの指針

```
テーブル種類の選択
      │
      ▼
 本番データとして永続化が必要？ ──Yes──▶ Permanent（デフォルト）
      │No
      ▼
 セッション内だけで使う一時データ？ ──Yes──▶ Temporary
      │No
      ▼
   Transient（中間処理、ETLステージング等）
```

**実際の使用例**:
- **Permanent**: 顧客マスタ、売上データ、トランザクションログ
- **Transient**: ETLの中間テーブル、集計前の一時テーブル
- **Temporary**: クエリ内の一時結果保存、セッション中の作業テーブル

### 6.3 作成構文

```sql
-- Permanentテーブル（デフォルト）
CREATE TABLE customers (
  id INT,
  name STRING
);

-- Transientテーブル
CREATE TRANSIENT TABLE staging_data (
  raw_data VARIANT
);

-- Temporaryテーブル
CREATE TEMPORARY TABLE temp_results (
  calc_value NUMBER
);
```

### 6.4 コスト面での注意

- **Permanent**: Time TravelとFail-safeの分、ストレージコストが高い
- **Transient/Temporary**: ストレージコストを抑えられるが、データ保護が弱い
- 大量の一時データを扱う場合はTransient/Temporaryを活用してコスト削減

## 7. 半構造化データ

> 🔗 **検証SQL**: [`verification/03_data_load/06_semi_structured.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/06_semi_structured.sql)

### 7.1 VARIANTデータ型

JSONやAvroなどの**半構造化データ**を格納できるデータ型です。スキーマ（カラム定義）が固定でないデータを柔軟に扱えます。

> **重要（試験頻出）**: VARIANT 型に格納できる値は **最大 16 MB** まで。これを超えるデータは格納できません。
> 関連するデータ型（OBJECT、ARRAY）についての詳細は `08_データ型_SQL基礎.md` セクション1.5 を参照。

```sql
CREATE TABLE events (
  id INT,
  event_data VARIANT  -- JSON等をそのまま格納
);

-- JSONデータの挿入
INSERT INTO events VALUES (
  1,
  PARSE_JSON('{"user": {"name": "田中", "age": 30}, "action": "login"}')
);
```

### 7.2 なぜVARIANTを使うのか

1. **スキーマの柔軟性**: 各行で異なる構造のデータを持てる
2. **ETLの簡略化**: 生のJSONをそのまま格納し、後で加工できる
3. **APIレスポンス格納**: 形式が頻繁に変わるデータに対応

### 7.3 データアクセス方法

```sql
-- サンプルデータ
-- event_data: {"user": {"name": "田中", "age": 30}, "items": [{"product": "A", "qty": 2}]}

-- ドット記法（推奨）
SELECT event_data:user:name FROM events;
-- 結果: "田中"

-- ブラケット記法
SELECT event_data['user']['name'] FROM events;
-- 結果: "田中"

-- 配列アクセス（0から始まるインデックス）
SELECT event_data:items[0]:product FROM events;
-- 結果: "A"

-- 型変換（::で指定）
SELECT event_data:user:name::STRING FROM events;
-- 結果: 田中（引用符なし）
```

### 7.3b JSONデータのクエリ実例

```sql
-- VARIANT型カラムからパス記法で値を取得し、VARCHARにキャスト
SELECT
  raw_author:FIRST_NAME::VARCHAR AS FIRST_NAME,
  raw_author:LAST_NAME::VARCHAR  AS LAST_NAME
FROM AUTHOR_INGEST_JSON;

-- ネストしたJSONをたどる（コロンで階層を区切る）
-- JSON例: {"user": {"name": "田中", "age": 30}}
SELECT
  raw_data:user:name::VARCHAR AS user_name,
  raw_data:user:age::INT      AS user_age
FROM my_json_table;
```

> **記法の覚え方**: `:` はJSONの階層をたどるナビゲーター。`::` はキャスト（型変換）。混同しないように注意。

### 7.4 FLATTEN関数

配列データを複数行に展開する関数です。1件のJSONに複数の商品が含まれる場合などに使います。

```sql
-- 元データ: {"items": [{"product": "A", "qty": 2}, {"product": "B", "qty": 1}]}

-- FLATTENで配列を行に展開
SELECT
  e.id,
  f.value:product::STRING AS product,
  f.value:qty::INT AS quantity
FROM events e,
LATERAL FLATTEN(input => e.event_data:items) f;

-- 結果:
-- | id | product | quantity |
-- |----|---------|----------|
-- | 1  | A       | 2        |
-- | 1  | B       | 1        |
```

#### FLATTENの実例：SNS投稿のハッシュタグ展開

```sql
-- JSON例（tweet_ingest テーブル）:
-- raw_status: {"entities": {"hashtags": [{"text": "Snowflake"}, {"text": "SQL"}]}}

-- FLATTENを使用して、配列を1行ずつに展開する
SELECT
  value:text::VARCHAR AS hashtag_used
FROM tweet_ingest,
LATERAL FLATTEN(input => raw_status:entities:hashtags);

-- 結果:
-- | hashtag_used |
-- |--------------|
-- | Snowflake    |
-- | SQL          |
```

> **LATERAL の意味**: 同じ FROM 句内の他のテーブルの値（`raw_status`）を参照するために必要なキーワードです。「横から参照する」イメージ。

### 7.5 FLATTENが返す列名（★試験頻出）

FLATTEN関数は、元の値だけでなく、配列の位置情報なども含む複数の列を返します。

| 列名 | 説明 | 例 |
|------|------|-----|
| **VALUE** | 展開された要素の値 | `{"product": "A", "qty": 2}` |
| **KEY** | オブジェクト内のキー名（MAP展開時） | `"product"` |
| **INDEX** | 配列内のインデックス（0から始まる） | `0`, `1`, `2` |
| **SEQ** | 入力行の通し番号（一意識別用） | `1`, `2`, `3` |
| **PATH** | 展開した要素のパス | `[0]`, `[1]` |
| **THIS** | 現在処理中の要素全体（元の構造を保持） | 元のJSON要素 |

> **覚え方**: "**V**ery **K**ind **I**nstructors **S**hare **P**ractice **T**ips" → VALUE, KEY, INDEX, SEQ, PATH, THIS

```sql
-- すべての列を確認する例
SELECT
  f.SEQ,
  f.INDEX,
  f.KEY,
  f.PATH,
  f.VALUE,
  f.THIS
FROM events e,
LATERAL FLATTEN(input => e.event_data:items) f;
```


>

## 9. データ変換（ロード時の加工）

### 9.1 ロード時に変換を行う

COPY INTOでは、ロードしながらデータを変換できます。

```sql
-- カラムの選択と変換
COPY INTO customers (id, name, created_at)
FROM (
  SELECT
    $1::INT,                                    -- 1列目をINTに変換
    UPPER($2),                                  -- 2列目を大文字に
    TO_TIMESTAMP($3, 'YYYY-MM-DD HH24:MI:SS')   -- 3列目をタイムスタンプに
  FROM @my_stage

```

| 参照 | 説明 | 例 |
|------|------|-----|
| `$1, $2, $3...` | CSVの列番号（1から始まる） | `$1::INT` |
| `$1:key` | JSONのキー参照 | `$1:user:name` |

### 9.3 変換の活用例

```sql

TRIM($2)
```

## 10. 非ロードデータパターン（データレイク）

> 🔗 **検証SQL**: [`verification/03_data_load/07_non_loaded_data.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/07_non_loaded_data.sql)

### 10.1 非ロードデータとは

**非ロードデータ（Non-loaded Data）** とは、テーブルにコピーせずにステージ上のファイルへ直接アクセスするデータです。プロトタイプ構築や素早いデータ探索に特に有効です。

```
ステージ上のファイル
  ↓ ファイルフォーマット適用
  ↓ $1, $2... で列参照
  ↓ REPLACE(), TRIM() でクレンジング
  ↓ ビューとして定義
  → テーブルと同様にSELECT可能
```

### 10.2 ディレクトリテーブル（Directory Table）

`DIRECTORY(@stage_name)` 関数でステージ内のファイル一覧をテーブルとして取得できます。

```sql
-- ディレクトリテーブルへのクエリ
SELECT * FROM DIRECTORY(@my_stage);
```

| カラム | 説明 |
|--------|------|
| `RELATIVE_PATH` | ファイルの相対パス |
| `FILE_URL` | ファイルへの参照URL |
| `LAST_MODIFIED` | 最終更新日時 |
| `SIZE` | ファイルサイズ（バイト） |

```sql
-- 非構造化データ（画像など）の一時的な署名付きURLを生成
SELECT
  RELATIVE_PATH,
  GET_PRESIGNED_URL(@image_stage, RELATIVE_PATH) AS image_url
FROM DIRECTORY(@image_stage);
```

> **ディレクトリテーブルと通常テーブルのJOIN:** `SWEATSUITS` テーブルとディレクトリテーブルをJOINすることで、商品マスタデータと画像URLを統合したカタログビューを作れます。

### 10.3 データレイクハウス（Data Lakehouse）

| アプローチ | 特徴 | 限界 |
|-----------|------|------|
| **データウェアハウス** | 構造化データの高速分析 | 非構造化データ対応困難 |
| **データレイク** | あらゆる形式のデータを安価に保存 | クエリが遅い、品質管理困難 |
| **データレイクハウス** | ロード済みデータと非ロードデータの組み合わせ | — |

> 🔗 **検証SQL**: [`verification/03_data_load/08_metadata_columns.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/08_metadata_columns.sql)

### 11.1 ロード時に使えるメタデータカラム

```sql
COPY INTO ed_pipeline_logs
```

## 12. MERGE文（差分ロード）

> 🔗 **検証SQL**: [`verification/03_data_load/09_merge.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/09_merge.sql)

### 12.1 MERGEとは

MERGE文はソースデータとターゲットテーブルを比較し、「一致した行は更新」「一致しない行は挿入」という差分ロードを1つのSQL文で実現します。

```sql
MERGE INTO target_table t
USING source_query s
ON t.key1 = s.key1
AND t.key2 = s.key2
WHEN MATCHED THEN
    UPDATE SET t.col = s.col
WHEN NOT MATCHED THEN
    INSERT (col1, col2) VALUES (s.col1, s.col2);
```

### 12.2 Update Merge vs Insert Merge

| 種類 | WHEN句 | 動作 | ユースケース |
|------|--------|------|-------------|
| **Update Merge** | `WHEN MATCHED` | 既存行を更新 | データの修正・上書き |
| **Insert Merge** | `WHEN NOT MATCHED` | 新規行のみ挿入 | 差分ロード（重複防止） |
| **アップサート** | 両方 | 更新と挿入を同時 | UPSERT（存在すれば更新、なければ挿入） |

### 12.3 べき等性（IDEMPOTENCY）

**べき等** = 何回実行しても同じ結果になる性質

| 操作 | べき等か | 理由 |
|------|----------|------|
| 単純 INSERT | **いいえ** | 実行のたびに重複行が増える |
| Insert MERGE | **はい** | NOT MATCHED の行のみ挿入される |
| TRUNCATE + COPY INTO | **はい** | 毎回全データをリセットして再ロード |

> **自動化パイプラインでべき等性が重要な理由:** タスクは障害時に再実行されることがあります。べき等でないパイプラインは再実行のたびに重複データを生み、データ品質を損ないます。

### 12.4 実践例：地理情報エンリッチメントを伴うInsert Merge

```sql
MERGE INTO enhanced.logs_enhanced e
USING (
    SELECT
        r.user_login,
        r.datetime_iso8601,
        r.user_event,
        loc.city,
        convert_timezone('UTC', tz.time_zone, r.datetime_iso8601) AS game_event_ltz,
        tod.tod_name
    FROM raw.pl_logs r
    JOIN ipinfo_geoloc.demo.location loc
      ON ipinfo_geoloc.public.to_join_key(r.ip_address) = loc.join_key
    JOIN raw.time_of_day_lu tod
      ON HOUR(game_event_ltz) = tod.hour
) s
ON s.user_login       = e.gamer_name
AND s.datetime_iso8601 = e.game_event_utc
AND s.user_event       = e.game_event_name
WHEN NOT MATCHED THEN
INSERT (gamer_name, game_event_utc, game_event_name, city, game_event_ltz, tod_name)
VALUES (s.user_login, s.datetime_iso8601, s.user_event, s.city, s.game_event_ltz, s.tod_name);
```

## 13. Snowflakeタスク（Task）

> 🔗 **検証SQL**: [`verification/03_data_load/10_tasks.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/10_tasks.sql)

### 13.1 タスクとは

**タスク**はSQL文をスケジュール実行するオブジェクトです。データパイプラインの自動化に使います。

```sql
CREATE OR REPLACE TASK my_task
    WAREHOUSE = 'COMPUTE_WH'
    SCHEDULE  = '5 minute'
AS
    <SQL文>;
```

### 13.2 タスクの操作コマンド

| 操作 | コマンド |
|------|---------|
| 手動実行 | `EXECUTE TASK <名前>` |
| スケジュール開始 | `ALTER TASK <名前> RESUME` |
| スケジュール停止 | `ALTER TASK <名前> SUSPEND` |
| タスク一覧 | `SHOW TASKS IN ACCOUNT` |
| 実行履歴確認 | `TABLE(INFORMATION_SCHEMA.TASK_HISTORY(task_name => '<名前>'))` |

> **権限:** `GRANT EXECUTE TASK ON ACCOUNT TO ROLE SYSADMIN;`

### 13.3 タスクの依存関係（DAG）

**ルートタスク**が完了したら**依存タスク**を自動起動する「タスクチェーン」を構築できます。

```sql
-- ルートタスク（スケジュール実行 / サーバーレスコンピュート使用）
CREATE OR REPLACE TASK get_new_files
    USER_TASK_MANAGED_INITIAL_WAREHOUSE_SIZE = 'XSMALL'
    SCHEDULE = '5 Minutes'
AS
    COPY INTO raw.pl_game_logs FROM @ags_game_audience.raw.uni_kishore_pipeline;

-- 依存タスク（ルートタスク完了後に自動起動）
CREATE OR REPLACE TASK load_logs_enhanced
    USER_TASK_MANAGED_INITIAL_WAREHOUSE_SIZE = 'XSMALL'
    AFTER get_new_files               -- ← 依存関係の設定
AS
    MERGE INTO enhanced.logs_enhanced ...;

-- 依存タスクを先に、ルートタスクを後にRESUMEする（順番が重要）
ALTER TASK load_logs_enhanced RESUME;
ALTER TASK get_new_files       RESUME;
```

> **DAG（Directed Acyclic Graph）:** タスク依存チェーンはDAGとも呼ばれます。データエンジニアリングで広く使われる概念です。

### 13.4 ウェアハウスタスク vs サーバーレスタスク

| 設定 | 用途 | 特徴 |
|------|------|------|
| `WAREHOUSE = 'COMPUTE_WH'` | 指定ウェアハウスで実行 | 大量データ処理向き |
| `USER_TASK_MANAGED_INITIAL_WAREHOUSE_SIZE = 'XSMALL'` | サーバーレスコンピュート | 頻繁な小さいタスク向き、常時スピンアップ不要 |

**サーバーレスが向いている理由:** 5分ごとに実行するタスクにウェアハウスを使うと、自動サスペンドされる前に次の実行が始まるため永遠にサスペンドされず、クレジットを大量消費します。

## 14. ストリーム（Stream）とCDC

> 🔗 **検証SQL**: [`verification/03_data_load/11_streams_cdc.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/03_data_load/11_streams_cdc.sql)

### 14.1 ストリームとは

**ストリーム** はテーブルへの変更（INSERT/UPDATE/DELETE）を追跡する**変更データキャプチャ（CDC: Change Data Capture）** オブジェクトです。

```sql
-- ストリームの作成
CREATE OR REPLACE STREAM my_stream
ON TABLE source_table;

-- 変更データが存在するか確認（TRUE/FALSE）
SELECT SYSTEM$STREAM_HAS_DATA('my_stream');

-- ストリームの変更データを取得
SELECT * FROM my_stream;
```

### 14.2 ストリームのメタカラム

| カラム | 説明 |
|--------|------|
| `METADATA$ACTION` | INSERT または DELETE |
| `METADATA$ISUPDATE` | UPDATEの場合 TRUE |
| `METADATA$ROW_ID` | 行の一意識別子 |

### 14.3 ストリームでパイプラインを効率化する仕組み

```
全行スキャン型タスク（非効率）:
  毎回ターゲットの全行とソースの全行を比較 → データ増加とともに遅くなる

ストリーム＋タスク型（効率的、CDC）:
  前回タスク実行後の変更分だけを取得 → 常に差分のみを処理
```

```sql
-- 変更がある時だけ実行するタスク（WHEN句でストリームをチェック）
CREATE OR REPLACE TASK process_changes
    USER_TASK_MANAGED_INITIAL_WAREHOUSE_SIZE = 'XSMALL'
    SCHEDULE = '5 Minutes'
    WHEN SYSTEM$STREAM_HAS_DATA('ed_cdc_stream')
AS
    MERGE INTO enhanced.logs_enhanced t
    USING (SELECT * FROM raw.ed_cdc_stream) s
    ON t.gamer_name = s.user_login
    AND t.game_event_utc = s.datetime_iso8601
    AND t.game_event_name = s.user_event
    WHEN NOT MATCHED AND s.METADATA$ACTION = 'INSERT' THEN
        INSERT (...) VALUES (...);
```

### 14.4 パイプライン方式の比較

| 方式 | 仕組み | メリット | デメリット |
|------|--------|---------|-----------|
| **時間駆動** | `SCHEDULE = '5 Minutes'` で定期実行 | シンプル | データがなくても実行されコスト発生 |
| **イベント駆動** | Snowpipeの `AUTO_INGEST = TRUE` | ファイル到着で即時起動 | クラウド側設定が必要 |
| **CDC（変更検知）** | Stream + `WHEN SYSTEM$STREAM_HAS_DATA()` | 差分のみ処理で効率的 | 設計が複雑 |

## 15. よくある間違いと注意点

### 10.1 ステージ関連

| 間違い | 正しい理解 |
|--------|-----------|
| User Stageを他のユーザーと共有できる | User Stage(`@~`)は個人専用。共有にはNamed Stageを使う |
| Table Stageを明示的に作成する | Table Stageはテーブル作成時に自動で作られる |
| Web UIからPUTコマンドを実行する | PUT/GETはSnowSQL/コネクターからのみ実行可能 |

### 10.2 COPY INTO関連

| 間違い | 正しい理解 |
|--------|-----------|
| COPY INTOは同じファイルを何度もロードする | デフォルトではロード済みファイルはスキップ（FORCEで上書き可） |
| ON_ERRORのデフォルトはCONTINUE | デフォルトはABORT_STATEMENT（エラーで停止） |
| VALIDATION_MODEでもデータがロードされる | 検証のみで、実際のロードは行われない |

### 10.3 Snowpipe関連

| 間違い | 正しい理解 |
|--------|-----------|
| SnowpipeにはWarehouseが必要 | Snowpipeはサーバーレス、Warehouse不要 |
| Snowpipeはリアルタイム | ニアリアルタイム（通常数分の遅延） |
| AUTO_INGESTは何も設定せず動く | クラウド側でイベント通知の設定が必要 |

## 16. 理解度チェック

### Q1. ステージの参照
User Stage、Table Stage、Named Stageの参照方法をそれぞれ答えてください。

<details>
<summary>解答</summary>

- User Stage: `@~`
- Table Stage: `@%テーブル名`（例: `@%customers`）

### Q2. ON_ERRORのデフォルト
COPY INTOでON_ERRORを指定しない場合、デフォルトの動作は何ですか？

<details>
<summary>解答</summary>

`ABORT_STATEMENT` - エラーが発生すると即座に中止します。
</details>

### Q3. Snowpipeの特徴
SnowpipeとCOPY INTOの違いについて、以下の観点で説明してください。
1. Warehouseの必要性
2. 実行タイミング

<details>
<summary>解答</summary>

1. **Warehouse**: COPY INTOは必要、Snowpipeは不要（サーバーレス）
2. **実行タイミング**: COPY INTOは手動/スケジュール実行、Snowpipeはファイル到着で自動実行
</details>

Fail-safeが適用されるテーブルの種類はどれですか？

**Permanent**テーブルのみ。Transient/TemporaryにはFail-safeがありません。
</details>
