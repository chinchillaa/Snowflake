## 1. 外部テーブル（External Table）

> 🔗 **検証SQL**: [`verification/09_advanced_objects/01_external_table.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/09_advanced_objects/01_external_table.sql)

### 1.1 外部テーブルとは

ステージ上のファイルを**テーブルオブジェクト**として扱えるオブジェクトです。データをSnowflakeにロードせず、外部ストレージのファイルに直接クエリできます。

```sql
CREATE OR REPLACE EXTERNAL TABLE t_cherry_creek_trail(
    my_filename VARCHAR(100) AS (METADATA$FILENAME::VARCHAR(100))
)
LOCATION   = @trails_parquet
AUTO_REFRESH = TRUE
FILE_FORMAT = (TYPE = PARQUET);
```

### 1.2 ビューとの構文・機能の違い

| 項目 | ビュー（View） | 外部テーブル（External Table） |
|------|--------------|-------------------------------|
| 列定義の順序 | `$1:field::type AS 列名`（定義→名前） | `列名 型 AS (定義)`（名前→定義） |
| データ変更の検知 | なし | **AUTO_REFRESH** で自動検知可能 |
| マテリアライズドビューの基盤 | **なれない** | **なれる** |
| Snowflakeへのロード | 不要（毎回ステージを参照） | 不要（毎回ステージを参照） |

### 1.3 なぜ外部テーブルが存在するのか

| 理由 | 詳細 |
|------|------|
| **セキュリティ/ガバナンス** | 規制によりデータを別の場所にコピーできない |
| **複数コピーの回避** | S3とSnowflakeに同じデータを持つとストレージコストが二重になる |
| **ベンダーロックイン回避** | Snowflake固有形式のみでのデータ保有を避けたい |
| **マルチプラットフォーム** | Spark、Prestoなど他システムからも同じデータにアクセスしたい |

> **保存場所:** 外部テーブルのデータは通常 AWS S3、Azure Blob Storage、GCS 上に置かれます。

## 2. マテリアライズドビュー（Materialized View）

> 🔗 **検証SQL**: [`verification/09_advanced_objects/02_materialized_view.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/09_advanced_objects/02_materialized_view.sql)

### 2.1 通常のビューとの違い

| 特性 | 通常のビュー | マテリアライズドビュー |
|------|------------|---------------------|
| データの保存 | しない（毎回再計算） | **する**（クエリ結果をキャッシュ） |
| クエリ速度 | 元データの複雑さに依存 | テーブルと同等に高速 |
| 自動更新 | 常に最新（再計算のため） | 基礎データ変更時にSnowflakeが自動リフレッシュ |
| 追加コスト | なし | ストレージ＋自動リフレッシュの計算コスト |

```sql
-- 通常のマテリアライズドビュー
CREATE MATERIALIZED VIEW mv_region_sales AS
  SELECT region, SUM(amount) AS total
  FROM sales
  GROUP BY region;

-- セキュアマテリアライズドビュー（定義を他ユーザーから隠す）
CREATE SECURE MATERIALIZED VIEW smv_cherry_creek_trail(
    POINT_ID, TRAIL_NAME, LNG, LAT, COORD_PAIR, DISTANCE_TO_MELANIES
) AS
SELECT
    value:sequence_1              AS point_id,
    value:trail_name::VARCHAR     AS trail_name,
    value:latitude::NUMBER(11, 8) AS lng,
    value:longitude::NUMBER(11, 8) AS lat,
    lng || ' ' || lat              AS coord_pair,
    distance_to_mc(ST_MAKEPOINT(lng, lat)) AS distance_to_melanies
FROM t_cherry_creek_trail;
```

### 2.2 最適なユースケース

- **集中的な計算**（距離計算、集計など）を**頻繁にクエリ**する
- 基礎データは**滅多に変更されない**

> 毎秒更新されるデータには不向き。自動リフレッシュが頻発しコストが膨大になります。

### 2.3 ステージに直接マテリアライズドビューを配置できない理由

マテリアライズドビューの自動リフレッシュは基礎テーブルの変更追跡に依存します。ステージ上のファイルはSnowflakeのテーブルではないため変更追跡が適用できません。

**回避策 — 外部テーブル → マテリアライズドビューの2層構造:**

```
ステージ（Parquetファイル）
    ↓ 参照
外部テーブル（External Table）    ← AUTO_REFRESHで変更を検知
    ↓ 基盤
セキュアマテリアライズドビュー（SMV） ← クエリ結果をキャッシュ
    ↓ クエリ
ユーザー（高速に応答）
```

### 2.4 SECUREキーワードの効果

`SECURE`を付けると、ビュー定義（SELECT文）が他ユーザーから見えなくなります。共有データに対するセキュリティを高めるために使用します。

## 3. 外部ボリューム（External Volume）

> 🔗 **検証SQL**: [`verification/09_advanced_objects/03_external_volume.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/09_advanced_objects/03_external_volume.sql)

### 3.1 外部ボリュームとは

Snowflakeの外部にあるクラウドストレージを**Icebergテーブルのデータ保存先**として登録するオブジェクトです。

```sql
CREATE OR REPLACE EXTERNAL VOLUME iceberg_external_volume
   STORAGE_LOCATIONS = (
      (
         NAME                   = 'iceberg-s3-us-west-2'
         STORAGE_PROVIDER       = 'S3'
         STORAGE_BASE_URL       = 's3://my-iceberg-bucket/'
         STORAGE_AWS_ROLE_ARN   = 'arn:aws:iam::123456789:role/iceberg_role'
         STORAGE_AWS_EXTERNAL_ID = 'iceberg_external_id'
      )
   );
```

### 3.2 外部ステージとの違い

| 項目 | 外部ステージ | 外部ボリューム |
|------|------------|--------------|
| 主な用途 | ファイルの**読み込み元** | Icebergテーブルのデータ**保存先** |
| 操作 | 読み取り中心 | 読み取り + 書き込み（IAMロールに書き込み権限が必要） |
| 使用場面 | COPY INTO、ビュー | CREATE ICEBERG TABLE |

## 4. Apache Icebergテーブル

> 🔗 **検証SQL**: [`verification/09_advanced_objects/04_iceberg_table.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/09_advanced_objects/04_iceberg_table.sql)

### 4.1 Icebergとは

**Apache Iceberg** はApache Software Foundationが管理するオープンソースのテーブルフォーマットです。SnowflakeのみならずSpark、Flink、Trinoなど業界全体で広く使われる標準規格です。

### 4.2 外部テーブルとの最大の違い — DML操作の可否

| 操作 | 外部テーブル | Icebergテーブル |
|------|------------|---------------|
| SELECT（読み取り） | ✓ できる | ✓ できる |
| INSERT（行追加） | ✗ できない | ✓ **できる** |
| UPDATE（行更新） | ✗ できない | ✓ **できる** |
| DELETE（行削除） | ✗ できない | ✓ **できる** |

> Icebergテーブルは外部ストレージ（S3等）上のParquetファイルに対して、通常のSnowflakeテーブルと同じようにDMLを実行できます。

### 4.3 作成方法

```sql
-- 前提: 外部ボリュームが作成済みであること

-- 1. Icebergテーブル用データベースを作成
CREATE DATABASE my_iceberg_db
  CATALOG        = 'SNOWFLAKE'             -- Snowflakeがメタデータを管理
  EXTERNAL_VOLUME = 'iceberg_external_volume';

-- 2. Icebergテーブルを作成（CTAS形式）
CREATE ICEBERG TABLE cct_trail_data (
    point_id              NUMBER(10, 0),
    trail_name            STRING,
    coord_pair            STRING,
    distance_to_melanies  DECIMAL(20, 10),
    user_name             STRING
)
BASE_LOCATION = 'cct_trail_data'
AS SELECT TOP 100 * FROM smv_cherry_creek_trail;

-- 3. 通常のテーブルと同様にDMLを使える
UPDATE cct_trail_data SET user_name = 'updated' WHERE point_id = 1;
```

> **`CATALOG = 'SNOWFLAKE'`:** IcebergテーブルのメタデータをSnowflakeが管理することを意味します。AWS Glueなど外部カタログも選択可能ですが、Snowflakeカタログが最もシンプルです。

### 4.4 データ管理アプローチの進化

| アプローチ | 時代 | 特徴 | 限界 |
|-----------|------|------|------|
| **データウェアハウス** | 1990年代〜 | 構造化データの高速分析 | 非構造化データ対応困難、ベンダーロックイン |
| **データレイク** | 2010年代〜 | あらゆる形式のデータを安価に保存 | クエリが遅い、データ品質管理困難 |
| **データレイクハウス** | 2020年代〜 | 両者の長所を組み合わせ | — |

> SnowflakeはIcebergテーブルをサポートすることで、データレイクハウスのすべての要素に対応できるようになりました。

## 5. 高度なオブジェクトの比較まとめ

| オブジェクト | データの場所 | 読み取り | 書き込み | 自動更新 | MV基盤 |
|------------|------------|:--------:|:--------:|:--------:|:------:|
| **ビュー** | ステージ / テーブル | ✓ | — | — | ✗ |
| **外部テーブル** | ステージ（外部） | ✓ | — | AUTO_REFRESH | ✓ |
| **マテリアライズドビュー** | キャッシュ（高速） | ✓ | — | 自動（基礎変更時） | — |
| **Icebergテーブル** | 外部ボリューム | ✓ | ✓ | — | — |

## 6. 理解度チェック

### Q1. 外部テーブルとビューの違い
マテリアライズドビューの基盤になれるのはビューと外部テーブルのどちらですか？その理由は？

<details>
<summary>解答</summary>

**外部テーブル**。マテリアライズドビューの自動リフレッシュには変更追跡が必要ですが、ビューには変更追跡の仕組みがありません。外部テーブルは `AUTO_REFRESH = TRUE` で変更を検知できるため、マテリアライズドビューの基盤になれます。
</details>

### Q2. Icebergテーブルの特徴
外部テーブルとIcebergテーブルの最も重要な違いは何ですか？

<details>
<summary>解答</summary>

**DML操作（INSERT/UPDATE/DELETE）の可否**。外部テーブルは読み取り専用ですが、Icebergテーブルは外部ストレージに対してもDMLを実行できます。
</details>

### Q3. マテリアライズドビューの適切なユースケース
「毎秒更新される株価データの最新値を表示するビュー」にマテリアライズドビューは適切ですか？

<details>
<summary>解答</summary>

**いいえ**。マテリアライズドビューは「基礎データが滅多に変更されない場合」に適します。毎秒更新されるデータでは自動リフレッシュが頻発し、コストが膨大になります。通常のビューの方が適切です。
</details>

### Q4. 外部ボリュームの用途
外部ステージと外部ボリュームの違いを説明してください。

<details>
<summary>解答</summary>

- **外部ステージ**: ファイルの**読み込み元**。COPY INTOやビューで参照する。
- **外部ボリューム**: Icebergテーブルのデータ**保存先**。読み書き両方の権限が必要。
</details>

## 7. 試験対策ポイント

- [ ] 外部テーブル（External Table）がビューと異なる点（AUTO_REFRESH、マテリアライズドビューの基盤になれる）を説明できる
- [ ] 外部テーブルの列定義構文がビューとは逆順（`列名 型 AS (定義)`）であることを知っている
- [ ] なぜデータをロードしない選択をする組織があるかを（ガバナンス・コスト・ロックイン回避等）説明できる
- [ ] マテリアライズドビューが通常のビューより高速な理由（クエリ結果をキャッシュ）を説明できる
- [ ] ステージに直接マテリアライズドビューを配置できない理由と回避策（外部テーブル2層構造）を説明できる
- [ ] SECUREキーワードがビュー定義を隠す効果を知っている
- [ ] 外部ボリュームがIcebergテーブルのデータ保存先として機能することを知っている
- [ ] IcebergテーブルがDML（INSERT/UPDATE/DELETE）をサポートすることを外部テーブルと比較して説明できる
- [ ] Apache IcebergがSnowflake固有ではなくオープンソース標準であることを知っている（Spark, Flink, Trinoでも使用）
- [ ] `CATALOG = 'SNOWFLAKE'` の意味（Snowflakeがメタデータを管理）を説明できる
- [ ] データレイク / データウェアハウス / データレイクハウスの違いを説明できる
