# アクセス制御とロール

> SOL-C01 対応 | 出題比率: 15%

---

## 1. アクセス制御の基本概念

### 1.1 アクセス制御とは

「誰が」「何に」「どのような操作を」できるかを管理する仕組みです。Snowflakeでは、**ロール（役割）** を通じてアクセス権限を管理します。

### 1.2 Identity（認証）vs Access（認可）

アクセス制御を理解する前に、2つの概念の違いを把握することが重要です。

| 概念 | 英語 | 意味 | 確認に使うもの | 通過後の呼称 |
|------|------|------|--------------|------------|
| **Identity（識別）** | Authentication（認証） | **WHO you are**（あなたは誰か） | ユーザー名・パスワード・MFA | 認証済み |
| **Access（アクセス）** | Authorization（認可） | **WHAT you can do**（何ができるか） | RBACのロール割り当て | 認可済み |

```
ログイン時の流れ:
  Step1: ユーザー名 + パスワードで本人確認 → 認証（Authentication）= Identity
         ↓ ログイン成功
  Step2: どのロールを持っているかを確認  → 認可（Authorization）= Access
         ↓ 権限確認完了
  オブジェクトへのアクセス開始
```

> **覚え方**: 「認証 = 本人証明」「認可 = 入場許可」。コンサート会場に例えると、認証はチケットの名前確認、認可はそのチケットで入れる席（VIP / 一般）の確認です。

### 1.3 Snowflakeの2つのアクセス制御モデル

| モデル | 説明 | 特徴 |
|--------|------|------|
| **RBAC（Role-Based Access Control）** | ロールを通じて権限を管理 | 集中管理しやすい |
| **DAC（Discretionary Access Control）** | オブジェクト所有者が権限を管理 | 「作成者 = 所有者」の原則 |

Snowflakeでは両方を組み合わせて使用します。基本的には**RBAC（ロールベース）** で権限を管理し、**DAC（裁量的）** でオブジェクトの所有権を定めます。

#### DAC の核心：「You create it, you own it」

DAC の本質は **「作成したロールがそのオブジェクトを所有する」** という原則です。

```
例: SYSADMIN ロールで作業中に CREATE DATABASE を実行
  → そのデータベースのオーナーは SYSADMIN になる
  → SYSADMIN はそのデータベースを削除・名前変更・移動できる

例: ACCOUNTADMIN ロールで作業中に CREATE DATABASE を実行
  → そのデータベースのオーナーは ACCOUNTADMIN になる
  → SYSADMIN に切り替えると、そのデータベースが見えなくなる場合がある！
```

> **実務への影響**: データベースやテーブルを作る前に、**必ずロールを確認する**こと。ロールが間違っていると、SYSADMINで見えないオブジェクトが生まれてしまいます。

### 1.4 アクセス制御の流れ（RBAC の動作イメージ）

```
┌────────────────────────────────────────────────────┐
│  ユーザー（John）                                    │
│      │                                             │
│      ▼                                             │
│  ロール（analyst_role）← 権限が付与されている        │
│      │                                             │
│      ▼                                             │
│  オブジェクト（テーブル、Warehouse等）               │
│  ・SELECT権限があれば → 読み取り可能                │
│  ・USAGE権限があれば → 使用可能                     │
└────────────────────────────────────────────────────┘
```

**ポイント**: ユーザーが直接オブジェクトにアクセスするのではなく、必ずロールを経由します。

## 2. システム定義ロール

### 2.1 なぜシステム定義ロールがあるのか

Snowflakeには最初から用意されたロールがあります。これにより、アカウント管理に必要な権限が整理されています。

### 2.2 ロール階層

```
ORGADMIN（組織管理者）← 複数アカウントを持つ組織向け
    │
ACCOUNTADMIN（アカウント管理者）← 最強の権限、慎重に使用
    ├── SYSADMIN（システム管理者）← オブジェクト作成を担当
    │       └── カスタムロール ← ここに自分で作ったロールを配置
    └── SECURITYADMIN（セキュリティ管理者）← 権限管理を担当
            └── USERADMIN（ユーザー管理者）← ユーザー/ロール作成
                    │
PUBLIC（パブリック）← 全ユーザーに自動付与
```

### 2.3 各ロールの役割と使い分け

| ロール | 主な役割 | 日常使用 | 代表的な権限 |
|--------|---------|:--------:|-------------|
| **ORGADMIN** | 組織全体の管理 | × | アカウント作成、組織設定 |
| **ACCOUNTADMIN** | アカウント最高権限 | × | すべての権限を継承 |
| **SECURITYADMIN** | セキュリティ管理 | △ | MANAGE GRANTS（全オブジェクトの権限管理） |
| **USERADMIN** | ユーザー管理 | △ | CREATE USER、CREATE ROLE |
| **SYSADMIN** | システム管理 | ○ | CREATE DATABASE、CREATE WAREHOUSE |
| **PUBLIC** | 基本権限 | ○ | 全ユーザーに付与される最小権限 |

### 2.4 よくある質問

**Q: ACCOUNTADMINを日常的に使ってはいけないのはなぜ？**

A: ACCOUNTADMINは最強の権限を持つため、誤操作のリスクが大きいです。日常業務には、必要最小限の権限を持つロールを使用すべきです。

**Q: カスタムロールはどこに配置すべき？**

A: **SYSADMIN**の下に配置することが推奨されます。これにより、SYSADMINがカスタムロールを管理でき、組織的な権限管理が維持されます。

```sql
-- カスタムロールをSYSADMINの下に配置
GRANT ROLE analyst_role TO ROLE SYSADMIN;
```

## 3. ロール管理

> 🔗 **検証SQL**: [`verification/05_access_control/01_role_management.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/05_access_control/01_role_management.sql)

### 3.1 ロールの作成

```sql
-- カスタムロールの作成
CREATE ROLE analyst_role;
CREATE ROLE developer_role;

-- コメント付きで作成（推奨）
CREATE ROLE data_engineer_role
  COMMENT = 'Data Engineering Team - ETL作業用';
```

### 3.2 ロールの付与

```sql
-- ユーザーにロールを付与（JohnがAnalyst_roleを使えるようになる）
GRANT ROLE analyst_role TO USER john;

-- ロールにロールを付与（階層を構築）
-- SYSADMINがanalyst_roleの権限も持つようになる
GRANT ROLE analyst_role TO ROLE SYSADMIN;
```

### 3.3 ロールの使用

```sql
-- 現在のロールを変更
USE ROLE analyst_role;

-- 現在のロールを確認
SELECT CURRENT_ROLE();  -- 結果: ANALYST_ROLE

-- セカンダリロールを有効化（複数ロールの権限を同時に使用）
USE SECONDARY ROLES ALL;
```

### 3.4 セカンダリロールとは

ユーザーは複数のロールを持つことができますが、通常は1つのロール（プライマリロール）だけがアクティブです。セカンダリロールを有効にすると、複数のロールの権限を同時に使用できます。

```sql
-- セカンダリロールを有効化
USE SECONDARY ROLES ALL;

-- これで、付与されているすべてのロールの権限を同時に利用可能
```

### 3.5 デフォルトロール

ログイン時に自動的に使用されるロールを設定できます。

```sql
-- ユーザーのデフォルトロールを設定
ALTER USER john SET DEFAULT_ROLE = analyst_role;
```

#### デフォルトロールの動作を理解する

デフォルトロールと「現在のロール」は別物です。混同しないようにしましょう。

| 項目 | 説明 |
|------|------|
| **デフォルトロール** | ログイン時に**自動的に設定される**ロール（USER設定） |
| **現在のロール** | 今この瞬間に**使用中のロール**（作業中に切り替え可能） |

```
動作の流れ:
  ① ログイン → デフォルトロール（例: ACCOUNTADMIN）で開始
  ② 作業中に USE ROLE SYSADMIN → 現在のロールが SYSADMIN に変わる
  ③ ログアウト → セッション終了
  ④ 再ログイン → デフォルトロール（ACCOUNTADMIN）に戻る ★
```

> **ポイント**: ロールを切り替えてログアウトしても、次のログイン時には必ずデフォルトロールに戻ります。ロール変更はそのセッション限りです。

#### "does not exist" エラーの第1チェック項目

Snowflakeでオブジェクトが見つからないエラーが出たら、まず確認するべきは**現在のロール**です。

```
エラー: SQL compilation error: Object 'TABLE_NAME' does not exist or not authorized.

チェックリスト:
1. ✅ 現在のロールを確認（画面右上 or SELECT CURRENT_ROLE()）
2. ✅ そのロールはオブジェクトにアクセスできるか？
3. ✅ ワークシートのコンテキスト（Database/Schema）は正しいか？
4. ✅ オブジェクト名のタイポはないか？
5. ✅ そもそもオブジェクトを作成したか？
```

## 4. 権限（Privileges）

> 🔗 **検証SQL**: [`verification/05_access_control/02_privileges.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/05_access_control/02_privileges.sql)

### 4.1 権限とは

オブジェクト（テーブル、Warehouse等）に対して「何ができるか」を定義するものです。

### 4.2 主な権限の種類

| カテゴリ | 権限 | 説明 | 対象オブジェクト |
|---------|------|------|-----------------|
| **グローバル** | CREATE DATABASE | データベース作成 | アカウント |
| | CREATE WAREHOUSE | Warehouse作成 | アカウント |
| | CREATE ROLE | ロール作成 | アカウント |
| | MANAGE GRANTS | 権限管理 | アカウント |
| **データベース** | USAGE | データベースの使用 | Database |
| **スキーマ** | USAGE | スキーマの使用 | Schema |
| | CREATE TABLE | テーブル作成 | Schema |
| **テーブル** | SELECT | 読み取り | Table |
| | INSERT | 挿入 | Table |
| | UPDATE | 更新 | Table |
| | DELETE | 削除 | Table |
| | TRUNCATE | 全行削除 | Table |
| **全般** | OWNERSHIP | 完全な制御（所有権） | 全オブジェクト |

### 4.3 USAGEとSELECTの違い

初学者がよく混乱するポイントです。

- **USAGE**: そのオブジェクト自体を「使う」権限（入り口に入れる）
- **SELECT**: テーブルのデータを「読む」権限（中のデータを見れる）

**例**: テーブルを読むには両方が必要
```sql
-- 1. データベースに入れる
GRANT USAGE ON DATABASE sales_db TO ROLE analyst_role;

-- 2. スキーマに入れる
GRANT USAGE ON SCHEMA sales_db.public TO ROLE analyst_role;

-- 3. テーブルのデータを読める
GRANT SELECT ON TABLE sales_db.public.customers TO ROLE analyst_role;
```

### 4.4 権限の付与

```sql
-- 単一テーブルへの権限
GRANT SELECT ON TABLE sales_db.public.customers TO ROLE analyst_role;

-- スキーマ内の全テーブルへの権限
GRANT SELECT ON ALL TABLES IN SCHEMA sales_db.public TO ROLE analyst_role;

-- 将来作成されるテーブルへの権限（重要！）
GRANT SELECT ON FUTURE TABLES IN SCHEMA sales_db.public TO ROLE analyst_role;
```

### 4.5 FUTURE GRANTSの重要性

`FUTURE GRANTS`を設定しないと、新しいテーブルが作成されるたびに権限を付与する必要があります。

```sql
-- FUTUREを使わない場合
-- テーブルAを作成 → 権限付与
-- テーブルBを作成 → 権限付与（忘れがち！）
-- テーブルCを作成 → 権限付与（また忘れる！）

-- FUTUREを使う場合
GRANT SELECT ON FUTURE TABLES IN SCHEMA sales_db.public TO ROLE analyst_role;
-- 以降、このスキーマに作成される全テーブルに自動でSELECT権限が付く
```

### 4.6 権限の取消

```sql
-- 権限の取消
REVOKE SELECT ON TABLE customers FROM ROLE analyst_role;

-- 将来のオブジェクトへの権限取消
REVOKE SELECT ON FUTURE TABLES IN SCHEMA sales_db.public FROM ROLE analyst_role;
```

### 4.7 Warehouseの権限

| 権限 | できること |
|------|-----------|
| **USAGE** | クエリ実行にWarehouseを使用 |
| **OPERATE** | 起動、停止、サスペンド、レジューム |
| **MODIFY** | サイズ変更、設定変更 |
| **MONITOR** | 使用状況の監視 |
| **OWNERSHIP** | 完全な制御（削除含む） |

```sql
-- クエリ実行のみ許可
GRANT USAGE ON WAREHOUSE compute_wh TO ROLE analyst_role;

-- Warehouseの操作（起動/停止）も許可
GRANT OPERATE ON WAREHOUSE compute_wh TO ROLE admin_role;
```

## 5. 所有権（Ownership）

> 🔗 **検証SQL**: [`verification/05_access_control/03_ownership.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/05_access_control/03_ownership.sql)

### 5.1 所有権とは

- すべてのオブジェクトには**1つの所有者ロール**がある
- 所有者は**完全な制御権**（削除を含む）を持つ
- オブジェクト作成時、**現在のロール**が所有者になる

### 5.2 所有権の移譲

オブジェクトの所有権をACCOUNTADMINからSYSADMINに渡す場面は頻出パターンです。

```sql
-- データベースの所有権をSYSADMINに移譲
GRANT OWNERSHIP ON DATABASE DEMO_DB TO ROLE SYSADMIN;

-- スキーマの所有権を移譲
GRANT OWNERSHIP ON SCHEMA GARDEN_PLANTS.PUBLIC TO ROLE SYSADMIN;

-- テーブルの所有権を移譲
GRANT OWNERSHIP ON TABLE customers TO ROLE data_owner_role;

-- Warehouseの所有権を移譲
GRANT OWNERSHIP ON WAREHOUSE COMPUTE_WH TO ROLE SYSADMIN;

-- 複数オブジェクトの所有権を一括移譲
GRANT OWNERSHIP ON ALL TABLES IN SCHEMA public TO ROLE data_owner_role;
```

> **ベストプラクティス**: オブジェクトを作成する前に `USE ROLE SYSADMIN;` でロールを設定しておくと、所有権移譲の手間が省けます。作成後に所有権を移譲するより、最初から正しいロールで作成する方が確実です。

### 5.3 ベストプラクティス

```
推奨: オブジェクト所有専用のロールを作成

data_owner_role ← テーブルを所有
    │
    ├── analyst_role ← SELECTのみ
    └── developer_role ← SELECT, INSERT, UPDATE
```

ACCOUNTADMINでオブジェクトを作成すると、ACCOUNTADMINが所有者になってしまいます。これは推奨されません。

## 6. MANAGED ACCESS スキーマ

> 🔗 **検証SQL**: [`verification/05_access_control/04_managed_access_schema.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/05_access_control/04_managed_access_schema.sql)

### 6.1 通常スキーマの問題点

通常のスキーマでは、オブジェクト所有者が自由に権限を付与できます。これはセキュリティの観点から問題になることがあります。

```
通常スキーマ:
- テーブル所有者Aが勝手にユーザーXに権限付与 → 管理者が把握できない
- テーブル所有者Bも独自に権限付与 → 権限が散らばる
```

### 6.2 MANAGED ACCESSスキーマ

スキーマ所有者（またはSECURITYADMIN）のみが権限を管理できるスキーマです。

```sql
-- Managed Accessスキーマの作成
CREATE SCHEMA managed_schema WITH MANAGED ACCESS;

-- 既存スキーマを変更
ALTER SCHEMA my_schema ENABLE MANAGED ACCESS;
```

### 6.3 比較

| 項目 | 通常スキーマ | MANAGED ACCESS |
|------|-------------|----------------|
| 権限を付与できる人 | スキーマ所有者 + 各オブジェクト所有者 | スキーマ所有者のみ |
| 管理の集中度 | 分散 | 集中 |
| セキュリティ | 柔軟だがリスクあり | 厳格で安全 |
| 推奨シーン | 小規模、柔軟性重視 | 本番環境、セキュリティ重視 |

## 7. データマスキング（Dynamic Data Masking）

> 🔗 **検証SQL**: [`verification/05_access_control/05_masking_policy.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/05_access_control/05_masking_policy.sql)

### 7.1 データマスキングとは

クエリ結果に対して、ロールに応じてデータを隠す（マスクする）機能です。

```
admin_role  → SELECT email → john@example.com（実データ）
analyst_role → SELECT email → ***@***.***（マスクされる）
```

### 7.2 マスキングポリシーの作成

```sql
-- メールアドレスをマスクするポリシー
CREATE MASKING POLICY email_mask AS (val STRING)
RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('ADMIN_ROLE') THEN val  -- 管理者は実データ
    ELSE '***@***.***'                              -- その他はマスク
  END;

-- テーブルのカラムに適用
ALTER TABLE customers MODIFY COLUMN email
  SET MASKING POLICY email_mask;
```

### 7.3 マスキングの例

```sql
-- クレジットカード番号のマスキング
CREATE MASKING POLICY cc_mask AS (val STRING)
RETURNS STRING ->
  CASE
    WHEN CURRENT_ROLE() IN ('FINANCE_ROLE') THEN val
    ELSE CONCAT('****-****-****-', RIGHT(val, 4))  -- 末尾4桁のみ表示
  END;
```

> **注意**: Dynamic Data Maskingは**Enterprise Edition以上**で利用可能です。

## 8. Row Access Policy（行レベルセキュリティ）

> 🔗 **検証SQL**: [`verification/05_access_control/06_row_access_policy.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/05_access_control/06_row_access_policy.sql)

### 8.1 Row Access Policyとは

テーブルの**行単位**でアクセスを制御する機能です。

```
sales テーブル:
| region | amount |
|--------|--------|
| APAC   | 1000   |  ← APAC_ROLEのみ見える
| APAC   | 2000   |  ← APAC_ROLEのみ見える
| EMEA   | 1500   |  ← EMEA_ROLEのみ見える
| NA     | 3000   |  ← NA_ROLEのみ見える
```

### 8.2 ポリシーの作成

```sql
-- リージョンに基づく行アクセスポリシー
CREATE ROW ACCESS POLICY region_policy AS (region STRING)
RETURNS BOOLEAN ->
  CASE
    WHEN CURRENT_ROLE() = 'ADMIN_ROLE' THEN TRUE  -- 管理者は全行
    WHEN CURRENT_ROLE() = 'APAC_ROLE' AND region = 'APAC' THEN TRUE
    WHEN CURRENT_ROLE() = 'EMEA_ROLE' AND region = 'EMEA' THEN TRUE
    ELSE FALSE
  END;

-- テーブルに適用
ALTER TABLE sales ADD ROW ACCESS POLICY region_policy ON (region);
```

### 8.3 Dynamic Data Masking vs Row Access Policy

| 項目 | Dynamic Data Masking | Row Access Policy |
|------|---------------------|-------------------|
| 制御対象 | カラムの値 | 行全体 |
| 結果 | データがマスクされる | 行が見えなくなる |
| ユースケース | 機密情報の部分的な隠蔽 | 部門別データの分離 |

## 9. オブジェクトタギング

> 🔗 **検証SQL**: [`verification/05_access_control/07_tagging.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/05_access_control/07_tagging.sql)

### 9.1 タグとは

オブジェクトにメタデータ（ラベル）を付けて管理・分類する機能です。

### 9.2 タグの作成と使用

```sql
-- タグの作成（許可される値を制限可能）
CREATE TAG cost_center ALLOWED_VALUES 'Sales', 'Marketing', 'Engineering';

-- タグの割り当て
ALTER TABLE customers SET TAG cost_center = 'Sales';

-- タグの確認
SELECT SYSTEM$GET_TAG('cost_center', 'customers', 'TABLE');
```

### 9.3 タグの活用例

- **コスト配分**: 部門ごとのコストを計算
- **データ分類**: 機密度レベル（High/Medium/Low）
- **ガバナンス**: PII（個人情報）を含むテーブルの識別

## 10. 権限の確認

> 🔗 **検証SQL**: [`verification/05_access_control/02_privileges.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/05_access_control/02_privileges.sql)

### 10.1 SHOW GRANTS

```sql
-- ロールに付与されている権限を確認
SHOW GRANTS TO ROLE analyst_role;

-- オブジェクトに付与されている権限を確認
SHOW GRANTS ON TABLE customers;

-- ユーザーに付与されているロールを確認
SHOW GRANTS TO USER john;

-- ロールがどのロールに付与されているか確認
SHOW GRANTS OF ROLE analyst_role;
```

### 10.2 結果の読み方

```sql
SHOW GRANTS TO ROLE analyst_role;
```

| privilege | granted_on | name | granted_to | grantee_name |
|-----------|------------|------|------------|--------------|
| SELECT | TABLE | CUSTOMERS | ROLE | ANALYST_ROLE |
| USAGE | DATABASE | SALES_DB | ROLE | ANALYST_ROLE |

→ analyst_roleは、SALES_DBの使用権限とCUSTOMERSテーブルのSELECT権限を持っている

## 11. よくある間違いと注意点

### 11.1 ロール関連

| 間違い | 正しい理解 |
|--------|-----------|
| ACCOUNTADMINでオブジェクトを作成する | 専用のロールで作成し、適切な所有者を設定すべき |
| カスタムロールをPUBLICの下に配置 | SYSADMINの下に配置することを推奨 |
| ロールの階層を無視して権限を付与 | 階層を意識して、上位ロールから継承させる |

### 11.2 権限関連

| 間違い | 正しい理解 |
|--------|-----------|
| USAGEだけでテーブルを読める | USAGE（入り口）+ SELECT（読み取り）の両方が必要 |
| ALL TABLESで今後のテーブルも含まれる | 今後のテーブルにはFUTURE TABLESが必要 |
| REVOKEで所有権を剥奪できる | 所有権の移譲はGRANT OWNERSHIPで行う |

## 12. 理解度チェック

### Q1. ロールの階層
カスタムロールを作成した場合、どのロールの下に配置することが推奨されますか？

<details>
<summary>解答</summary>

**SYSADMIN**の下に配置することが推奨されます。
```sql
GRANT ROLE custom_role TO ROLE SYSADMIN;
```
</details>

### Q2. 権限の組み合わせ
テーブルからデータを読み取るために必要な権限をすべて答えてください。

<details>
<summary>解答</summary>

1. **USAGE ON DATABASE** - データベースに入る
2. **USAGE ON SCHEMA** - スキーマに入る
3. **SELECT ON TABLE** - テーブルを読む

この3つすべてが必要です。
</details>

### Q3. FUTURE GRANTS
新しく作成されるテーブルにも自動で権限を付与したい場合、どのような構文を使いますか？

<details>
<summary>解答</summary>

```sql
GRANT SELECT ON FUTURE TABLES IN SCHEMA スキーマ名 TO ROLE ロール名;
```
`FUTURE`キーワードを使用します。
</details>

### Q4. MANAGED ACCESS
通常のスキーマとMANAGED ACCESSスキーマの違いを説明してください。

<details>
<summary>解答</summary>

- **通常スキーマ**: スキーマ所有者とオブジェクト所有者の両方が権限を付与できる
- **MANAGED ACCESS**: スキーマ所有者（またはSECURITYADMIN）のみが権限を付与できる

MANAGED ACCESSは権限管理を集中化し、セキュリティを強化します。
</details>

### Q5. マスキングと行アクセス
Dynamic Data MaskingとRow Access Policyの違いを説明してください。

<details>
<summary>解答</summary>

- **Dynamic Data Masking**: カラムの値をマスクする（例: メールを***@***に変換）
- **Row Access Policy**: 行全体を見えなくする（例: 他部門のデータを非表示）
</details>

## 13. 試験対策ポイント

- [ ] Authentication（認証）= Identity（誰か）、Authorization（認可）= Access（何ができるか）の違いを説明できる
- [ ] システム定義ロール（ACCOUNTADMIN、SYSADMIN等）の役割と階層を説明できる
- [ ] ACCOUNTADMINを日常使用しない理由を理解している
- [ ] DAC「You create it, you own it」＝作成ロールが所有者になる原則を理解している
- [ ] デフォルトロールはログイン時に自動設定され、ログアウト後リセットされることを理解している
- [ ] "does not exist" エラー時、最初にチェックすべきはロール
- [ ] ロールの付与構文（GRANT ROLE TO USER/ROLE）を知っている
- [ ] 権限の種類（USAGE、SELECT、OWNERSHIP等）の違いを把握している
- [ ] テーブルを読むにはUSAGE + SELECTが必要なことを理解している
- [ ] FUTURE GRANTSの使い方と重要性を理解している
- [ ] Managed Accessスキーマの特徴を説明できる
- [ ] Dynamic Data MaskingとRow Access Policyの違いを理解している
- [ ] SHOW GRANTSで権限を確認できる
