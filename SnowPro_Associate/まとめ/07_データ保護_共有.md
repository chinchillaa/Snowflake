## 1. Time Travel

> 🔗 **検証SQL**: [`verification/07_data_protection/01_time_travel.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/07_data_protection/01_time_travel.sql)

### 1.1 Time Travelとは

**Time Travel**は、過去のある時点のデータ状態にアクセスできる機能です。誤ってデータを削除・変更してしまった場合でも、過去の状態に戻せます。

```
現在時刻: 15:00
         │
         ▼
┌────────────────────────────────────────────┐
│ Time Travel範囲（過去0〜90日）              │
│                                            │
│  14:00   13:00   12:00   11:00   ...       │
│    ↓      ↓       ↓      ↓                │
│  [データ][データ][データ][データ]           │
│                                            │
│  ★ どの時点のデータも参照可能              │
└────────────────────────────────────────────┘
```

### 1.2 何ができるか

- **誤操作の復元**: DELETEやUPDATEを間違えても元に戻せる
- **過去データの参照**: 昨日の時点でのデータを確認
- **過去時点のクローン**: 特定時点のテーブルをコピー

### 1.3 保持期間

| エディション | Permanent | Transient | Temporary |
|-------------|:---------:|:---------:|:---------:|
| **Standard** | 0-1日 | 0-1日 | 0-1日 |
| **Enterprise+** | **0-90日** | 0-1日 | 0-1日 |

> **重要**: 長い保持期間を設定できるのは**Permanentテーブル**のみ。Transient/Temporaryは最大1日。

```sql
-- テーブルの保持期間を設定
ALTER TABLE customers SET DATA_RETENTION_TIME_IN_DAYS = 30;

-- データベースレベルで設定（配下のテーブルに継承）
ALTER DATABASE sales_db SET DATA_RETENTION_TIME_IN_DAYS = 14;
```

### 1.4 Time Travelクエリの書き方

過去のデータを参照する方法は3つあります。

```sql
-- 方法1: タイムスタンプ指定
SELECT * FROM customers
  AT(TIMESTAMP => '2026-03-01 10:00:00'::TIMESTAMP);

-- 方法2: オフセット指定（秒単位、現在からの経過秒数）
SELECT * FROM customers
  AT(OFFSET => -3600);  -- 1時間前（3600秒前）

-- 方法3: クエリID指定（特定クエリ実行前の状態）
SELECT * FROM customers
  BEFORE(STATEMENT => '01a2b3c4-5d6e-7f8g-9h0i-jklmnopqrst');
```

**ATとBEFOREの違い**:
- `AT`: その時点**の**データ
- `BEFORE`: その時点（またはクエリ）の**直前**のデータ

### 1.5 UNDROPによる復元

削除したオブジェクトをTime Travel期間内なら復元できます。

```sql
-- 削除したテーブルの復元
DROP TABLE customers;       -- 誤って削除
UNDROP TABLE customers;     -- 復元！

-- 削除したスキーマの復元
DROP SCHEMA my_schema;
UNDROP SCHEMA my_schema;

-- 削除したデータベースの復元
DROP DATABASE my_database;
UNDROP DATABASE my_database;
```

### 1.6 過去データからの復元

誤ったUPDATEやDELETEを取り消す方法です。

```sql
-- 過去のデータで現在のテーブルを上書き
CREATE OR REPLACE TABLE customers AS
SELECT * FROM customers AT(TIMESTAMP => '2026-03-01 10:00:00'::TIMESTAMP);

-- 別テーブルとして復元（安全）
CREATE TABLE customers_backup AS
SELECT * FROM customers AT(OFFSET => -86400);  -- 1日前
```

## 2. Fail-safe

### 2.1 Fail-safeとは

**Fail-safe**は、Time Travel期間終了後の**追加保護期間**です。災害復旧などの緊急時にSnowflakeサポートを通じてデータを復元できます。

```
データ更新
    │
    ▼
┌───────────────────────┬─────────────────┐
│    Time Travel期間     │   Fail-safe     │
│  （0-90日、設定可能）   │  （7日間固定）   │
│                       │                 │
│  ユーザーが自分で復元可能│ Snowflakeのみ復元可能│
└───────────────────────┴─────────────────┘
    │                            │
    ▼                            ▼
  データ編集可能              完全に削除
```

### 2.2 Time Travel vs Fail-safe

| 項目 | Time Travel | Fail-safe |
|------|-------------|-----------|
| **期間** | 0-90日（設定可能） | 7日間（固定） |
| **復元方法** | ユーザーが自分で | Snowflakeサポート経由 |
| **用途** | 日常的な復元 | 災害復旧 |
| **コスト** | ストレージ課金 | ストレージ課金 |

### 2.3 対象オブジェクト

| オブジェクト | Time Travel | Fail-safe |
|-------------|:-----------:|:---------:|
| **Permanent Table** | ○ | ○ |
| **Transient Table** | ○ | **×** |
| **Temporary Table** | ○ | **×** |

> **重要**: Transient/TemporaryテーブルにはFail-safeがありません。コスト削減には有効ですが、災害時の保護がないことを理解してください。

## 3. クローン（Zero-Copy Clone）

> 🔗 **検証SQL**: [`verification/07_data_protection/02_zero_copy_clone.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/07_data_protection/02_zero_copy_clone.sql)

### 3.1 クローンとは

オブジェクトの**即時コピー**を作成する機能です。実際のデータはコピーせず、メタデータのみをコピーするため、追加ストレージは最初は不要です。

### 3.2 なぜ「Zero-Copy」なのか

```
元テーブル: [A][B][C][D]
              │  │  │  │
クローン:    [A][B][C][D]  ← 同じマイクロパーティションを参照
              ↑
             コピーではなく参照

クローンで変更後:
元テーブル: [A][B][C][D]
クローン:   [A][B'][C][D]  ← 変更された[B']のみ新規ストレージ消費
              ↑
            元のデータはそのまま
```

**メリット**:
- **高速**: 数秒で作成完了（データサイズに関係なく）
- **コスト効率**: 変更した部分のみストレージ消費
- **独立性**: クローン後は完全に独立したオブジェクト

### 3.3 対象オブジェクト

- Database
- Schema
- Table
- Stage（内部ステージのみ）
- Stream

### 3.4 作成方法

```sql
-- テーブルのクローン
CREATE TABLE customers_clone CLONE customers;

-- スキーマのクローン（配下のテーブルも全てクローン）
CREATE SCHEMA schema_clone CLONE production;

-- データベースのクローン（配下のスキーマ・テーブル全てクローン）
CREATE DATABASE db_clone CLONE production_db;

-- Time Travelと組み合わせ（過去時点のクローン）
CREATE TABLE customers_backup CLONE customers
  AT(TIMESTAMP => '2026-03-01 10:00:00'::TIMESTAMP);
```

### 3.5 よくある使用シーン

1. **テスト環境の構築**: 本番データのコピーを数秒で作成
2. **バックアップ**: 定期的なスナップショット
3. **開発環境**: 最新の本番データで開発

### 3.6 権限について

> **重要**: クローン元の**権限は継承されません**。クローン作成者のロールが所有者になります。

## 4. Secure Data Sharing

> 🔗 **検証SQL**: [`verification/07_data_protection/03_secure_data_sharing.sql`](https://github.com/chinchillaa/snowflake_lessons/blob/main/verification/07_data_protection/03_secure_data_sharing.sql)

### 4.1 Data Sharingとは

Snowflakeアカウント間で**データをリアルタイム共有**する機能です。データをコピーせずに、他のアカウントにデータを提供できます。

```
┌─────────────────────┐     ┌─────────────────────┐
│   Provider Account   │     │  Consumer Account   │
│   （データ提供者）    │     │  （データ利用者）    │
│                     │     │                     │
│  ┌───────────────┐  │     │  ┌───────────────┐  │
│  │   Database    │  │────▶│  │   Database    │  │
│  │   (原本)      │  │ 共有 │  │  (読取専用)   │  │
│  └───────────────┘  │     │  └───────────────┘  │
│                     │     │                     │
│   自分のWarehouse   │     │   自分のWarehouse   │
└─────────────────────┘     └─────────────────────┘
                               ↑
                          クエリコストは
                          コンシューマー負担
```

### 4.2 主な特徴

| 特徴 | 説明 |
|------|------|
| **コピー不要** | データは1か所のみに存在 |
| **リアルタイム** | プロバイダーの更新が即座に反映 |
| **読み取り専用** | コンシューマーは変更不可 |
| **コスト分離** | クエリコストはコンシューマー負担 |
| **セキュア** | 既存のアクセス制御を適用可能 |

### 4.3 共有の作成（プロバイダー側）

```sql
-- 1. 共有オブジェクトの作成
CREATE SHARE my_share;

-- 2. データベースへのアクセスを許可
GRANT USAGE ON DATABASE sales_db TO SHARE my_share;
GRANT USAGE ON SCHEMA sales_db.public TO SHARE my_share;
GRANT SELECT ON TABLE sales_db.public.customers TO SHARE my_share;

-- 3. コンシューマーアカウントに共有を付与
ALTER SHARE my_share ADD ACCOUNTS = consumer_account;
```

### 4.4 共有の使用（コンシューマー側）

```sql
-- 共有からデータベースを作成
CREATE DATABASE shared_data FROM SHARE provider_account.my_share;

-- データにアクセス（通常のSELECTで参照可能）
SELECT * FROM shared_data.public.customers;
```

### 4.5 Secure View（セキュアビュー）

共有時にデータをフィルタリングしたい場合は、Secure Viewを使用します。

```sql
-- セキュアビューの作成
CREATE SECURE VIEW customer_view AS
SELECT customer_id, name, region
FROM customers
WHERE region = CURRENT_ACCOUNT();  -- アカウントに応じたフィルタ
```

**SECUREキーワードの効果**:
- ビュー定義（SQL文）が隠蔽される
- コンシューマーはビューの中身を見れない

## 5. Listing & Marketplace

### 5.1 Snowflake Marketplace

**データ製品**を発見・取得できるプラットフォームです。サードパーティが提供する有料/無料のデータセットを利用できます。

### 5.2 Listingの種類

| 種類 | 説明 | アクセス |
|------|------|---------|
| **Free Listing** | 無料で利用可能 | 誰でも |
| **Paid Listing** | 有料データ製品 | 購入者のみ |
| **Private Listing** | 特定アカウントのみ | 招待制 |

### 5.3 実際の使用例

- **気象データ**: 天気予報データを分析に活用

## 6. レプリケーション

### 6.1 Database Replication

異なるリージョンやクラウドにデータベースを複製する機能です。

**用途**:
- **災害復旧（DR）**: 別リージョンにバックアップ
- **低遅延アクセス**: ユーザーに近いリージョンにレプリカを配置

```sql
-- プライマリデータベースの設定（レプリケーション元）
ALTER DATABASE my_db ENABLE REPLICATION TO ACCOUNTS account2.region2;

-- セカンダリでレプリカを作成（レプリケーション先）
CREATE DATABASE my_db_replica AS REPLICA OF account1.region1.my_db;

-- レプリカの更新（定期的に実行）
ALTER DATABASE my_db_replica REFRESH;
```

### 6.2 Replication vs Data Sharing

| 項目 | Replication | Data Sharing |
|------|-------------|--------------|
| **データの場所** | 複製（複数箇所） | 1箇所のみ |
| **更新** | 手動/定期リフレッシュ | リアルタイム |
| **ストレージコスト** | 複製分かかる | 追加なし |
| **用途** | 災害復旧、遅延軽減 | データ配布 |

### 6.3 Account Replication

ユーザー、ロールなどのアカウントレベルオブジェクトを複製する機能です。

> **注意**: Account ReplicationはBusiness Critical以上のエディションで利用可能。

## 7. データ保持に関するコスト

### 7.1 ストレージコスト

| 期間 | 課金 | 説明 |
|------|------|------|
| 現在のデータ | ストレージ料金 | 通常のデータ |
| Time Travel期間 | ストレージ料金 | 過去バージョンのデータ |
| Fail-safe期間 | ストレージ料金 | 追加保護データ |

> Time Travel + Fail-safeで、実データの数倍のストレージを消費する可能性があります。

### 7.2 コスト削減策

1. **短いTime Travel期間**を設定
2. **Transient/Temporary**テーブルを活用（Fail-safeなし）
3. 不要な**クローンを削除**
4. 大規模テーブルの保持期間を**最適化**

```sql
-- Transientテーブルでコスト削減（ステージングデータなど）
CREATE TRANSIENT TABLE staging_data (...);

-- 保持期間を短く設定（頻繁に更新されるログなど）
ALTER TABLE logs SET DATA_RETENTION_TIME_IN_DAYS = 1;
```

## 8. ストレージ使用量の確認

### 8.1 テーブルレベル

```sql
-- テーブルのストレージ情報
SELECT
  table_name,
  active_bytes / (1024*1024*1024) AS active_gb,
  time_travel_bytes / (1024*1024*1024) AS time_travel_gb,
  failsafe_bytes / (1024*1024*1024) AS failsafe_gb
FROM SNOWFLAKE.ACCOUNT_USAGE.TABLE_STORAGE_METRICS
WHERE table_name = 'CUSTOMERS';
```

### 8.2 アカウントレベル

```sql
-- アカウント全体のストレージ使用量
SELECT *
FROM SNOWFLAKE.ACCOUNT_USAGE.STORAGE_USAGE
ORDER BY usage_date DESC
LIMIT 30;
```

## 9. よくある間違いと注意点

### 9.1 Time Travel関連

| 間違い | 正しい理解 |
|--------|-----------|
| 全テーブルで90日間保持できる | Permanentのみ、かつEnterprise Edition以上 |
| Time Travelは無料 | ストレージコストがかかる |
| 保持期間を過ぎたら自動復元される | Fail-safe期間ならSnowflakeサポートに依頼が必要 |

### 9.2 クローン関連

| 間違い | 正しい理解 |
|--------|-----------|
| クローンは即座にストレージを消費 | 最初は0、変更分のみ消費 |
| クローン元の権限が引き継がれる | 権限は継承されない |
| 外部ステージもクローンできる | 内部ステージのみ可能 |

### 9.3 Data Sharing関連

| 間違い | 正しい理解 |
|--------|-----------|
| コンシューマーはデータを編集できる | 読み取り専用 |
| データはコピーされる | コピーなし、参照のみ |
| クエリコストはプロバイダー負担 | コンシューマー負担 |

## 10. 理解度チェック

### Q1. Time Travel保持期間
Enterprise EditionでPermanentテーブルの最大Time Travel保持期間は何日ですか？

<details>
<summary>解答</summary>

**90日**です。Standard Editionでは最大1日、Transient/Temporaryテーブルはエディションに関わらず最大1日です。
</details>

### Q2. Fail-safe
Fail-safeの特徴を2つ挙げてください。

<details>
<summary>解答</summary>

1. **7日間固定**（変更不可）
2. **Snowflakeサポートを通じてのみ**復元可能（ユーザーが直接アクセスできない）
</details>

### Q3. Zero-Copy Clone
クローンが「Zero-Copy」と呼ばれる理由を説明してください。

<details>
<summary>解答</summary>

クローン作成時に実際のデータをコピーせず、**メタデータ（参照情報）のみ**をコピーするためです。元データとクローンは同じマイクロパーティションを参照し、変更があった部分のみ新しいストレージを消費します。
</details>

### Q4. Data Sharing
Data Sharingでクエリを実行した場合、コストはどちらが負担しますか？

<details>
<summary>解答</summary>

**コンシューマー（データ利用者）** が負担します。プロバイダーはストレージコストのみを負担し、クエリのコンピュートコストはコンシューマーの Warehouse で処理されます。
</details>

### Q5. テーブルの種類とFail-safe
Fail-safeが適用されないテーブルの種類を2つ答えてください。

<details>
<summary>解答</summary>

1. **Transient Table**
2. **Temporary Table**

Permanentテーブルのみ Fail-safe が適用されます。
</details>

## 11. 試験対策ポイント

- [ ] Time Travelの保持期間（Standard: 最大1日、Enterprise: 最大90日）を覚えている
- [ ] Time Travelクエリの構文（AT/BEFORE、TIMESTAMP/OFFSET）を知っている
- [ ] UNDROPコマンドでテーブル/スキーマ/データベースを復元できることを理解している
- [ ] Fail-safeの特徴（7日間固定、Snowflakeのみアクセス可能）を把握している
- [ ] Fail-safeがPermanentテーブルのみに適用されることを知っている
- [ ] Zero-Copy Cloneの仕組み（メタデータコピー、変更分のみストレージ消費）を説明できる
- [ ] クローン元の権限が継承されないことを理解している
- [ ] Secure Data Sharingの特徴（コピー不要、リアルタイム、読み取り専用）を理解している
- [ ] Data Sharingでクエリコストがコンシューマー負担であることを知っている
- [ ] 共有の作成と使用のコマンド（CREATE SHARE、GRANT、FROM SHARE）を知っている
