## 問題1: Snowflakeアーキテクチャ

Snowflakeのアーキテクチャは3つの独立した層で構成されています。以下のうち、正しい組み合わせはどれですか？

- A) Presentation Layer、Application Layer、Data Layer
- B) Database Storage、Query Processing (Compute)、Cloud Services
- C) Client Layer、Server Layer、Storage Layer
- D) Frontend、Backend、Database

### あなたの回答

```
選択: [B]

選んだ理由・メモ:データ層・コンピュート層・アプリケーション層でそれぞれが独立して拡張ができる


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: B) Database Storage、Query Processing (Compute)、Cloud Services**

Snowflakeは「データの保存」「計算処理」「管理サービス」の3つを完全に独立させたアーキテクチャを持っています。これが従来のデータベースとの最大の違いです。

| 層 | 役割 | わかりやすく言うと |
|---|------|-----------------|
| Database Storage | データを圧縮された列指向（カラムナ）形式で保存 | データの「倉庫」 |
| Query Processing (Compute) | Virtual Warehouseによるクエリ実行・計算処理 | データを「加工する工場」 |
| Cloud Services | 認証・メタデータ管理・クエリ最適化を担当 | 全体を管理する「管理棟」 |

**なぜ3つに分けることが重要なのか？**
「工場（計算）」の規模を増やしたいときに「倉庫（ストレージ）」のコストは増えません。逆に大量データを蓄積してもコンピュートコストは変わりません。これにより**コスト効率よく柔軟なスケーリング**が実現できます。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション1（アーキテクチャ概要）

</details>

## 問題2: Virtual Warehouse

Virtual Warehouseに関する説明として、**誤っているもの**はどれですか？

- A) Virtual Warehouseは独立したコンピュートクラスタである
- B) 複数のVirtual Warehouseが同じデータに同時にアクセスできる
- C) あるVirtual Warehouseの負荷が他のVirtual Warehouseのパフォーマンスに影響する
- D) Virtual WarehouseはSQL実行やデータロードに必要なCPU、メモリ、一時ストレージを提供する

### あなたの回答

```
選択: [C]

選んだ理由・メモ:Virtual Warehouseはそれぞれ独立しているため、他のVirtual Warehouseの動作に影響は与えない


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: C) あるVirtual Warehouseの負荷が他のVirtual Warehouseのパフォーマンスに影響する（これが誤り）**

Virtual Warehouseは「完全に独立したコンピュートクラスタ（計算処理の集まり）」です。それぞれが自分専用のCPU・メモリ・SSDを持っているため、1つのWarehouseが重い処理をしていても、別のWarehouseのスピードには全く影響しません。

| 特徴 | 説明 |
|------|------|
| 完全独立 | 各Warehouseは自分専用のCPU・メモリ・SSDを持つ |
| 同時アクセス可能 | ストレージとコンピュートが分離されているため、複数のWarehouseが同じデータに同時アクセスできる |
| ワークロード分離 | 開発チームと本番チームで別々のWarehouseを持てば互いに干渉しない |

**身近な例え：** 同じ図書館（ストレージ）から複数の人（Virtual Warehouse）が同時に本を借りて読めますが、1人が大量の本を読んでいても他の人のスピードは落ちません。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション3（Virtual Warehouse）

</details>

### 実際に確認してみよう

**Snowsight（UI）での手順:**
1. 左メニューの **Admin > Warehouses** を開く
2. **+ Warehouse** ボタンから新規作成（名前・サイズ・自動停止時間を設定）
3. 既存のWarehouseの行をクリック → **Suspend**（一時停止）/ **Resume**（再開）ボタンを確認
4. 複数のWarehouseを作成し、それぞれ独立して稼働できることを確認

**SQLでの確認:**
```sql
-- Warehouseの作成
CREATE WAREHOUSE my_wh
  WAREHOUSE_SIZE = 'X-SMALL'
  AUTO_SUSPEND = 60
  AUTO_RESUME = TRUE;

-- Warehouseの一覧表示（状態の確認）
SHOW WAREHOUSES;

-- 別のWarehouseに切り替え（独立動作の確認）
USE WAREHOUSE my_wh;

-- Warehouseのサイズ変更
ALTER WAREHOUSE my_wh SET WAREHOUSE_SIZE = 'SMALL';

-- Warehouseの停止・再開
ALTER WAREHOUSE my_wh SUSPEND;
ALTER WAREHOUSE my_wh RESUME;
```

## 問題3: マイクロパーティション

Snowflakeのマイクロパーティションについて、正しい説明はどれですか？

- A) ユーザーが手動で作成・管理する必要がある
- B) 各パーティションのサイズは約50〜500MBで、圧縮された列指向フォーマットで保存される
- C) マイクロパーティションは変更可能（mutable）である
- D) パーティションにはメタデータは含まれない

### あなたの回答

```
選択: [B]

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: B) 各パーティションのサイズは約50〜500MBで、圧縮された列指向フォーマットで保存される**

マイクロパーティションとは、Snowflakeがデータを自動的に分割して管理する「小さなデータのまとまり」のことです。ユーザーが意識することなく、Snowflakeが最適な形で自動管理します。

| 特徴 | 説明 |
|------|------|
| 自動管理 | ユーザーの操作は不要。データをINSERTするだけで自動作成される |
| サイズ | 圧縮前で約50〜500MB。自動的に圧縮されてさらに小さくなる |
| 不変性（immutable） | 一度作成されると中身を変えられない。変更が必要なときは古いものを削除して新しいものを作る仕組み |
| メタデータ | 各列の最小値・最大値・行数などの統計情報を保持する |

**クエリプルーニング（不要なデータを読み飛ばす仕組み）について：**
各マイクロパーティションが「このパーティションには2020〜2021年のデータがある」というメタデータを持っているため、例えば「2023年のデータを検索」するクエリでは、2020〜2021年のパーティションを最初から読み飛ばせます。これにより大量データでも高速に検索できます。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション5（マイクロパーティション）

</details>

## 問題4: キャッシング

Snowflakeには3種類のキャッシュがあります。以下のうち、**Cloud Services層**で管理されるキャッシュはどれですか？

- A) Result Cache（結果キャッシュ）
- B) Local Disk Cache（ローカルディスクキャッシュ）
- C) Remote Disk Cache（リモートディスクキャッシュ）
- D) Query Compilation Cache（クエリコンパイルキャッシュ）

### あなたの回答

```
選択: [    ]

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: A) Result Cache（結果キャッシュ）**

キャッシュとは「一度計算した結果を保存しておき、同じ処理の繰り返しを省く仕組み」です。Snowflakeには役割の異なる3種類のキャッシュがあります。

| キャッシュ | どの層が管理するか | 何を保存するか | 特徴 |
|-----------|----------------|--------------|------|
| **Result Cache** | **Cloud Services層** | クエリの実行結果そのもの | 過去24時間以内にまったく同じクエリを実行した場合、コンピュートを使わず即座に返答。Virtual Warehouseが停止していても使える |
| Local Disk Cache | Compute層（Warehouse内） | S3等から取得したデータファイル | WarehouseのSSDに保存。Warehouseを停止するとクリアされる |
| Metadata Cache | Cloud Services層 | テーブルの統計情報・スキーマ情報 | クエリ最適化に使用される |

**なぜResult CacheがCloud Services層なのか？**
Result Cacheはコンピュート（Warehouse）に依存しない「システム全体の共有キャッシュ」だからです。どのWarehouseから実行した場合でも共有されます。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション4（キャッシング）

</details>

## 問題5: COPY INTOコマンド

COPY INTOコマンドを使用してデータをロードする際、**正しい説明**はどれですか？

- A) COPY INTOはリアルタイムストリーミング専用のコマンドである
- B) 外部ステージまたは内部ステージからテーブルにデータをロードできる
- C) COPY INTOはJSON形式のファイルのみをサポートする
- D) COPY INTOの実行にはVirtual Warehouseは不要である

### あなたの回答

```
選択: [    ]

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: B) 外部ステージまたは内部ステージからテーブルにデータをロードできる**

COPY INTOとは「ステージ（ファイルの一時置き場）からテーブルにデータをまとめて取り込む命令」です。

| 項目 | 説明 |
|------|------|
| 用途 | バッチロード（一括処理）用。リアルタイムではない |
| ステージの種類 | **内部ステージ**（Snowflake内部）と**外部ステージ**（S3・Azure Blob・GCSなどのクラウドストレージ）の両方に対応 |
| 対応ファイル形式 | CSV・JSON・Parquet・Avro・ORC・XMLなど多数。JSONのみではない |
| Virtual Warehouse | 実行には**Virtual Warehouseが必要**（コンピュートを使う処理のため） |

**ステージ（Stage）とは：** Snowflakeにデータをロードする前の「一時的な置き場所」のことです。ローカルPCのファイルもまずステージに置いてからロードします。

📖 **参照先:** まとめ/03_データ管理_ロード.md > セクション3（COPYコマンド）

</details>

### 実際に確認してみよう

**Snowsight（UI）での手順:**
1. 左メニューの **Data > Databases** から対象テーブルを選択
2. テーブル詳細画面の **Load Data** ボタン → ウィザードに従ってファイルをアップロード（内部ステージ経由でCOPY INTOが実行される）
3. ロード結果は **Activity > Copy History** で確認できる

**SQLでの確認:**
```sql

  FROM @~/staged/sample.csv
  FILE_FORMAT = (TYPE = 'CSV' FIELD_OPTIONALLY_ENCLOSED_BY = '"' SKIP_HEADER = 1);

-- ロード履歴の確認
SELECT * FROM TABLE(INFORMATION_SCHEMA.COPY_HISTORY(
  TABLE_NAME => 'MY_TABLE',
  START_TIME => DATEADD(HOUR, -1, CURRENT_TIMESTAMP())
));
```

## 問題6: Snowpipe

Snowpipeに関する説明として、**正しいもの**はどれですか？

- A) Snowpipeはバッチ処理専用であり、リアルタイムロードには対応していない
- B) Snowpipeは専用のVirtual Warehouseを必要とする
- C) Snowpipeはサーバーレスで動作し、ファイルが到着すると自動的にデータをロードする
- D) Snowpipeで使用できるファイル形式はCSVのみである

### あなたの回答

```
選択: [    ]

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: C) Snowpipeはサーバーレスで動作し、ファイルが到着すると自動的にデータをロードする**

Snowpipeは「ファイルが届いたら自動的に読み込む、継続的なデータ取り込みサービス」です。

| 特徴 | 説明 |
|------|------|
| サーバーレス | 「サーバーレス」とはユーザーがサーバー（コンピュータ）を用意・管理しなくてよいという意味。Snowflake側がリソースを自動的に調整してくれる |
| Virtual Warehouse | ユーザー自身のWarehouseは**不要**。Snowflakeが内部で処理する |
| 動作のしくみ | クラウドストレージ（S3やAzure Blobなど）に新しいファイルが置かれたときに自動的に通知が来て、Snowpipeがロードを開始する |
| ファイル形式 | COPY INTOと同様に複数形式（CSV・JSON・Parquetなど）に対応 |

**COPY INTO との違い：**
- `COPY INTO` → 手動または決まった時間にまとめて取り込む（バッチ処理）
- `Snowpipe` → ファイルが届いたら即座に自動で取り込む（継続的・自動処理）

## 問題7: ロールベースアクセス制御（RBAC）

Snowflakeのロールベースアクセス制御について、**正しい説明**はどれですか？

- A) 1人のユーザーには1つのロールしか割り当てられない
- B) ロールは他のロールに付与することができ、権限の階層構造を作成できる
- C) ACCOUNTADMIN ロールは他のロールから権限を継承しない
- D) すべてのユーザーはデフォルトでSYSADMINロールを持つ

### あなたの回答

```
選択: [    ]

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: B) ロールは他のロールに付与することができ、権限の階層構造を作成できる**

RBAC（Role-Based Access Control）とは「ロール（役割）に権限をまとめ、ユーザーにロールを割り当てることでアクセスを管理する仕組み」です。

| 項目 | 説明 |
|------|------|
| 複数ロール割り当て | 1人のユーザーに複数のロールを割り当てられる |
| ロール階層 | ロールを別のロールに付与（継承）できる。「上位ロールは下位ロールの権限もすべて使える」 |
| ACCOUNTADMIN | 最上位ロール。SYSADMINとSECURITYADMINの権限を継承している |
| デフォルトロール | 新規ユーザーのデフォルトはPUBLIC（最低限の権限のみ） |

**システム定義ロール階層（上位が下位の権限を含む）：**
```
ACCOUNTADMIN（全権限）
├── SYSADMIN（オブジェクト管理）
│   └── （カスタムロールを付与できる）
└── SECURITYADMIN（ユーザー・ロール管理）
    └── USERADMIN（ユーザー作成・ロール付与）
```
PUBLIC（全ユーザーが自動付与される基本ロール）

**なぜロール階層が便利なのか？**
例えば「データアナリストロール」を作って「特定テーブルの読み取り権限」を付与すれば、何百人いてもそのロールを割り当てるだけで管理できます。個人ごとに権限を設定する必要がありません。

📖 **参照先:** まとめ/05_アクセス制御_ロール.md > セクション1（RBAC）

</details>

### 実際に確認してみよう

**Snowsight（UI）での手順:**
1. 左メニューの **Admin > Users & Roles** を開く
2. **Roles** タブでシステム定義ロール（ACCOUNTADMIN・SYSADMIN等）の階層構造を確認
3. **+ Role** ボタンからカスタムロールを作成し、親ロールを指定して階層を確認

**SQLでの確認:**
```sql
-- カスタムロールの作成
CREATE ROLE analyst_role;

-- ロールに権限を付与（データベースの使用権限）
GRANT USAGE ON DATABASE my_db TO ROLE analyst_role;
GRANT USAGE ON SCHEMA my_db.public TO ROLE analyst_role;
GRANT SELECT ON ALL TABLES IN SCHEMA my_db.public TO ROLE analyst_role;

-- ロールを上位ロールに付与（階層構造の作成）
GRANT ROLE analyst_role TO ROLE SYSADMIN;

-- ロールをユーザーに付与
GRANT ROLE analyst_role TO USER my_user;

-- ロールの一覧と階層確認
SHOW ROLES;

-- 特定ロールに付与されている権限の確認
SHOW GRANTS TO ROLE analyst_role;
```

## 問題9: クラスタリングキー

テーブルのクラスタリングキーについて、**正しい説明**はどれですか？

- A) すべてのテーブルにクラスタリングキーを設定することが推奨される
- B) クラスタリングキーはマイクロパーティション内のデータの物理的な並び順を定義する
- C) クラスタリングキーを設定するとストレージコストは下がる
- D) クラスタリングキーは一度設定すると変更できない

### あなたの回答

```
選択: [    ]

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: B) クラスタリングキーはマイクロパーティション内のデータの物理的な並び順を定義する**

クラスタリングキーとは「どの列（カラム）を基準にデータをまとまった順番で保存するか」を指定するものです。

| 項目 | 説明 |
|------|------|
| 効果がある対象 | 数TB以上の大規模テーブルで特に有効。小規模テーブルにはほぼ不要 |
| 物理的な並び順 | 指定した列の値が近いデータが同じマイクロパーティションにまとまって保存される |
| コスト | 「Automatic Clustering（自動クラスタリング）」という処理が継続的に実行されるため**ストレージと処理コストが追加発生する** |
| 変更 | 後からキーを変更・削除することも可能 |

**なぜクラスタリングキーでクエリが速くなるのか？**
例えば「日付」列でクラスタリングしておくと、「2024年1月のデータ」を検索するとき、その期間のデータが入ったマイクロパーティションだけを読めばよくなります（クエリプルーニング＝不要なパーティションの読み飛ばし）。全データをスキャンしなくてよいため高速化します。

❌ **すべてのテーブルに推奨：** 追加コストが発生するため小規模テーブルには不向きです。大規模テーブルかつ特定の列でフィルタが多い場合のみ検討します。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション6（クラスタリング）

</details>

### 実際に確認してみよう

**SQLでの確認:**
```sql
-- クラスタリングキー付きでテーブルを作成
CREATE TABLE sales (
  sale_date DATE,
  product_id NUMBER,
  amount NUMBER
)
CLUSTER BY (sale_date);

-- 既存テーブルにクラスタリングキーを追加
ALTER TABLE sales CLUSTER BY (sale_date, product_id);

-- クラスタリング状態の確認（クラスタリングの深さ・幅）
SELECT SYSTEM$CLUSTERING_INFORMATION('sales', '(sale_date)');

-- クラスタリングキーの確認
SHOW TABLES LIKE 'sales';

-- クラスタリングキーの削除
ALTER TABLE sales DROP CLUSTERING KEY;
```

## 問題10: オブジェクト階層

Snowflakeのオブジェクト階層について、**正しい順序**はどれですか？（上位から下位へ）

- A) Organization → Account → Schema → Database → Table
- B) Account → Database → Table → Schema → Column
- C) Organization → Account → Database → Schema → Table/View/Stage等
- D) Database → Account → Schema → Table → Column

### あなたの回答

```
選択: [    ]

選んだ理由・メモ:


```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**正解: C) Organization → Account → Database → Schema → Table/View/Stage等**

Snowflakeのオブジェクト（データやリソースを管理する単位）は入れ子構造になっています。上位のものが下位のものをまとめて管理します。

```
Organization（組織）
└── Account（アカウント）← Snowflakeの「契約単位」。独自のURLを持つ
    └── Database（データベース）← スキーマをまとめる器
        └── Schema（スキーマ）← 関連するオブジェクトをグループ化する名前空間
            ├── Table（テーブル）← データそのものを行・列で保存
            ├── View（ビュー）← データは持たず保存したSQLクエリの結果を見せる
            ├── Stage（ステージ）← データロード前の一時置き場
            ├── Stored Procedure（ストアドプロシージャ）← Snowflake内で実行できる手続きをまとめたプログラム
            ├── UDF（ユーザー定義関数）← 独自の計算関数を作って使いまわせる仕組み
            └── Stream / Task など
```

| 階層 | 役割 |
|------|------|
| Organization | 複数のアカウントをまとめて管理できる（オプション） |
| Account | Snowflakeの基本契約単位。独自のURL（例: xyz.snowflakecomputing.com）を持つ |
| Database | スキーマのコンテナ（入れ物）。論理的なデータのまとまり |
| Schema | テーブルなどのオブジェクトをグループ化する名前空間 |

**なぜSchemaが必要か：** 同じDatabaseの中に「営業部門のテーブル」と「経理部門のテーブル」を混在させると管理が大変です。Schemaで分けることで整理・管理がしやすくなります。

📖 **参照先:** まとめ/01_アーキテクチャ基礎.md > セクション7（オブジェクト階層）

</details>

### 実際に確認してみよう

**Snowsight（UI）での手順:**
1. 左メニューの **Data > Databases** を開く
2. 既存データベースを展開し、Schema → Tables の階層構造をツリーで確認
3. **+ Database** ボタンからデータベースを作成し、その中にスキーマとテーブルを作成してみる

**SQLでの確認:**
```sql

CREATE DATABASE my_practice_db;

-- スキーマの作成（データベース内）
CREATE SCHEMA my_practice_db.sales_schema;
CREATE SCHEMA my_practice_db.hr_schema;

-- テーブルの作成（スキーマ内）
CREATE TABLE my_practice_db.sales_schema.orders (
  order_id NUMBER,
  order_date DATE,
  amount NUMBER
);

-- ビューの作成（スキーマ内）
CREATE VIEW my_practice_db.sales_schema.recent_orders AS
  SELECT * FROM my_practice_db.sales_schema.orders
  WHERE order_date >= DATEADD(DAY, -30, CURRENT_DATE());

-- オブジェクト階層の確認
```
