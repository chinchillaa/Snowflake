## 問題1

**Snowflakeのクローンに関する記述のうち、正しいものはどれですか？**

- [ ] クローンはデータベースからのみ作成可能であり、テーブルやスキーマからは作成できない。
    
- [x] **正解：クローン作成時点ではストレージをほとんど（または全く）消費せず、元のデータかクローンされたデータのいずれかに変更が加えられた場合にのみストレージが消費される。**
    
- [ ] クローンは元のオブジェクトと自動的に同期される。
    
- [ ] クローンは元のオブジェクトと同じ容量のストレージを消費する。
    

## 解説

✅ **クローン作成時点ではストレージをほとんど（または全く）消費せず、元のデータかクローンされたデータのいずれかに変更が加えられた場合にのみストレージが消費される。**

Snowflakeのクローン機能は、作成時点では「メタデータ（データを管理するための情報）」のみをコピーする操作です。これには以下の特徴があります。

- クローンは、作成時点では元のオブジェクトと同じデータを共有します。
    
- そのため、作成時には追加のストレージ容量を消費しません。
    
- 元のデータ、またはクローンされたデータのどちらかに変更が加えられた時に初めて、変更分のストレージが消費されます（コピー・オン・ライト動作）。
    

これにより、テスト環境や開発環境の構築、バックアップなどの作業を非常に効率的に行うことができます。

❌ **クローンは元のオブジェクトと同じ容量のストレージを消費する**：作成時点では消費しません。

❌ **クローンは自動的に同期される**：作成後はそれぞれ独立したオブジェクトとして扱われ、同期はされません。

❌ **クローンはデータベースからのみ作成可能**：スキーマやテーブルなど、サポートされている他のオブジェクトもクローン可能です。

**ドメイン:** データ保護とデータ共有

## 問題2

- [ ] データファイルのバックアップを作成するため。
    
- [ ] データファイルのセキュリティポリシーを定義するため。
    
- [ ] 仮想ウェアハウスのサイズを管理するため。
    
- [ ] **正解：ロード（読み込み）するデータファイルの形式と構造（例：CSVの区切り文字、JSONの圧縮形式など）を指定するため。**
    

## 解説

✅ **ロードするデータファイルの形式と構造を指定するため**

SnowflakeのFILE FORMATオブジェクトは、データの読み込み（ロード）や書き出し（アンロード）を行う際に、データファイルの構造や文字コードなどをどのように解釈するかを定義します。以下の設定が含まれます。

- ファイルの種類（CSV、JSON、Parquet、Avroなど）
    
- 区切り文字、圧縮の種類、文字コード

このオブジェクトを使うことで、ファイルの解析ルールを1箇所にまとめることができ、`COPY INTO`処理がスムーズになります。複数のステージやテーブルで同じルールを使い回せるため、コードがすっきりします。

❌ **セキュリティポリシーを定義するため**：これは「ロール（役割）」とアクセス制御機能によって管理されます。

❌ **仮想ウェアハウスのサイズを管理するため**：ウェアハウスの個別設定で管理されます。

❌ **データファイルのバックアップを作成するため**：バックアップは、Time Travel（タイムトラベル）やFail-safe（フェイルセーフ）といったSnowflakeの内部機能によって処理されます。

## 問題10

- [ ] LOAD DATA
    
- [ ] INSERT INTO
    
- [ ] IMPORT DATA

## 解説

✅ **COPY INTO**

Snowflakeにおいて、ステージ（Snowflake内部の置き場、またはAmazon S3などの外部の置き場）からテーブルへデータを大量に読み込む際の標準的なコマンドが `COPY INTO` です。以下の特徴があります。

- CSV、JSON、Parquetなどのファイルからの高速な一括読み込み（バルクロード）をサポート。
    
- ファイル形式の指定、エラー発生時の動作、読み込み時の簡単なデータ変換が可能。

SQL

```

```

❌ **INSERT INTO**：1行ずつ、あるいは少量のデータを挿入するためのもので、ステージからの大量データの読み込みには使いません。

## 問題13

**Snowflakeのコンピュート（計算処理）レイヤーは、どのようにクエリを実行しますか？**

- [ ] シングルスレッド処理（単一の処理経路）を使用する。
    
- [x] **正解：MPP（超並列処理）コンピュートを使用する。**
    
- [ ] 共有ディスク型アーキテクチャで実行する。
    
- [ ] クラウドストレージ内のデータを最適化することで実行する。
    

## 解説

✅ **MPP（超並列処理）コンピュートを使用する**

Snowflakeの計算処理層は、MPP（Massively Parallel Processing：超並列処理）方式でクエリを実行します。具体的には以下のように動きます。

- クエリを細かいタスク（作業の単位）に分割し、複数のコンピュートノード（計算を行うサーバー群＝仮想ウェアハウス）に割り振ります。
    
- 各ノードがデータの断片を同時に（並列に）処理するため、パフォーマンスと処理能力が飛躍的に向上します。
    

この仕組みにより、システム全体でディスクを共有する手軽さと、複数のコンピューターで手分けして処理する速さの両方を実現しています。

❌ **シングルスレッド処理**：Snowflakeは並列処理を前提に設計されています。

❌ **共有ディスク型アーキテクチャ**：データ自体は中央のストレージで管理されますが、実際の計算処理は分散されたコンピューター群が行います。

❌ **クラウドストレージ内のデータを最適化する**：データの最適化は保存されるタイミングで行われるものであり、クエリ実行そのものの仕組みではありません。

**ドメイン:** Snowflakeの操作とアーキテクチャ

## 問題36

**「標準（Standard）ウェアハウス」と「マルチクラスターウェアハウス」の決定的な違いは何ですか？**

- [x] **正解：マルチクラスターウェアハウスは、クラスター（計算サーバーの群）を追加することで、自動的に水平方向の拡張（スケールアウト）ができる。**
    
- [ ] 標準ウェアハウスの方が、複雑なクエリに対してより高いパフォーマンスを発揮する。
    
- [ ] 標準ウェアハウスは、データのロード（読み込み）にしか使用できない。

## 解説

✅ **マルチクラスターウェアハウスは、自動的に水平方向の拡張（スケールアウト）ができる**

2つのウェアハウスの違いは「同時アクセスの処理能力（並行処理）」にあります。

- **標準ウェアハウス**：計算サーバーの群（クラスター）が「1つ」だけです。性能を上げるにはサイズを大きくする（スケールアップ）しかありません。

❌ **標準ウェアハウスの方が複雑なクエリに強い**：複雑な処理能力は「ウェアハウスのサイズ」に依存するため、標準とマルチクラスターの違いではありません。

❌ **マルチクラスターウェアハウスは自動停止できない**：利用状況に応じて全体を自動停止させる機能は、どちらのウェアハウスでも利用可能です。

## 問題60

**テーブルに対する `SELECT`（読み取り）権限を持っているのに、テーブルのデータを読み取ることができません。原因として考えられるものはどれですか？**

- [x] **正解：そのテーブルが含まれている「データベース」または「スキーマ」に対する USAGE（使用）権限を持っていない。**

- **管理機能**：クエリの計画と最適化、システム情報の管理（メタデータ）。
    

❌ **データストレージレイヤー**：暗号化されたデータを実際に「保管」する場所ですが、暗号化の仕組み自体を管理しているわけではありません。

❌ **コンピュート（クエリ処理）レイヤー**：実際の計算処理を力仕事として実行する場所ですが、認証などの管理機能は持ちません。

**ドメイン:** Snowflakeの操作とアーキテクチャ

## 問題64

**Snowflakeのクエリにおいて、文字列を「JSONデータとして解析・変換」するために使用されるSQL関数はどれですか？**

- [x] **正解：PARSE_JSON()**
    
- [ ] TO_JSON()
    
- [ ] EXTRACT_JSON()
    
- [ ] CONVERT_JSON()
    

## 解説

✅ **PARSE_JSON()**

`PARSE_JSON()` 関数は、単なる文字列（テキスト）として扱われているJSONデータを読み取り、Snowflakeが半構造化データとして扱える `VARIANT` 型に変換する役割を持ちます。

この変換を行うことで、ドット記法（例: `parsed_data:name`）を使ってJSONの中身を簡単に検索できるようになります。

**例:**

SQL

```
SELECT PARSE_JSON('{"name": "Random", "city": "Mumbai"}') AS parsed_data;
```

❌ **TO_JSON()**：これは逆に、`VARIANT` 型のデータを単なる「JSON形式の文字列」に変換して出力する機能です。

❌ **EXTRACT_JSON() / CONVERT_JSON()**：Snowflakeにはこのような関数は存在しません。

## 問題65

**CSVデータを読み込む際、FILE FORMATの `SKIP_HEADER` オプションはどのような働きをしますか？**

- [ ] ヘッダー行をデータの一部として含める。
    
- [ ] ヘッダー行から列の名前を自動的に作成する。
    
- [ ] データファイル内の空行を無視する。
    
- [x] **正解：指定した数のヘッダー（見出し）行を読み飛ばす。**
    

## 解説

✅ **指定した数のヘッダー（見出し）行を読み飛ばす。**

CSVファイルの上部に列名などの見出し行がある場合、それを実データとして読み込んでしまうとエラーになったりノイズになったりします。

`SKIP_HEADER` オプションに数字（通常は `1`）を指定することで、ファイルの一番上から数えて指定した行数分だけ、読み込み処理をスキップさせることができます。

❌ **空行を無視する**：これは `SKIP_BLANK_LINES = TRUE` という別のオプションで行います。

❌ **列の名前を自動的に作成する**：このオプションには列名を定義する機能はありません。
