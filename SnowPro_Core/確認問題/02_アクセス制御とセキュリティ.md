### 正解・解説

<details>
<summary>クリックして確認</summary>

**RBAC (Role-Based Access Control):** ロールベースのアクセス制御フレームワーク

**権限付与の流れ:**
権限（Privilege） → **ロール（Role）** → **ユーザー（User）**

ユーザーに直接権限を付与するのではなく、ロールを介して権限を付与する。

**セキュリティ可能オブジェクトの例:**
1. データベース
2. スキーマ
3. テーブル
4. ビュー

<details>
<summary>クリックして確認</summary>

## 問題4: GRANT文

権限を付与するGRANT文の構文を書け。

### あなたの回答

```
テーブルへのSELECT権限をロールに付与:
  GRANT [        ] ON [        ] table_name TO [        ] role_name;

データベースのUSAGE権限を付与:
  GRANT [        ] ON [        ] db_name TO ROLE role_name;

スキーマ内の全テーブルにSELECT権限を付与:
  GRANT SELECT ON [        ] TABLES IN SCHEMA schema_name TO ROLE role_name;

将来作成されるテーブルにも自動で権限を付与:
  GRANT SELECT ON [        ] TABLES IN SCHEMA schema_name TO ROLE role_name;
```

### 正解・解説

<details>
<summary>クリックして確認</summary>

```sql
-- テーブルへのSELECT権限
GRANT SELECT ON TABLE table_name TO ROLE role_name;

-- データベースのUSAGE権限
GRANT USAGE ON DATABASE db_name TO ROLE role_name;

-- スキーマ内の全テーブルに権限付与
GRANT SELECT ON ALL TABLES IN SCHEMA schema_name TO ROLE role_name;

-- 将来のテーブルにも自動付与
GRANT SELECT ON FUTURE TABLES IN SCHEMA schema_name TO ROLE role_name;
```

**ポイント:**
- `ALL` は既存のオブジェクトすべて
- `FUTURE` は今後作成されるオブジェクト

</details>

## 問題5: ロール階層とカスタムロール

カスタムロールのベストプラクティスを説明せよ。

### あなたの回答

```
カスタムロールを作成する際の推奨事項:
  最終的な親ロール: [            ]

理由:


ロールをユーザーに付与するコマンド:
  GRANT ROLE [          ] TO USER [          ];

ロールを別のロールに付与（階層化）:
  GRANT ROLE [          ] TO ROLE [          ];
```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**推奨事項:**
カスタムロールは最終的に **SYSADMIN** の子孫ロールとする

**理由:**
- SYSADMINがすべてのオブジェクトを管理可能になる
- ACCOUNTADMINは日常的に使用しないため、SYSADMINを通じて管理
- 権限管理が一貫して行える

```sql
-- ロールをユーザーに付与
GRANT ROLE analyst_role TO USER john;

-- ロールを別のロールに付与（階層化）
GRANT ROLE analyst_role TO ROLE sysadmin;
```

</details>

## 問題6: セキュアビュー

セキュアビューについて説明せよ。

### あなたの回答

```
セキュアビューとは:


通常のビューとの違い:
  定義の可視性: 
  クエリ最適化: 

作成構文:
  CREATE [          ] VIEW view_name AS SELECT ...;

使用ケース:
  1. 
  2. 
```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**セキュアビュー:** ビューの定義（SQL）が権限のないユーザーから見えないようにするビュー

**通常のビューとの違い:**
- 定義の可視性: **非表示**（SHOW VIEWSでも定義が見えない）
- クエリ最適化: 一部の最適化が制限される（セキュリティのため）

**作成構文:**
```sql
CREATE SECURE VIEW view_name AS SELECT ...;
```

**使用ケース:**
1. データ共有時（Secure Data Sharing）
2. 行レベル・列レベルのセキュリティ実装
3. ビジネスロジックの隠蔽

</details>

### あなたの回答

```
方法1: セキュアビュー + [                    ]関数

CURRENT_ROLE()の戻り値: 
CURRENT_USER()の戻り値: 

ビューでの使用例（概念）:
  CREATE SECURE VIEW my_view AS
  SELECT * FROM my_table
  WHERE department = [                          ];
```

### 正解・解説

<details>
<summary>クリックして確認</summary>

**方法:** セキュアビュー + **コンテキスト関数**

**コンテキスト関数:**
- `CURRENT_ROLE()`: 現在のセッションのアクティブロール
- `CURRENT_USER()`: 現在のユーザー名

**使用例:**
```sql
CREATE SECURE VIEW my_view AS
SELECT * FROM my_table
WHERE department = CURRENT_ROLE();
```

または、マッピングテーブルを使用:
```sql
CREATE SECURE VIEW my_view AS
SELECT t.* FROM my_table t
JOIN access_mapping m ON t.department = m.department
WHERE m.role_name = CURRENT_ROLE();
```

## 問題10: セキュリティ総合

以下の文の正誤を判定せよ。

### あなたの回答

```
1. ACCOUNTADMINはスーパーユーザーである
   [正 / 誤]
   理由:

2. SECURITYADMINはMANAGE GRANTS権限を持つ
   [正 / 誤]
   理由:

3. PUBLICロールに権限を付与すると全ユーザーがその権限を得る
   [正 / 誤]
   理由:

4. OWNERSHIPは複数のロールが同時に持てる
   [正 / 誤]
   理由:

5. セキュアビューはデータ共有時に必須である
   [正 / 誤]
   理由:
```

### 正解・解説

<details>
<summary>クリックして確認</summary>

1. **誤** - ACCOUNTADMINはスーパーユーザー**ではない**。適切な権限がなければオブジェクトにアクセスできない

2. **正** - SECURITYADMINはMANAGE GRANTS権限を持ち、任意のGRANT/REVOKEを実行可能

3. **正** - PUBLICは全ユーザーに自動付与されるため、PUBLICへの権限付与は全員に影響

4. **誤** - OWNERSHIPは**1つのロールのみ**が持てる。GRANTすると移動する

5. **正** - Secure Data Sharingでは、基盤テーブルへのアクセスを防ぐためセキュアビューが必要

</details>
